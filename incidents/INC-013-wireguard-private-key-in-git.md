# INC-013: Приватный ключ WireGuard-сервера в git

## Описание
Файл `infrastructure/vpn/wireguard/wg0.conf` отслеживается git и содержит **реальный приватный ключ сервера WireGuard** (не плейсхолдер), а также 2 рабочих `PresharedKey` для пиров. Файл присутствует в репозитории с самого первого коммита `dedecf4c` ("initial private release — clean history") — то есть пережил пересоздание репозитория, выполненное в рамках ликвидации INC-004.

Это **рецидив INC-004**: "чистая история", созданная для устранения утечки секретов, изначально содержала новый секрет.

## Severity
🔴 **CRITICAL** — приватный ключ VPN-сервера. Компрометация даёт возможность выдать себя за сервер туннеля, расшифровать/перехватить трафик пиров, подключиться к приватной сети `10.66.66.0/24`.

## Затронутые данные

- **Файл:** `infrastructure/vpn/wireguard/wg0.conf`
- **`PrivateKey`** — приватный ключ сервера WireGuard (реальный)
- **`Address`** = `10.66.66.1/24` — адрес сервера в туннеле
- **`Endpoint`** = `109.252.133.118:2229` — реальный публичный IP:порт
- **`PresharedKey` ×2** — реальные PSK пиров
- **8 секций `[Peer]`** — публичные ключи и AllowedIPs всех клиентов VPN

## Хронология

- **2026-05-17** — репозиторий пересоздан с "чистой" историей в рамках INC-004; коммит `dedecf4c` уже содержит `wg0.conf` с реальными ключами.
- **2026-06-16** — при глубоком аудите секретов в git (сессия owl) обнаружено, что `wg0.conf` отслеживается и содержит реальный приватный ключ сервера. `git ls-files` подтверждает, `git log --follow` указывает на `dedecf4c` как первый коммит файла.

## Почему `.gitignore` не сработал
`.gitignore` прикрывает только клиентские конфиги:
- `projects/*/peers/*.conf`
- `client_*.conf`

Путь `infrastructure/vpn/**` (где лежит серверный конфиг с приватным ключом) не покрыт ни одним правилом.

## ⚠️ Уточнение по факту (16.06.2026): WireGuard НЕ РАБОТАЕТ

Проверка реального состояния опровергла исходное предположение о «живом туннеле»:
- `wg0` интерфейса нет (`ip link show wg0` — пусто).
- `wg show` — ни одного туннеля/рукопожатия. **Клиентов нет.**
- `systemctl wg-quick@wg0` — `inactive` + `disabled`.
- Порт `2229` (Endpoint из конфига) никто не слушает.
- xray (`systemctl`) — inactive, awg0 — inactive/нет интерфейса. Весь `infrastructure/vpn/` фактически не используется.

**Следствие:** ротация ключа НЕ грозит простоем (отключать некого). Риск «разорвать 8 пиров» снят — пиров нет.

## План ремедиации

### Этап 1 — Ротация ключей (ПЕРВЫМ ДЕЛОМ, до любых git-операций)
Любой ключ, попавший в git, считается скомпрометированным. Переписывание истории НЕ отменяет компрометацию — ключ нужно сменить.
1. Сгенерировать новую пару ключей сервера (`wg genkey | tee privatekey | wg pubkey > publickey`).
2. Перегенерировать `PresharedKey` для каждого пира (`wg genpsk`).
3. Обновить серверный `wg0.conf` и раздать новые публичные ключи/PSK всем 8 пирам.
4. Перезапустить туннель (`wg-quick down/up wg0`), проверить связность.

### Этап 2 — Убрать конфиг из-под версионирования
1. `git rm --cached infrastructure/vpn/wireguard/wg0.conf` (оставить файл на диске).
2. Добавить в `.gitignore`:
   ```
   infrastructure/vpn/**/*.conf
   ```
3. Положить рядом обезличенный `wg0.conf.example` с плейсхолдерами для документации.

### Этап 3 — Переписывание истории (по согласованию с Котом/ЗавЛабом)
1. Удалить файл из всей истории (`git filter-repo --path infrastructure/vpn/wireguard/wg0.conf --invert-paths` или BFG).
2. Force-push в origin **только после явного согласования** — переписывание истории затрагивает всех, кто работает с репозиторием.
3. Сверить с правилом owl: мерж/перезапись main — только через Кота или ЗавЛаба.

> ⚠️ Сова инцидент **не устраняет самостоятельно**: ротация ключей и rewrite истории — операции с высоким blast radius (живой VPN, общий remote). Требуется решение ЗавЛаба.
