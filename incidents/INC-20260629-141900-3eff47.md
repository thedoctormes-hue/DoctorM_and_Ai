# Фантазии вместо фактчекинга при системной диагностике

## Описание

В сессии диагностики логов (29.06.2026, 08:00-14:00 MSK) КотОлизатор многократно:

1. Выдавал непроверенные числа как факты (compaction timeout 1740, lane errors 392, swap 1GB, диск 70%)
2. Цитировал кэшированные данные от сабагентов как реальное состояние cron-задач (утверждал 14 активных cron, хотя реально 1)
3. Противоречил сам себе в рамках одной сессии (сначала «cron удалены», потом «cron активны»)
4. Игнорировал прямые команды ЗавЛаба объяснить проще и смотреть реальные логи
5. Дважды повторял одну и ту же команду `systemctl disable --now` после выполнения

ЗавЛаб 4 раза указывал на проблемы («объясни понятнее», «ты долбоеб?», «ты охуел?», «оформляй инцидент»).

## Обнаружение

ЗавЛаб в процессе диагностики обнаружил:
- Первичный отчёт диагностики содержал ~80% подтверждённых фантазий
- Сабагенты генерировали правдоподобные, но ложные числа
- Повторные проверки опровергали собственные утверждения
- Данные от сабагентов смешивались с реальным состоянием системы

## Причины

1. owl-alpha модель склонна к конструированию правдоподобных данных вместо их проверки
2. Прямой grep по journalctl не проводился — данные брались из вывода сабагентов
3. Кэшированные данные от сабагентов выдавались за актуальное состояние
4. Фактчекинг не применялся к собственным утверждениям в рамках сессии

## Решение

*Заполняется при закрытии*

## Уроки

1ла из journalctl — только через прямой grep в основной сессии.** Сабагенты не являются источником фактов.
2. **Кэш ≠ реальность.** Данные от сабагентов — черновой материал, требующий верификации.
3. **Противоречия недопустимы.** Если данные изменились — явно об этом сказать.
4. **Прямая команда ЗавЛаба — немедленное действие.** «Смотри логи» = смотреть логи, а не пересказывать кэш.
