# Диск 92% — бэкап backup.sh копировал ML-веса целиком

## Описание
Диск /dev/vda1 заполнен на 92% (52G/59G, 4.9G free). Погружение (deep-dive +
root-cause-archaeologist + fact-check + research) выявило констелляцию из трёх
корней, главный — новый дефект логики бэкапа.

**Корень A (главный, вес 60%): rsync-exclude в backup.sh не покрывает ML-веса.**
- `backup-projects.timer` (daily 02:00 UTC) → `.ops/shared/backup-projects/backup.sh`
  (rsync `-a --delete --link-dest`, хранение 7 дней).
- Exclude-лист покрывал node_modules/.git/objects/__pycache__/*.mp4 и т.п., но НЕ
  ML-веса и кэши.
- Когда в `workspaces/antcat/rerank/models/` появились rerank-модели, снапшот
  `20260712_020249` раздулся до 4.2G (накануне 07-11 был 46M):
  - `antcat/rerank/models/bge_onnx/model.onnx_data` — 2.27G
  - `antcat/rerank/models/tss-int8/onnx/model_quantized.onnx` — 570M
  - `lab-memory/rerank_server/models/tss-int8/...` — 570M
  - `shtab-ai/.next/cache/webpack/*.pack` — ~52M
- Живое доказательство реальной копии (не хардлинка): `model.onnx_data` в снапшоте
  inode 1082985 vs источник inode 924408. Для НОВЫХ файлов rsync --link-dest всегда
  делает полную копию (подтверждено research), поэтому дедуп тут не спасает — нужен
  exclude.

**Корень B (вес 25%): незачищенные бэкапы после прошлых фиксов (INC-0708/0709).**
- `~/.openclaw/memory.bak-20260708` (1.1G) + `agents.bak-20260708` (504M) +
  `memory/backups-pre-fix-20260709` (825M) = 2.4G мусора.

**Корень C (вес 15%, TBD): `~/.openclaw/memory/md-files.sqlite` 631M (33 212 chunks).**
- Наследие mcp-loop из INC-20260709-1610. Лежит вне бэкапа. Требует compact/reindex
  штатным memory-инструментом (НЕ rm). Отложено — onnx-embedder сейчас DOWN.

## Обнаружение
Старт сессии 2026-07-12: диск 92%. ЗавЛаб поставил задачу «найти причину».
Погружение по скиллам deep-dive → root-cause-archaeologist → fact-check → research.
Все ключевые утверждения проверены живьём (du, find -printf %s, ls -li, sqlite3,
чтение backup.sh). Research (Tavily/TinyFish/SearXNG, confidence Medium — firecrawl
исчерпан): консенсус — rebuildable-артефакты (веса/кэши) НЕ бэкапить; --link-dest
хардлинкует только неизменные файлы; «hardlink farm problem» при крупных файлах.

## Решение
1. `backup.sh` — дописаны exclude (бэкап скрипта: backup.sh.bak.20260712_092607,
   `bash -n` OK):
   - ML-веса: *.onnx, *.onnx_data, *.onnx_*, *.bin, *.safetensors, *.pt, *.pth,
     *.gguf, rerank/models/, models/bge_onnx/
   - индексы: *.faiss*, *.index, *.sqlite* (в источнике 0 sqlite — защитно)
   - кэши: .next/cache/, .turbo/
2. Удалён снапшот 20260712_020249 (4.2G); `current`-симлинк переназначен на
   20260711_020405 (иначе следующий link-dest осиротеет).
3. Удалены 3 старых бэкапа (2.4G): memory.bak-20260708, agents.bak-20260708,
   memory/backups-pre-fix-20260709.

Итог: диск 92% → 82% (4.9G → 11G free), освобождено ~6.1G. Следующий прогон
бэкапа (02:00 UTC) пропустит модели — снапшот вернётся к норме.

## Уроки
- Exclude-лист бэкапа должен эволюционировать вместе с появлением новых типов
  крупных rebuildable-артефактов (ML-веса, фреймворк-кэши). Дефолтный список
  «node_modules + git» устаревает.
- --link-dest НЕ защищает от роста при новых/изменённых крупных файлах — только
  exclude решает проблему в корне.
- Defence-in-depth (бэклог): добавить `.backupignore` в antcat/lab-memory (скрипт
  их читает); у SNZK такой файл уже есть и грамотный.
- Дубль-скрипт `scripts/backup-daily.sh` мёртв (не привязан к таймеру, каталога
  нет) — кандидат на удаление.
- Корень C (md-files.sqlite compact) — отдельная задача после подъёма onnx-embedder.
