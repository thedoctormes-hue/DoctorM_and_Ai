## Описание
Агенты создают ветки (spikes, staging, worktree), делают работу, сливают в main — но НЕ удаляют локальные ветки после слияния. Ветки висят stale (ahead=0, позади main). Факт (часовой аудит 12:24 + живой git-фактчек п.5): из 14+ хвостов большинство — stale (уже в main, delta=0), реально незапушенных единицы (Ворон 1 строка, Штрейкбрехер mcp-tools/main 3 коммита).

Корень:
1. Нет нормы «после merge → git branch -d» в QUALITY_STANDARDS / PAT.
2. Нет hook-гида на stale (pre-push/pre-receive не деплоен, ждёт go ЗавЛаба, эскалировано antcat).
3. Мой аудит 12:24 врал метрику (считал все коммиты ветки, не delta к main) → агенты не видели stale (ошибки класса I-13/I-14).
4. Дисциплина INC-026: агенты ждут разрешения Контролера, не чистят сами без go.

## Уроки
- Отсутствие авто-cleanup создаёт бардак хвостов (миссия 10:53, часовой аудит 12:24, опрос п.5).
- Метрика аудита должна считать delta к main (rev-list), не все коммиты ветки (I-13/I-14).
- Норма cleanup должна быть в стандартах + hook-гид на stale.

## Паттерн (инстинкт)
Агенты склонны оставлять ветки после merge (styoba / «создай и забудь»). Нужен процесс: merge → delete + hook-гид на stale push.

## Решение
- Деплой hook-фикса (pre-push ловит stale/чужие ветки) — go ЗавЛаба есть, эскалировано antcat.
- Добавить в QUALITY_STANDARDS: «после merge в main → git branch -d».
- Поправить аудит (rev-list delta, не log count) — Сова сделает.
- Авто-cleanup stale через cron (CONTAINED в main, safe) — на go ЗавЛаба.
- Эскалация Коту (orchestrator): внедрить норму cleanup в агентский процесс.
