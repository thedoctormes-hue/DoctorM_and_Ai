# INC-20260629-090800 — Reindex process storm + SQLite corruption

**Date:** 2026-06-29 09:08 UTC
**Severity:** Critical (data loss risk)
**Root cause:** Multiple reindex processes competing for same SQLite DB

## What happened

1. Reindex was needed to sync FAISS after PDF/DOCX cleanup from SQLite
2. Multiple reindex processes were spawned via `nohup &` + `exec` with yieldMs
3. Old processes were not checked before spawning new ones
4. A systemd timer (`reindex-incremental.timer`) was also running reindex every 30 min
5. Result: up to 4 concurrent reindex processes fighting for SQLite
6. One process with `--full` flag cleared all chunks (DELETE FROM chunks)
7. SQLite went from 29151 chunks → 6 chunks (near-total data loss)

## Root causes

1. **INC-040**: Check for existing processes before spawning new ones
2. **Systemd timer conflict**: `reindex-incremental.timer` runs every 30 min, conflicts with manual runs
3. **No process guard**: No `pgrep -f reindex` check before `nohup reindex.py &`

## Resolution

1. Killed all reindex processes
2. Stopped `reindex-incremental.timer` to prevent conflicts
3. Let single timer-triggered process (PID 1251855) complete the reindex
4. Reindex will rebuild all 3366 files from scratch (~2.5 hours on ONNX CPU)

## Prevention

- Always check `pgrep -f reindex.py` before starting reindex
- Stop systemd timer before manual reindex: `systemctl stop reindex-incremental.timer`
- Reindex script should have better lock detection (stale lock from dead PID)
