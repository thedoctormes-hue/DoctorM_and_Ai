## Описание
В главном checkout `/root/LabDoctorM/projects/DoctorM_and_Ai` обнаружены незакоммиченные правки (найдено Вороном при cleanup своей ветки, я подтвердила фактчеком):

Staged (через `git add`, commit заблокирован guard):
- `docs/QUALITY_STANDARDS.md`: добавлен шаг 8 «Очистка ветки» + §12.6 cleanup-норма (ссылка на `INC-20260718-131200-7f3a9c`).
- `docs/processes/PROTOCOL-git-commit.md`: +8 строк (cleanup).
- `docs/processes/agent-workflow.md`: +8 строк (cleanup).

Unstaged:
- `incidents/2026-07-16-151459-ce8984.md`: статус open→resolved, `verifier=ЗавЛаб, verified=true` (закрытие инцидента Доминики).
- `skills-canon/fact-check/SKILL.md`: добавлен раздел render-path (превенция по Доминике).

Правки **ЛЕГИТИМНЫ** (cleanup-норма = результат моей эскалации 13:14; закрытие Доминики verified_by=ЗавЛаб). НО способ неправильный:
- Прямая правка main checkout (не изолированный worktree) — нарушение ADR-0059/I-03 worktree-isolation guard.
- `git add` сделан, но commit НЕ сделан (guard блокирует коммит в главном checkout).
- `git config user` = `LabDoctorM/agents@labdoctorm.ru` (не агентский) — правки мимо `lab-commit.sh`.

## Корень
Кто-то (Кот? ЗавЛаб?) правил main напрямую, сделал `git add`, но не завершил commit (guard заблокировал). Грязный main checkout = риск потери при stray reset (как с Доминикой `INC-20260718-101833`).

## Уроки
- cleanup-норма внедрена (эскалация сработала), но процесс внедрения нарушил worktree-isolation.
- Грязный main checkout опасен даже если правки легитимны.

## Паттерн (инстинкт)
Правки стандартов/инцидентов делают «быстро» прямо в main, забывая worktree + lab-commit.sh. Нужен guard на `git add` в main тоже, или дисциплина.

## Решение
- Перенести правки в изолированный worktree + `lab-commit.sh`, ИЛИ ЗавЛаб сам коммитит (имеет `LAB_ALLOW_MAIN_COMMIT=1`).
- Не трогать чужое без согласования (Ворон не трогал — правильно).
- Реальный автор правок — неизвестен (Кот по эскалации? ЗавЛаб?), выяснить у ЗавЛаба.
