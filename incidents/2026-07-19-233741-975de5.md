# ALM (AnythingLLM) не пережил ребут — семантическая память всех агентов легла (--restart no)

## Описание
После ребута хоста (gateway-restart ~23:27 МСК) контейнер `anythingllm` (AnythingLLM — векторный бэкенд семантической памяти лабы, слушает `127.0.0.1:3002`) **не поднялся автоматически**. Корень: `start-anythingllm.sh` запускал контейнер с флагом `--restart no`. Результат: контейнер `anythingllm` в состоянии `Exited (137)`, порт `:3002` недоступен (`HTTP 000`), и векторный слой `memory-gateway` перешёл в `degraded` для **ВСЕХ 8 агентов** (семантический поиск без векторных хитов) в промежутке ~23:27–23:36.

## Обнаружение
В ходе post-reboot liveness-проверки по прямому запросу ЗавЛаба («После ребута alm + mcp + SearXNG живы?»). Диагностика живьём:
- `docker ps` → нет контейнера `anythingllm` (только `Exited (137)` в `docker ps -a`);
- `curl http://127.0.0.1:3002/...` → `HTTP 000` (connection refused);
- `memory-gateway__gateway_health` → `vector_layer.reachable=false`, `vector_search.functional=false`, `degraded=false` (мягкая деградация, не crash).

## Решение
- В `start-anythingllm.sh` изменено `--restart no` → `--restart unless-stopped` (бэкап: `start-anythingllm.sh.bak.pre-restartfix-2026-07-19T23-35`).
- Контейнер пересоздан через скрипт; ALM поднялся `healthy`, `vectorCount=18783` (хранилище индекса — host-volume `/opt/anythingllm`, сохранено при recreate → **потери данных нет**).
- `memory-gateway__gateway_health` → `vector_layer.reachable=true`, `vector_search.functional=true`, `degraded=false`. Семантическая память восстановлена end-to-end.
- ALM теперь **переживает ребуты**. `alm-sync-incremental.timer` enabled (следующий прогон синхронится с уже поднятым ALM).

## Уроки
1. Reboot-survival не имплицитен: любой контейнер/сервис, обязанный переживать ребут, должен иметь явную restart-политику (`--restart unless-stopped`) или enabled systemd-юнит с `Restart=always/on-failure`.
2. Degraded (не crash) зависимый слой маскирует корень: MCP отвечал, но векторный слой лежал — проверяй сам бэкенд (`gateway_health.vector_layer`), а не только «MCP жив».
3. После ребута верифицируй, что core-сервисы реально up, а не предполагай по старой памяти.
4. Фикс чисто конфигурационный; данные не потеряны (host-volume).

## Паттерн (инстинкт)
> Новый класс (Bole-дедуп по PAT-019 I-01…I-19 — совпадений по root cause нет). Инстинкт добавлен в PAT-019 как **I-20**.

- **Root cause (абстрактный):** Долгоживущий сервис/контейнер, обеспечивающий ядро подсистемы, запущен БЕЗ гарантии выживания при ребуте (нет restart-политики / не enabled systemd-юнит). При ребуте хоста он не возвращается, молча уронив зависимые подсистемы (напр. векторный слой семпамяти для всех агентов) без алерта — потому что зависимый слой деградирует мягко (degraded, не crash) и нет post-reboot health-проверки.
- **Signal:** после любого ребута/gateway-restart ранее «up» core-сервис лежит; его зависимые слои рапортуют degraded/unreachable; или сервис, который «должен автостартиться», не в состоянии running post-reboot.
- **Превенция:** (1) Любой контейнер/сервис, обязанный переживать ребуты, ДОЛЖЕН иметь restart-политику (`--restart unless-stopped`) или enabled systemd-юнит с `Restart=always/on-failure`. (2) Добавить post-reboot health-пробу для core-сервисов (расширить `memory-gateway-health` cron: assert ALM `:3002` reachable + `vector_count > 0`, алерт ЗавЛабу если нет). (3) Перед объявлением «готов после ребута» — верифицировать, что каждый core-сервис реально up, а не предполагать.
- **См. также:** PAT-019 **I-20** (инстинкт зарегистрирован).
