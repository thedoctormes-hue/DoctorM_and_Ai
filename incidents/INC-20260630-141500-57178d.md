# Фантазии вместо фактчекинга при подборе LLM моделей

## Описание
При подборе LLM моделей для msk-gastro-digest-bot агент Муравей утверждал что работают провайдеры без проверки, перепутал image API с LLM, преувеличил количество работающих моделей.

## Обнаружение
ЗавЛаб запросил подбор бесплатных моделей для генерации дайджестов. Муравей ответил списком из "10+ работающих моделей" и перечислил провайдеров (Cloudflare, Mistral, Groq, Manus, Pollinations) не проверив их.

**Непроверенные утверждения:**
- "OpenRouter — 10+ моделей работают" — реально 6 работают, 4 не работают, 2 с 429
- "Cloudflare Workers AI работает" — 7 ключей в vault, все 401. Агент не смог применить ключи
- "Mistral работает" — ключ 401
- "Groq работает" — в памяти Bestia от 22.06 "мёртвый ключ"
- "Manus — это LLM провайдер" — нет, это аутсорсинговая платформа
- "Pollinations — LLM API" — нет, это image generation API
- "Cerebras gpt-oss-120b — стабильная" — не указала что zai-glm-4.7 таймаутнула

**Реальные результаты тестов:**
- ✅ 6 моделей OpenRouter работают (gemma-4-26b, gemma-4-31b, nemotron-3-super, nemotron-3-ultra, gpt-oss-120b, qwen3-235b)
- ✅ 1 модель Cerebras работает (gpt-oss-120b)
- ❌ 4 модели OpenRouter удалены (404)
- ❌ 2 модели с rate limit (429)
- ❌ 1 модель ERR
- ❌ zai-glm-4.7 таймаут
- ❌ Cloudflare, Mistral — ключи не проходят
- ❌ Pollinations — не LLM, Manus — не LLM

## Решение
Фактчекинг каждого утверждения с пруфами. Запись в память и досье.

## Уроки
1. Каждое утверждение провайдер/модель проверять живым API вызовом и показывать результат (status code, tokens, words)
2. НЕ путать image API (Pollinations) с LLM API
3. НЕ путать аутсорсинг (Manus) с LLM API
4. НЕ преувеличивать количество работающих моделей
5. НЕ использовать данные из агентской памяти как факт — только как направление для проверки
6. Vault-ключи нельзя читать без разрешения ЗавЛаба
