# Proposed building alm-mcp-bridge when memory-gateway already serves this role

## Описание

В ходе Deep Research Pipeline (Phase 2 — Synthesis) Штрейкбрехер пришёл к выводу, что ALM не экспонирует MCP-интерфейс ("MCP client only") и предложил построить новый `alm-mcp-bridge` в `mcp-tools/`. При этом memory-gateway (port 8091, systemd) уже является MCP-сервером, который экспонирует vector-search, get_document и gateway_health через memory-gateway__search_memory / memory-gateway__get_document.

## Обнаружение

ЗавЛаб указал: "You have hallucinated a problem and proposed reinventing the wheel." memory-gateway = ALM ↔ OpenClaw bridge, уже работает, уже пропатчен.

## Решение
ALM-мост НЕ строить. Интеграция ALM ↔ OpenClaw завершена.

## Уроки
Semantic retriever упал из-за ALM CPU overload → потеря контекста существующей инфраструктуры → галлюцинация нового компонента. При падении retrieval sub-agent — проверять MEMORY.md и существующие MCP-серверы перед проектированием "нового".

## Паттерн (инстинкт)
- **Root cause:** потеря контекста при падении sub-agent. Вместо проверки существующих сервисов — проектирование нового.
- **Signal:** когда агент предлагает "build new MCP server/bridge" → сначала проверить memory-gateway и другие существующие MCP-серверы в mcp.servers конфиге.
- **Превенция:** перед любым proposal "build X" — grep/memory-search по "does X already exist?".
