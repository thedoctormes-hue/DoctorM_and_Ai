# Ложная классификация «НЕ СУЩЕСТВУЕТ» для nginx-раздаваемого сервиса snablab-frontend

## Описание
В ходе аудита жизненного цикла агентов (2026-07-10/11) персонализированный starting-session агента bestia (и Мангуст в сводке) классифицировал snablab-frontend как «НЕ СУЩЕСТВУЕТ» (нет юнита systemd, нет docker-контейнера, нет порта). Это было ложью: фронт — живой статический SPA, развёрнутый в /var/www/snablab и отдаваемый nginx (конфиги snablab-e2e.conf на :8080 и snablab.shtab-ai.ru на :8443), API проксируется на snablab-backend :8200. Отдельного юнита/контейнера нет, но сервис СУЩЕСТВУЕТ и работает.

## Обнаружение
При повторном прогоне bestia (re-test) 2026-07-11 ~02:00, затем re-run ~06:5x. ЗавЛаб усомнился («snablab-frontend 100% уверен?») — сомнение вскрыло ошибку. Широкая проверка вживую доказала: фронт развёрнут и отдаётся nginx (HTTP 200 на :8080 и :8443), /var/www/snablab содержит собранный SPA.

## Решение
1. Исправлен bestia/startup-profile.md: snablab-frontend описан как живой nginx-served SPA; способ проверки — ls /var/www/snablab + grep nginx sites-enabled + backend :8200 up.
2. Расширено правило в starting-session SKILL.md (шаг 6): перед «НЕ СУЩЕСТВУЕТ» обязательно проверить ВСЁ (юнит + контейнер + порт + nginx-конфиг + деплой-директория + процесс). Отсутствие юнита/контейнера/порта в отдельности ≠ несуществование.
3. Расширенная перепроверка всех 8 профилей: остальные 3 претензии «не существует» (onnx-worker-8092, context-api, audit_lab.sh) подтверждены как ИСТИННЫЕ после широкой проверки. Других ложных негативов нет.
4. Перезапуск bestia доказал фикс: snablab-frontend: EXISTING/live, 5/5 живых.

## Уроки
- Отсутствие systemd/docker-юнита ≠ отсутствие сервиса. Веб-фронт может жить как статика под nginx без собственного процесса/порта.
- Проверка «существует ли сервис» должна охватывать юнит + контейнер + порт + nginx-конфиг + деплой-директория + процесс.
- При сомнении ЗавЛаба — проверять вживую, не оправдываться старым грепом.
