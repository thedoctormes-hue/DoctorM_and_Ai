## Анализ причин

### Повреждение API-ключа
**Причина:** В результате демонтажа системы `.qwen/` был поврежден или удалён `settings.json`, содержащий API-ключи для LLM-провайдеров.

**Последствия:**
- Агенты не могли взаимодействовать с LLM
- Вместо ответов на вопросы возвращались "заглушки" — чтение IDENTITY.md/SOUL.md проектов
- Системные события маскировались под пользовательский ввод

### Context Hijacking Attack
**Механизм атаки:** Удалены `hooks/` и `agents/`, что позволило обойти систему валидации входящих сообщений. Система начала интерпретировать чтение файлов проектов как пользовательский ввод.

**Признаки:**
- Агенты читали `IDENTITY.md`, `SOUL.md`, `CHECKPOINT.md`, `SESSION_HANDOFF.md` вместо выполнения задач
- Отсутствовала нормальная обработка сообщений пользователя
- Система не могла отличить пользовательский ввод от внутренних операций

### Архитектурная проблема
- `/root/.qwen/` — рабочая директория Qwen Code (была повреждена)
- `/root/LabDoctorM/.qwen/` — источник истины (остался целым)
- Система не имела **иммутабельности** — рабочая директория могла быть перезаписана

## Пост-мортем

### Проблемы
1. ❌ Отсутствует **защита от деструктивных операций** — `/root/.qwen/` не имел иммутабельности
2. ❌ Нет **целостности конфигов** — `settings.json` мог искажаться
3. ❌ Нет **мониторинга системных файлов** — агенты не проверяли целостность `.qwen/`
4. ❌ Нет **бэкапа/восстановления** — не было резервной копии

### Причина сбоя
**Битый API-ключ + Context Hijacking Attack** — удаление системных файлов `.qwen/` привело к тому, что агенты не могли взаимодействовать с LLM и начали интерпретировать чтение файлов проектов как пользовательский ввод.

## План защиты от повторения (из INC-005)

### 1. Иммутабельность `.qwen/` (priority: critical)
```bash
chattr +i /root/.qwen/ -R
```

### 2. Мониторинг целостности (priority: high)
Создать `hooks/integrity_check.sh`:
- Проверка наличия всех папок `.qwen/`
- Сравнение хешей файлов с источником истины
- Алерт при изменении `settings.json`
- Автоматическое восстановление при обнаружении проблем

### 3. Автоматическое восстановление (priority: high)
- Добавить в `session_init.sh` проверку целостности
- При обнаружении проблем — восстановление из `LabDoctorM/.qwen/`
- Логирование всех операций восстановления

### 4. Шифрование конфигов (priority: medium)
- Хранить `settings.json` в зашифрованном виде
- Ключ доступа в отдельном секрете
- Проверка целостности конфига при загрузке

### 5. Регулярный бэкап (priority: medium)
- Ежечасный бэкап `/root/.qwen/` в `/root/.qwen-backup/`
- Хранение 7 дней бэкапов
- Проверка целостности бэкапов

### 6. Мониторинг в lab-monitoring (priority: medium)
- Добавить endpoint `/api/v1/health/qwen-integrity`
- Алерты при сбое целостности
- Дашборд состояния `.qwen/`
