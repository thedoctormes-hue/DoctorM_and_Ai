# INC-20260710-onnx-42s-swap

**Дата:** 2026-07-10
**Серьёзность:** High (полная деградация скорости индексации ~20×, но не data-loss)
**Статус:** Resolved
**Автор:** Streikbrecher

## Симптом

Некоторые чанки при индексации эмбеддились 35–46с (эталон 1.8с). Казалось, что «сложные/
токсичные/длинные чанки» тормозят. Reindex полз со скоростью ~6 чанков/мин вместо ~120.

## Опровергнутые гипотезы (замером, не мнением — INC-047)

- «Токсичные тексты»: прямой замер обычных чанков `MEMORY.md` = **1.8с**, 0 deadlock, 0 FAIL.
- «Длинные чанки медленные»: HTTP-проба во время кризиса —
  короткий латинский (42с) = 34.0с, короткий кириллический (101с) = 32.0с,
  длинный латинский (1800с) = 21.2с, длинный кириллический (4000с) = 36.4с.
  → длина НЕ коррелирует с латентностью.
- «Конкуренция за CPU»: `renice -n 19` на reindex → латентность не изменилась (46.6с→46.6с).
- «Sequence length»: входной тензор воркера ВСЕГДА `(1,512)` (fixed), модель считает 512
  позиций независимо от длины текста → длина не может влиять.

## Реальный корень

Бродячий systemd-юнит **`onnx-test-8092.service`** (transient-тест, auto-restart,
счётчик рестартов = **9**) крутил ЕЩЁ один экземпляр `onnx-embedder.py` на порту **8092**,
пожирая ~1GB RAM. Это подтолкнуло систему (4 ядра / 7.9GB) в swap: веса трёх
продакшн-воркеров (8082/8084/8085) выгрузились в swap → каждый `session.run` дёргал
диск → 35–46с на ВСЕ чанки (не только «сложные»).

Все ранее убитые «осиротевшие» 4-е воркеры (pid 519521/522192/522580/523090/524588/...)
были РЕСПАУНАМИ именно этого тест-юнита — классическая спираль: auto-restart на
нестабильном процессе → постоянный лишний жрун RAM.

## Доказательства

- `free -m`: available 646MB, Swap 1269MB/4095MB USED, `vmstat` si/so активны (thrashing).
- 3 продакшн-воркера на 93–97% CPU; 4-й `python onnx-embedder.py` (pid 502918) ~12% MEM
  (~1GB), НЕ привязан ни к одному порту.
- `journalctl`: `onnx-test-8092.service: Scheduled restart job, restart counter is at 9`
  → `Started onnx-embedder.py` → pid 524588 на порту 8092.
- После `systemctl mask onnx-test-8092.service` + рестарта 3 воркеров:
  `ss -ltnp` → ровно 3 слушающих сокета (8082/8084/8085), `VmSwap ≈ 0`,
  latency 1.76/1.74с, `free -m` → SwapFree 2.3GB, available 1768MB.
- Steady-state latency 1.74/1.85/2.69с; throughput reindex вырос 6 → 158 чанков/мин.

## Фикс (применён)

```
systemctl disable --now onnx-test-8092.service
systemctl mask onnx-test-8092.service
systemctl restart onnx-embedder onnx-worker-8084 onnx-worker-8085
```

Плюс на всех продакшн-воркерах выставлен `Restart=no` (чтобы прерванный `systemctl restart`
из опросов шлюза не порождал осиротевшие процессы).

## Уроки

- «Медленные чанки» ≠ «сложные чанки». Мерай длину/контент vs латентность ДО вывода
  (INC-047). Здесь все гипотезы о контенте опровергнуты замером.
- Латентность воркера — честный индикатор swap. 1.8с = в RAM, 30с+ = swapped.
- Лишний ONNX-процесс на сервере с узкой RAM = tipping point в кризис. Любой stray-воркер
  маскировать сразу.
- `Restart=on-failure`/`on-abnormal` на воркерах при прерванных рестартах (SIGTERM от шлюза)
  = спираль осиротов. `Restart=no` безопаснее для инфраструктурных воркеров.
- Источник субсекундного орфан-шума (без порта/RAM) до конца не изолирован, но доказано
  безвреден — НЕ путать со спиралью `onnx-test-8092` (та была постоянной и ела ~1GB).
