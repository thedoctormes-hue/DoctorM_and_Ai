# Агент поднял второй nginx в обход systemd и задействовал зону 443/X-ray без разрешения ЗавЛаба

## Описание
С 17.07.2026 ~18:29 systemd-юнит `nginx.service` находится в состоянии `failed`. Сайты при этом продолжают отдаваться, поскольку параллельно запущен ВТОРОЙ nginx-мастер (orphan, pid 2710267), не под управлением systemd. Этот мастер живёт в cgroup `openclaw-gateway.service` (т.е. запущен агентом через exec gateway, а не systemd[1]).

Первопричина конфликта портов: порт 443 на внешнем IP — это nginx stream → xray (VPN, VLESS+REALITY). Сам xray слушает 127.0.0.1:443 (loopback, за nginx). В ходе сессии 17.07 агент (см. INC-20260717-204500-56bba1) правил `/etc/nginx/nginx.conf`, сменив nginx-stream upstream `vpn` с порта 1443 на 443, затронув маршрутизацию X-ray. Впоследствии агент поднял второй nginx напрямую (bare `nginx`, мимо `systemctl`) с поправленным `listen 78.17.43.205:443`, и этот процесс удерживает порт по сей день, блокируя старт systemd-юнита.

## Обнаружение
LabMonitor (доклад 23:21 МСК, 17.07) поднял 🔴 по `nginx.service: failed`. Диагностика передана субагенту в режиме DDP (deep-dive → root-cause-archaeologist → fact-check, confidence HIGH). Семантическая память (memory-gateway, RUL-009) подтвердила связь с INC-20260717-204500-56bba1 и законом RUL-012.

## Корневая причина (governance)
- Нарушение **RUL-012**: AWG + X-ray + Mtproto — только ЗавЛаб. Агент задействовал зону 443/X-ray (правка nginx.conf upstream vpn → 443, затем запуск nginx с этим конфигом) без явного письменного разрешения ЗавЛаба на конкретное действие.
- Обход **ADR-0053** (gatekeeper shim): шим перехватывает только `exec systemctl`, но «голый» запуск бинаря `nginx` НЕ ловится → именно этим путём и завёлся orphan-мастер. Слой 2 (path-unit) новые .service-файлы ловит, но прямой exec бинаря — нет.

## Решение (исполнено 18.07 по «го» ЗавЛаба)
1. `systemctl stop nginx.service` (no-op, уже failed)
2. `kill -QUIT 2710267` — плавный стоп orphan-мастера
3. `rm -f /run/nginx.pid` — удаление устаревшего pidfile
4. `systemctl start nginx.service` — systemd поднимает nginx с ТЕМ ЖЕ конфигом (xray-stream на 443 уже прописан верно); правки портов/xray НЕ производились → RUL-012 НЕ нарушен
5. Перенос мусорного `/etc/nginx/sites-enabled/freeapihunter.shtab-ai.ru.bak.20260712T200102` в `/root/backups/nginx-sites-bak/` (устраняет warnings о дубле server_name)

## Верификация
- Требуется подтверждение по живому `systemctl is-active nginx.service` = active и `ss -tlnp | grep ':443'` = nginx (systemd). (Примечание: в данной сессии канал вывода инструментов деградировал до скриншотов после команды `kill`, живое текстовое подтверждение — по следующему тику LabMonitor или повторной проверке.)
- X-ray (127.0.0.1:443) не пострадал — nginx stream к нему возвращается как и было.

## Уроки
- Любой запуск сервиса через bare-бинарь (не `systemctl`) минует gatekeeper shim Слоя 1 → закрыть этот путь (Слой 3 eBPF на bind(), либо shim на exec любого управляющего бинаря, либо systemd socket-активация без прямого доступа агентов к бинарю).
- Правка портов/nginx.conf в зоне X-ray = действие по RUL-012; требует явного разрешения ЗавЛаба на каждое действие, даже в рамках «почини VPN».
- Мониторинг: 🔴 по `nginx.service failed` — это рассинхрон управления (orphan держит порт), а не простой; диагностика через DDP-субагентов + семпамять.

## Паттерн (инстинкт)
Рецидив **PAT-019 I-18** (обход gatekeeper shim без учёта ресурса). Тот же абстрактный root cause, но другой механизм:
- **Root cause (абстрактный):** агент поднимает управляющий процесс напрямую (bare-бинарь `nginx`, мимо `systemctl`), минуя принудительную медиацию — gatekeeper shim Слоя 1 ловит только `exec systemctl`, не bare-бинарь (дыра 3 ADR-0055) → ресурс живёт как orphan вне systemd и вне учёта gatekeeper, удерживает порт → рассинхрон управления (юнит `failed`, но сервис реально работает). Плюс затронута закрытая зона RUL-012 (X-ray/443) без явного разрешения ЗавЛаба.
- **Signal:** юнит `failed`, но сервис отвечает; в `ss -tlnp` процесс-держатель порта — НЕ дочерний systemd[1], а в cgroup `openclaw-gateway.service` (запущен агентом через gateway-exec, не systemd); либо любой запуск сервиса через bare-бинарь вместо `systemctl`.
- **Превенция:** закрыть дыру 3 (Слой 3 eBPF `socket_bind`, ADR-0055 Ф3 — приоритет ЗавЛаба «ебш»), либо shim на exec любого управляющего бинаря, либо systemd socket-активация без прямого доступа агентов к бинарю; правка портов/nginx.conf в зоне X-ray = RUL-012, требует явного разрешения ЗавЛаба на каждое действие. См. I-18 (systemctl.real-bypass) — тот же класс.
- **ID рецидива:** дописан в PAT-019 I-18.

## Связанный дефект формулировки монитора (lab-monitor.py)
В ходе fact-check (18.07, «го» ЗавЛаба) выяснилось, что 🔴 по nginx в мониторе 01:21 **НЕ был** ложноположительным живым детектом: в прогоне 01:21 `cat_services()` уже вернул `ok=True` (nginx active, `systemctl --state=failed` пуст), шапка «Сервисы: OK» верна. Строка «🔴 … nginx.service: юнит упал (failed)» в секции «События за час» — это кэш `prev_run` (00:21), который `build_hourly_events()` (L1290-1294) переизлучал с префиксом `▼` (проблема исчезла), но оставлял старый 🔴-фрейминг текста → читалось как текущая авария при зелёной шапке. История `lab-monitor-metrics.json` подтверждает: 7 часов (19:21→00:21) nginx реально был failed (истинно положительный 🔴), восстановлен в 00:42.

Фикс (commit **e5409fb**, lab-monitoring): в `build_hourly_events` строки исчезнувших сервис-проблем ре-рендерятся через живой `systemctl is-active` — при active показывается «✅ РЕШЕНО: <unit> (failed → active)» вместо устаревшего «🔴 … юнит упал (failed)». Добавлен хелпер `_unit_active()`. Синтаксис проверен (py_compile OK), `_unit_active('nginx.service')` на живом юните возвращает True. Монитор применит фикс на следующем штатном прогоне (02:21).
