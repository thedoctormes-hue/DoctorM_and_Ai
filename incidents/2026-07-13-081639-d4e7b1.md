# Дефект дизайна: чеклист старта показывал не своё и не ту метрику

## Цель (ЗавЛаб, 2026-07-13)
При старте сессии агент должен вспомнить **СВОИ** коммиты/работу, которую
не довёл до конца, и доложить ЗавЛабу, чтобы добить. Словами ЗавЛаба:
«агент вспомнил!!! "ААА, сорян, я изменения внёс, а запушить забыл,
сейчас все проверю и исправлю"». И позже уточнено: «агент должен вспомнить
все коммиты, которые он не довёл до конца» — **просто и понятно**.

## Корень (два слоя)
1. **Показывал ЧУЖОЕ.** Старый чек (v3.3.0) был repo-wide сканом
   `git status -sb | grep ahead` по всем `projects/*` БЕЗ фильтра по автору.
   В shared-git (все агенты коммитят в одни репы) агент видел чужие
   незапушенные коммиты. Цель «вспомни своё» не достигалась.
2. **Неверная метрика.** Даже после agent-scoping (v3.3.1) базой было
   `@{upstream}..HEAD` — это ловит только «ahead of своего upstream» и
   ПРОПУСКАЕТ коммиты, запушенные в feature-ветку, но не слитые в main
   (там `ahead(upstream)=0`). А «не довёл до конца» = коммита нет в
   `origin/main`, независимо от того, локальный он или на feature-ветке.

## Доказательства (живьём, 2026-07-13)
- Глобальный скан: незапушенных коммитов автора `mangust` (ahead of upstream) — НОЛЬ,
  но агент при старте видел коммиты КотОлизатора и Совы — явно чужое.
- `lab-memory`: upstream СЛОМАН (`origin/master [gone]`), поэтому
  `ahead(upstream)` не считался, и 3 локально-сиротских коммита
  (LabDoctorM/agents@) вообще не попадали ни в какой простой алерт.
- `branch -r --contains` подтвердил: из 33 коммитов «ahead of origin/main»
  ровно 29 УЖЕ на remote-ветках (запушены, не слиты в main), и только 4
  реально нигде не запушены (1 × owl/SNZK + 3 × agents/lab-memory).
  То есть метрика «ahead of upstream» давала искажённую картину.

## Фикс (применён в starting-session SKILL.md, v3.3.1 → v3.3.2)
Шаг 2 переписан под простую цель «вспомнить свою незавершёнку»:
- Всё СКОПИРОВАНО на `AGENT_ID` (из IDENTITY.md `(id)` / `$AGENT_ID` /
  `$LAB_AGENT` / `basename "$PWD"`). Агент видит только своё.
- База сравнения — **`origin/main`** (а не `@{upstream}`). Для КАЖДОЙ
  локальной ветки агента: `git log origin/main..<ветка> --author="$AGENT_ID"`.
  Одним запросом ловит и локально-сиротские, и запушенные в feature-ветку,
  но не слитые в main.
- Плюс незакоммиченное (`git status -s`) — только в репо, где у агента
  есть своя ветка (чтобы не показывать чужой грязный tree, как 32 файла
  других агентов в DoctorM_and_Ai).
- Две корзины в напоминании: (а) незакоммиченное, (б)+(в) свои коммиты
  не в origin/main. Формат: «ЗавЛаб, осталась незавершённая работа: <список>.
  Сейчас разберусь в корне и доведу до main / исправлю.»
- Фолбэк: нет `origin/main` / сломан upstream — сравнивать с `@{upstream}`
  либо явный алерт «не могу сравнить с main».
- Красная линия «не коммитить/пушить автоматически при старте» сохранена.

## Open questions — ЗАКРЫТЫ (2026-07-13, v3.3.3)
- **AGENT_ID во всех workspace:** резолвится надёжно — приоритет
  `$AGENT_ID` env → `IDENTITY.md` поле `id:` → `basename "$PWD"`.
  Edge case `cwd ≠ workspace` ловится через env/IDENTITY, не только
  именем каталога.
- **Фолбэк при сломанном upstream:** реализован. База =
  `origin/main` → `origin/master` → `@{upstream}`; если ни одной нет —
  явный алерт «⚠️ НЕТ БАЗЫ», а не молчаливый провал (как было в v3.3.2).
- **Показывать ли незакоммиченное:** ДА, но только в репо, где у агента
  есть своя ветка (префикс `AGENT_ID/`), чтобы не шуметь чужим dirty-tree.
  Плюс добавлен под-чек (г) на коммиты под общим `agents@` (утечка
  атрибуции) — профилактика повтора инцидента.

## Resolution (2026-07-13) — два вопроса ЗавЛаба закрыты

### Q1. Переатрибуция коммитов общего аккаунта `agents@labdoctorm.ru`
**Корень:** глобальный default-identity `LabDoctorM <agents@labdoctorm.ru>`
(в `git config --global user.*` И в repo-local конфигах ряда реп). Агенты
иногда делали `git commit` мимо `lab-commit.sh` → коммит шёл под общим
аккаунтом. `lab-commit.sh` выставляет `<agent>@labdoctorm.ru`, поэтому
настоящий владелец определялся по префиксу ветки (`owl/`, `streikbrecher/`,
`mangust/`).

**Сделано:** 18 коммитов переписаны на верного автора через
`git rebase -x 'git commit --amend --author=...'` (только те, что были
`agents@`):
- owl ×4 ветки: `SNZK/owl/git-hygiene-2026-06-29`,
  `free-api-hunter/owl/git-hygiene-2026-06-29`, `mcp-tools/owl/git-env-fix`,
  `zprr-tracker/owl/git-hygiene-2026-06-29` → `owl@labdoctorm.ru`
- `lab-playwright-expert/mangust/parsing-orchestrator` (8 коммитов) →
  `mangust@labdoctorm.ru`
- `SNZK/streikbrecher/dream-integration` (6 коммитов) →
  `streikbrecher@labdoctorm.ru`

**Проверка:** финальный подсчёт `agents@` по ВСЕМ remote-веткам
(кроме main) + local-only = **0**. Все 18 на remote под верными авторами.

**Удержано (не трогал):** `lab-memory/main` — 9 коммитов `agents@`.
Причина: (а) сломан upstream `origin/master [gone]` (ветка `main` —
default, force-push к ней рискованный), (б) владелец неочевиден (reindex/
onnx инфра-коммиты).
**РЕШЕНИЕ (ЗавЛаб, 2026-07-13): оставить как есть.** Риск правки
(форс-пуш в ствол + неясный автор = искажение истории) не оправдан —
коммиты уже в main, код работает, страж новые `agents@` блокирует.

**Заметка по среде:** пуш `streikbrecher/dream-integration` падал с
`/bin/sh: 4: Bad substitution` — это post-шаг (обновление remote-tracking /
credential-store) ПОСЛЕ успешной заливки; сам пуш проходил. Реальный remote
подтверждён живым `ls-remote` = reattributed-версия `b4b4788`. Известный
env-баг (dash vs bash при git-over-SSH), отдельного разбора не требует
для закрытия этого инцидента.

### Q2. Запрет общего аккаунта — агенты ВСЕГДА под своим id
**Решение:** глобальный pre-commit хук-страж.
- Файл: `/root/LabDoctorM/projects/DoctorM_and_Ai/git-hooks/pre-commit`
  (`#!/bin/bash`, см. тело). Блокирует коммит, если author identity =
  `LabDoctorM <agents@labdoctorm.ru>` (проверка через `git var
  GIT_AUTHOR_IDENT`, чтобы ловить и env-переопределение из lab-commit.sh).
  Пропускает rebase/cherry-pick/merge (replay существующих коммитов).
  Цепляет репо-локальный `pre-commit` (GUARDIAN), чтобы не сломать
  репо-специфичные проверки.
- Включено глобально: `git config --global core.hooksPath
  /root/LabDoctorM/projects/DoctorM_and_Ai/git-hooks`.
- Тест: коммит от `agents@labdoctorm.ru` → `COMMIT REJECTED` (exit 1);
  коммит от `goodagent@labdoctorm.ru` → проходит, автор корректен.
- `lab-commit.sh` уже выставляет и author, и committer
  (`GIT_AUTHOR_*`/`GIT_COMMITTER_*`), поэтому штатный путь агента
  (через lab-commit.sh) стражем не блокируется.

**Итог:** любой `git commit` мимо `lab-commit.sh` под общим аккаунтом
теперь физически отклоняется — повторение утечки 21 коммита невозможно
без явного `git commit --no-verify` (и то только локально).
