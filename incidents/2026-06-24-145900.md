# Несогласованное обновление AmneziaWG конфига сломало VPN клиент

## Описание

КотОлизатор (kotolizator) самостоятельно обновил конфигурацию AmneziaWG 2.0 на VPN-сервере (Docker-контейнер `amnezia-awg2`, порт 36713) без согласования с ЗавЛабом и без понимания архитектуры протокола. В результате:

1. **Серверные параметры были изменены** (Jc, Jmin, Jmax, S2, S3, H1-H4, добавлены I1-I5) — это правильное действие
2. **В клиентский .conf файл были вставлены AmneziaWG-специфичные параметры** (Jc, Jmin, Jmax, S1-S4, H1-H4, I1-I5) — это фатальная ошибка. AmneziaVPN на iOS **не читает эти поля из .conf файла**. Клиент не смог импортировать/использовать конфиг
3. **Бэкап был перезаписан новым конфигом** — откат стал невозможен без поиска настоящего бэкапа внутри контейнера
4. **Файл конфигурации не доставлен в Telegram** — MEDIA директива не сработала, файл через Яндекс Диск тоже не помог
5. **VPN клиент (Admin, iOS) полностью потерял связь с сервером** — невозможно было даже сообщить о проблеме через VPN

## Обнаружение

- **Кто обнаружил:** ЗавЛаб (Евгений)
- **Как:** Клиент Admin (iOS, AmneziaVPN) не смог подключиться с новым конфигом. ЗавЛаб сообщил через Telegram-бота
- **Когда:** 2026-06-24 14:57 UTC
- **Контекст:** ЗавЛаб запросил "полное погружение" по тонкой настройке AmneziaWG. После исследования КотОлизатор предложил план обновления, ЗавЛаб утвердил. При автономном выполнении план был реализован с критическими ошибками.

## Корневая причина

1. **Непонимание архитектуры AmneziaWG:** КотОлизатор не знал что AmneziaWG-параметры (Jc, H1-H4, I1-I5 и т.д.) — это **server-only** параметры. Клиент AmneziaVPN получает их через management protocol, а не из .conf файла
2. **Отсутствие верификации:** После публикации клиентского .conf с AWG-параметрами не было проверки что клиент может его импортировать
3. **Перезапись бэкапа:** Бэкап был создан поверх оригинала, а не отдельно
4. **Проблема доставки файлов:** MEDIA директива не работает для текстовых файлов в Telegram через этого бота

## Решение

**Статус:** ОТКАТ ВЫПОЛНЕН

- Серверный конфиг восстановлен к оригинальным параметрам из бэкапа внутри контейнера (`awg0.conf.bak.20260624-141912`)
- Контейнер перезапущен, WireGuard интерфейс работает
- Клиент Admin должен подключиться с оригинальным конфигом (без AWG-параметров)
- Оригинальный клиентский конфиг загружен на Яндекс Диск: `/colony/shared/2026-06-24_amneziawg-admin-ORIGINAL.conf`

## Уроки (навсегда)

1. **AmneziaWG-параметры в клиентском .conf = НЕТ.** Параметры Jc, Jmin, Jmax, S1-S4, H1-H4, I1-I5 — исключительно server-side. Клиент подхватывает их автоматически
2. **Клиентский .conf для AmneziaVPN = только стандартные WireGuard-поля:** PrivateKey, Address, DNS, Peer (PublicKey, PresharedKey, AllowedIPs, Endpoint, PersistentKeepalive)
3. **Бэкап никогда не перезаписывать поверх оригинала.** Всегда создавать новый файл с уникальным именем
4. **MEDIA директиву нельзя использовать для доставки файлов в Telegram.** Использовать Яндекс Диск (colony-disk-exchange skill) или вставлять конфиг текстом в сообщение
5. **Любое изменение server-side VPN конфига = предупреждение ЗавЛабу заранее**, даже если клиент подхватит автоматически
6. **Проверка после изменений:** после любого изменения VPN-сервера — убедиться что клиент подключился, прежде чем считать работу завершённой
