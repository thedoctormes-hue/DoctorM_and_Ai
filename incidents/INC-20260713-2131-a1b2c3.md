# INC-20260713-2131-a1b2c3 — zprr-tracker: уязвимости Dependabot + нет тестов

## Обнаружено
2026-07-13 ~18:17 UTC, при мерже `owl/git-hygiene-2026-06-29` в main zprr-tracker
(коммит `61d6a31`). Найдено Совой в рамках accepting-work приёмки задачи «добей».

## Симптом
GitHub при push вернул уведомление:
> GitHub found 4 vulnerabilities on thedoctormes-hue/zprr-tracker's default branch (1 high, 3 low).

Пуш НЕ заблокирован (прошёл), но уязвимости остаются на default-ветке.
Точный список CVE — в GitHub Security / Dependabot alerts репозитория.

Дополнительно: в репозитории отсутствуют реальные тесты.
`tests/test_placeholder.py` содержит только `assert True`. Покрытие = 0.

## Точные CVE (fact-check через GitHub Dependabot API, 2026-07-13)
Все 4 OPEN-алерта относятся к `backend/requirements.txt` → пакет **python-multipart** (локально `==0.0.28`).

| # | CVE | Severity | Суть | fixed в |
|---|-----|----------|------|---------|
| 54 | CVE-2026-53539 | **HIGH** | Quadratic-time querystring parsing с `;` separators → CPU DoS | 0.0.30 |
| 53 | CVE-2026-53540 | LOW | Negative Content-Length в parse_form буферизует всё тело в памяти | 0.0.31 |
| 52 | CVE-2026-53538 | LOW | `;` как разделитель полей → parameter smuggling | 0.0.30 |
| 51 | CVE-2026-53537 | LOW | Content-Disposition smuggling via RFC 2231/5987 | 0.0.30 |

Чтобы закрыть ВСЕ 4 — нужно `python-multipart>=0.0.31` (рекомендую `==0.0.31`, как остальные pin в requirements.txt).
Пакет реально задействован: backend использует fastapi `Form()` (OAuth2 password flow) → python-multipart парсит form-data на эндпойнтах аутентификации.

## План закрытия (НЕ выполнен — за ЗавЛабом)
1. `backend/requirements.txt`: `python-multipart==0.0.28` → `==0.0.31`.
2. `./bin/lab-commit.sh owl -m "security(zprr-tracker): bump python-multipart 0.0.28->0.0.31 (CVE-2026-53539 high + 3 low)"`.
3. Push в main → GitHub автоматически переведёт 4 алерта в `fixed`.
4. Проверить, не сломал ли апдейт парсинг форм (semicolon-handling изменился) — smoke-тест логина.

## Связанные находки (DDP)

- `SESSION_HANDOFF.md` утверждает «pytest 138 passed / vitest 166 passed», но реально `tests/` содержит только `test_placeholder.py` (assert True). Документация ложна (contradiction/blindspot).

## Cross-check (research-агент, ЕБШ-fallback, 2026-07-13)
- CVSS (подтверждено GitHub Advisory / GitLab / NVD / SUSE): CVE-2026-53539 HIGH v3.1=7.5; CVE-2026-53538 LOW v3.1=3.7 (v4.0=6.3); CVE-2026-53537 LOW v3.1=3.7; CVE-2026-53540 LOW v3.1=3.7.
- Апдейт `python-multipart 0.0.28 → 0.0.31` **БЕЗОПАСЕН** (patch-уровень, API не удалён), НО 2 поведенческих breaking-изменения:
  1. `x-www-form-urlencoded`: `;` больше НЕ разделитель полей (только `&`). Ломает только нестандартных клиентов (вне WHATWG; браузеры шлют `&`).
  2. `multipart filename`: RFC 2231/5987 extended params (`filename*=`) игнорируются, авторитетен plain `filename` (RFC 7578). Риск: аплоад файла с не-ASCII именем ТОЛЬКО через `filename*` теряет имя.
  - `parse_form()` бросает `ValueError` на отрицательный `Content-Length` (фикс 53540).
- Рекомендация агента: бамп до `==0.0.31` + регресс-тест форм и аплоада файлов (вкл. unicode-имена). GHSA: 5rvq-cxj2-64vf, 6jv3-5f52-599m, vffw-93wf-4j4q, v9pg-7xvm-68hf.

- 1 high + 3 low уязвимости в зависимостях на продакшн-ветке.
- Нет тестов → регрессии не ловятся автоматически (мерж hygiene-ветки
  вообще никак не проверялся тестами).
