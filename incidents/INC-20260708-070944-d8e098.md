# docker-prune.service мёртв из-за осиротевшего пути после миграции инфраструктуры

## Описание
Системный юнит `docker-prune.service` (еженедельная очистка Docker, ADR-039) падал при каждом запуске по таймеру `docker-prune.timer` (каждое воскресенье 03:00 UTC) с кодом `203/EXEC`. Причина: `ExecStart` указывал на `/root/LabDoctorM/scripts/docker-prune.sh`, которого не существует — директория `/root/LabDoctorM/scripts/` была удалена при миграции инфраструктуры 2026-06-21 (инфра перенесена в `projects/DoctorM_and_Ai/`). Сам скрипт выжил по новому пути `/root/LabDoctorM/projects/DoctorM_and_Ai/scripts/docker-prune.sh`, но ссылка в юните осталась до-миграционной.

## Обнаружение
В ходе диагностики лаборатории (скил ЕБШ, тип `audit`, 2026-07-08) выявлено 2 failed-сервиса. `docker-prune.service` классифицирован как CRITICAL. Корень вскрыт через скил `root-cause-archaeologist`: сравнение путей `disk-monitor.service` (уже обновлён на новый путь, работает, status=0/SUCCESS) и `docker-prune.service` (пропущен при реконциляции).

## Решение
Применён прямой фикс (одобрено ЗавЛабом 2026-07-08 07:02 UTC):
1. Бэкап юнита: `/etc/systemd/system/docker-prune.service.bak.20260708T070454`
2. Исправлен `ExecStart` → `/root/LabDoctorM/projects/DoctorM_and_Ai/scripts/docker-prune.sh`
3. `systemctl daemon-reload`
4. Ручной `systemctl start docker-prune.service` → `status=0/SUCCESS`, юнит ушёл из `state=failed`
5. Таймер `docker-prune.timer` active, следующий прогон Sun 2026-07-12 03:00 UTC

Верификация: `systemctl status` показал `code=exited, status=0/SUCCESS`; `systemctl --state=failed` не содержит docker-prune. Docker не раздут (reclaimable 0B), прямого ущерба не было.

## Уроки
- Миграция инфраструктуры (2026-06-21) нарушила целостность ссылок: абсолютные пути в systemd-юнитах (`/etc/systemd`) не были реконцилированы полностью. `disk-monitor.service` поправили, `docker-prune.service` — нет. Корень: отсутствие системного обхода всех внешних ссылок при переезде каталогов.
- Нет алертинга на `systemctl --state=failed` — тихие падения юнитов остаются незамеченными неделями (корень Б, ~25%). Рекомендация: добавить проверку failed-юнитов в heartbeat.
- Документационный дрейф ADR-039: текст ADR до сих пор ссылается на старые пути `scripts/` (корень В, ~10%). Освежить при случае.
- Процесс: изменение инфра-конфига (systemd unit) требует одобрения ЗавЛаба (change-management / INC-026) — соблюдено.
