# ЗавЛаб приказал отключить таймеры реиндексации ALM — не выполнено

## Описание
ЗавЛаб дал команду «стоп» автоматике семантической памяти ALM (16.07), что должно было
означать отключение таймеров реиндексации. Фактически команда была исполнена как
`systemctl stop` (остановка сейчас), а не `systemctl disable` (выключение навсегда).
В результате `alm-sync-incremental.timer` остался в состоянии `enabled` и продолжил
авто-синхронизацию каждые 5 минут. Это противоречило утреннему докладу агента ЗавЛабу
(«автоматика выключена») и делало режим «ручного контроля» семпамяти иллюзорным:
изменения на диске сами падали в канон без review.

## Обнаружение
В ходе DDP-погружения в миссию контроля (16.07 ~16:00 МСК) верифицировано живьём:
`systemctl is-enabled alm-sync-incremental.timer` → enabled; last trigger 12:59 UTC;
хост не ребутился (boot с 09.07); скрипты запуска (start-anythingllm.sh, run-sync.sh)
таймер не трогают. Корень: исполнение «стоп» через `stop`, а не `disable`.

## Решение
Таймеры реиндексации отключены явной командой ЗавЛаба (16:19 UTC 16.07):
- `systemctl disable --now alm-sync-incremental.timer` → removed symlink,
  is-enabled=disabled, is-active=inactive.
- `alm-sync-rebuild.timer` уже был disabled (подтверждено).
- Итог: оба reindex-таймера disabled, сервис inactive.
Конфигурационный дефект устранён. Data harm отсутствовал (см. INC-2026-07-16-000300, закрыт).

## Уроки
- `systemctl stop` ≠ `systemctl disable`. Для гарантированного отключения автоматики по
  команде ЗавЛаба использовать `disable --now`.
- Не докладывать «выключено», пока не проверено `is-enabled` (а не только is-active).
- Директивы типа «стоп»/RUL исполнять буквально и верифицировать результат до отчёта.
