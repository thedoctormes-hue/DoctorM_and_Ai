# INC-20260716-231700-a1b2c3 — Дайджест ALM врал про «0 новых файлов»

## Что произошло
Часовой дайджест семпамяти ALM (`alm-trigger-audit` → `incremental_hourly_digest.sh`)
при любом прогоне с `ok=0` на ПОСЛЕДНЕЙ минуте часа выдавал
«Новых файлов загружено в индекс: 0», хотя за час инкремент реально загружал новые файлы.
ЗавЛаб справедливо усомнился («не может быть, что не было новых файлов»). Проверка лога
`incremental_report.log` показала реальные загрузки: 19:05Z ok=4, 19:10Z ok=7, 19:20Z ok=1
(12 новых за вечер). Отчёт врал.

## Root Cause
1. `incremental_hourly_digest.sh` брал `uploaded` из ПОСЛЕДНЕЙ строки лога (`tail -1` → `ok=`
   последнего прогона), а не суммировал `ok=` по всем прогонам окна 3600с. Если последний прогон
   часа случайно `ok=0` (типично, т.к. новые файлы уже подхвачены ранее), отчёт показывал 0 для всего часа.
2. Вторичный баг: метка окна «МСК» формировалась через `date` без `TZ=Europe/Moscow`; хост в UTC,
   поэтому показывалось UTC-время под видом МСК (сдвиг ~3 ч).

## Evidence
- `logs/incremental_report.log`: строки `... ok=4 ...`, `... ok=7 ...`, `... ok=1 ...` в окне 19:05–19:20Z.
- До фикса дайджест: «Новых файлов загружено в индекс: 0».
- После фикса (прогон 23:16 МСК): «Новых файлов загружено в индекс: 2»; ручная сумма `awk` по окну = `uploaded_last_hour=2`. Совпадает.
- Метка окна после фикса: «22:16–23:16 МСК» (корректно).

## Resolution
- `incremental_hourly_digest.sh`:
  - `uploaded` теперь = сумма `ok=` по всем строкам лога в окне 3600с (через `awk -v cut=... '$1>=cut {match(/ok=([0-9]+)/,a); s+=a[1]}'`).
  - метка окна формируется с `TZ=Europe/Moscow date ...` (принудительно МСК).
- Скрипт в `/root/LabDoctorM/.ops/shared/anythingllm-sync/` (не git-репозиторий, фикс на диске достаточен).

## Secondary observation (отдельная аномалия, НЕ часть этого инцидента)
Счётчик доков/векторов в отчёте ДОСТОВЕРЕН: bind-mount `/opt/anythingllm` = контейнерная БД,
хост читает живую БД. Зафиксировано реальное падение 1348→1327 доков (~21 док, ~164 вектора)
между 17:00 и 18:00 МСК. Проверено: `sync.py` не удаляет осиротевшие доки (только dedup по
basename при пере-загрузке). Причина удаления 21 док — вне нашего инкремент-синка; требует
отдельного расследования (возможно удаление исходников или sweep в ALM).

## Уроки (Lessons)
- «0 новых файлов» в отчёте ≠ «индекс не менялся». Отчёт должен суммировать метрику по окну, а не брать последнюю точку.
- Часовые пояса в отчётах всегда фиксировать явно (`TZ=`), не полагаться на TZ хоста.
- Любая метрика, агрегируемая из лога, должна проверяться на тестовых данных с известным результатом.

## Паттерн (инстинкт) — выжато по Bole-принципу
- **Root cause (абстрактный):** метрика, читаемая из лога/потока, трактуется как последняя точка
  вместо агрегата по окну; временная метка не фиксирует TZ явно → отчёт врёт при реальной активности
  (ложный «0» или сдвиг времени).
- **Signal (триггер для агента):** «отчёт показывает 0 / N при живой системе или противоречит
  ожиданию → НЕ верь последней строке лога; проверь (а) агрегацию по окну, (б) явный TZ в метке».
- **Применение:** любой скрипт, читающий метрики из логов (health-чеки, дайджесты, счётчики загрузок)
  — агрегировать по окну, TZ= явно. Рецидивы этого паттерна (PAT-05 и др.) сливаются в ОДИН паттерн,
  а не регистрируются отдельно.
