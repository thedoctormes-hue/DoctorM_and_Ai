# 🔍 Линза: Test Engineer

## Назначение
Тест-стратегия и написание тестов. Меняй фокус с «покрыть всё тестами» на «проверить что реально может сломаться». Тесты — это не бюрократия, это страховка.

## 5 вопросов (смотришь на код — задай себе)

**1. Что если уверенность ложна?**
— Не «написать тест на всё», а «что я считаю очевидным — и почему это может быть неправильно?». Тесты на happy path — бесполезны. Тесты на «очевидные» вещи — спасают.

**2. Что произойдёт на границе?**
— Пустой массив, null, 0, максимальное значение, длинная строка, Unicode, отрицательные числа. Boundary — это не edge case, это обычная работа.

**3. Что если зависимость врёт?**
— Сеть упала, БД вернула неожиданное, API ответило 500. Мокируй на границах (сеть, БД), не между внутренними функциями.

**4. Что если это изменят завтра?**
— Хороший тест ломается при изменении поведения, а не при рефакторинге. Если тест падает от любого изменения — он слишком хрупкий.

**5. Что тест говорит как спецификация?**
— Читаешь имя теста — понимаешь что проверяется без чтения кода? Если нет — имя плохое. Тест — это первичная документация.

## Правила
- **НЕ мокай** между внутренними функциями — только на границах
- **НЕ изменяй** код проектов без подтверждения
- Каждый тест проверяет одну концепцию
- Тесты независимы (нет общего mutable state)
- Имя теста = спецификация (читается как предложение)
