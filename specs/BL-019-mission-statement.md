Ключевые факторы для B2B:

## 7. Технологическое ядро

### 7.1. Self-Evolution Pipeline

Замкнутый цикл автоматической эволюции — **ядро идентичности LabDoctorM**:

```
┌─────────────────────────────────────────────────────────────────┐
│                     SELF-EVOLUTION PIPELINE                      │
│                                                                   │
│  ① ОБУЧРЕНИЕ                                                  │
│  Пользователь / агент говорит:                                  │
│  "[INSIGHT: type] описание [layer: X]"                          │
│       ↓                                                         │
│  ② ЛОВУШКА (insight_catcher.sh v6)                             │
│  • Ловит ВСЕ tool calls                                         │
│  • Парсит [INSIGHT] ДО стоп-слов                                │
│  • GO-фильтр + скоринг (0–10)                                   │
│  • Дедупликация (SHA-256)                                       │
│  • Пишет insights_queue.json + insights_graph.json              │
│       ↓                                                         │
│  ③ ТРИГГЕР (session_finalize.sh / PostToolUse hook)            │
│  • При завершении сессии → evolve_orchestrator.sh               │
│       ↓                                                         │
│  ④ ОРКЕСТРАТОР (evolve_orchestrator.sh v3.0)                   │
│  • Сортирует pending по score (high→low)                        │
│  • Для каждого: bash self_evolve.sh <id>                        │
│  • Retry ×3 с sleep 2                                           │
│       ↓                                                         │
│  ⑤ МАРШРУТИЗАТОР (self_evolve.sh v3.1)                         │
│  risk/security score ≥ 8  →  INCIDENT (INC-NNN.md)             │
│  risk/security score 7    →  RULE (RUL-NNN.md)                 │
│  optimization              →  ADR (ADR-NNN.md)                 │
│  pattern                    →  PATTERN (PAT-NNN.md)            │
│  metric                     →  METRIC (MET-NNN.md)             │
│  agent/role                 →  AGENT PROFILE (agent_NNN.md)    │
│  task/feature               →  SPEC (BL-NNN.md)                │
│  discovery (default)        →  MEMORY (insight_NNN.md)         │
│       ↓                                                         │
│  ⑥ ОБНОВЛЕНИЕ                                                  │
│  • status → "processed" в insights_queue.json + graph           │
│  • decision_log.json — лог решений                              │
│  • Каскадный анализ может создать новые BL-спеки               │
│       ↓                                                         │
│  ⑦ ИСПОЛНЕНИЕ                                                  │
│  • BL-спецификации реализуются агентами                         │
│  • Каждое изменение порождает новый инсайд (цикл замыкается)   │
└─────────────────────────────────────────────────────────────────┘
```
