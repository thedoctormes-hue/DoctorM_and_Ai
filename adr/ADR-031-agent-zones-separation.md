# ADR-031: Разделение проектов и агентов в myrmex.json

## Контекст
В текущей архитектуре `myrmex.json` содержит две сущности в одном массиве `projects`:
1. **Реальные продукты и сервисы** (например, `lab-monitoring`, `myrmex-control`) — сканируются `project-indexer` каждые 5 минут
2. **Рабочие зоны агентов** (каталоги в `workspaces/`) — ошибочно включаются в `projects` из-за исторических причин

Это приводит к проблемам:
- Ложные срабатывания `project-indexer` на рабочие каталоги агентов
- Сложность интерпретации данных: unclear, является ли запись продуктом или зоной агента
- Потенциальная перезапись рабочих каталогов при обновлении `myrmex.json` через API
- Нарушение принципа "единой источники правды" — рабочие зоны не должны управляться через `myrmex.json`

## Решение
Добавить явное разделение в структуру `myrmex.json`:
1. Оставить массив `projects` **только для реальных продуктов и сервисов**
2. Добавить новый массив `agent_zones` для рабочих каталогов агентов
3. Оба массива будут полностью перезаписываться при обновлении через API
4. В `agent_zones` хранить минимальный набор полей: `id`, `path`, `agentId`, `lastSeen`

### Пример структуры после изменений:
```json
{
  "_meta": { "version": "2", "updatedAt": "2026-06-18T06:00:00Z" },
  "projects": [
    {
      "id": "lab-monitoring",
      "name": "Lab Monitoring",
      "owner": "zavlab",
      "createdAt": "2026-01-15T00:00:00Z",
      "updatedAt": "2026-06-10T12:30:00Z"
    }
  ],
  "agent_zones": [
    {
      "id": "kotolizator-hyperion-scout",
      "agentId": "kotolizator",
      "path": "/root/LabDoctorM/workspaces/kotolizator/hyperion-scout",
      "lastSeen": "2026-06-18T05:55:00Z"
    }
  ]
}
```

## Последствия
### ✅ Позитивные последствия
- Четкое разделение ответственности: `projects` — каталог продуктов, `agent_zones` — каталог рабочих зон
- Устранение ложных срабатываний `project-indexer` на рабочие каталоги
- Возможность безопасно обновлять `myrmex.json` через API без риска затереть рабочие зоны
- Улучшение производительности `project-indexer` (меньше элементов для сканирования)
- Соответствие факту: рабочие зоны агентов управляются через Git worktrees, а не через myrmex API
- Подготовка к будущим функциям: агент-специфичные настройки, мониторинг загрузки зон

### ⚠️ Негативные последствия / компромиссы
- Требуется обновление всех потребителей `myrmex.json`:
  - `project-indexer` — игнорировать `agent_zones`
  - Agent onboarding/offboarding скрипты
  - Дашборды и мониторинг, которые показывают "все сущности"
- Необходимость миграции существующих данных:
  - Перенести рабочие зоны из `projects` в `agent_zones`
  - Очистить `projects` от записей, соответствующих workspaces/
- Потенциальная временная несовместимость во время деплоя

### 📋 Что нужно сделать
1. [ ] Обновить API endpoints (`/api/projects`, `/api/agents`) для работы с новой структурой
2. [ ] Изменить `project-indexer` (в `.qwen/scripts/project_indexer.sh`) — фильтровать по типу сущности
3. [ ] Написать скрипт миграции существующего `myrmex.json`:
   - Выделить записи с путями, начинающимися с `/workspaces/` → переместить в `agent_zones`
   - Оставить остальные в `projects`
4. [ ] Обновить документацию:
   - `SOUL.md` — раздел "Структура `myrmex.json`"
   - Руководство по наboarding агентов
5. [ ] Добавить валидацию в API: запретить запись в `agent_zones` через обычные endpoints (только через специализированные)

## Альтернативы
| Вариант | Плюсы | Минусы | Почему отклонён |
|---------|-------|--------|-----------------|
| Сохранить текущую структуру | Нет изменений, обратная совместимость | Накапливание технического долга, риск рассинхронизации, confusion в интерпретации данных | Не решает системную проблему, противоречит факту что рабочие зоны не сканируются |
| Добавить поле `type: "project"\|"agent_zone"` в существующий массив | Минимальные изменения структуры, одна точка истины | Все равно нужно фильтровать по типу, риск забыть фильтр, усложнение запросов | Неявное разделение менее надёжно, чем явные массивы |
| Хранить agent_zones в отдельном файле (например, `agent_zones.json`) | Полная изоляция сущностей | Нарушение принципа "единой источники правды", требует синхронизации двух файлов, увеличение сложности | myrmex.json должно оставаться единым источника для всей лабораторной инфраструктуры |
