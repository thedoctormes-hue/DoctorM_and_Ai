# ADR-022: Стратегия монорепозитория LabDoctorM

## Статус

Принято

## Контекст

LabDoctorM содержит множество проектов (боты, мониторинг, VPN, парсеры, фронтенды,
тулинг агентов), которые исторически развивались в одном репозитории. Возник вопрос:
разбивать на отдельные репозитории или оставить монорепо?

## Решение

Оставить стратегию **монорепозитория** для текущего масштаба.

## Обоснование

**Плюсы монорепо для LabDoctorM:**
- Единый git history — логика агентов, тулинг и проекты в одном месте
- Агенты-лаборанты работают через worktrees одного репозитория
- Общая инфраструктура (scripts/, adr/, docs/, .qwen/) — одна точка истины
- Простота бэкапа и восстановления

**Минусы и риски:**
- Рост размера репозитория (особенно фронтенды с node_modules)
- CI/CD усложняется при росте (нужно фильтровать по paths)
- Путаница веток между агентами (частично решена ADR-012)
- Сложность разделения прав доступа

**Когда пересмотреть:**
- Когда время CI/CD превысит 5 минут
- Когда появятся внешние контрибьюторы, которым не нужен весь код
- Когда размер .git превысит 500MB

## Последствия

- Все проекты остаются в `projects/`
- Каждый проект имеет свой `IDENTITY.md`
- Общая инфраструктура (scripts, adr) — в корне
- При необходимости выделения — использовать `git subtree split` или `git filter-branch`
