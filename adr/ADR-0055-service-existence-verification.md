# ADR-0055: Верификация существования сервисов агентами

## Статус
Accepted (2026-07-11)

## Контекст
В ходе аудита жизненного цикла агентов (2026-07-10/11) выявлен дефект: агенты классифицировали snablab-frontend как «НЕ СУЩЕСТВУЕТ», проверяя только systemd/docker/ss. Фронт оказался живым — статический SPA, отдаваемый nginx из /var/www/snablab (конфиги snablab-e2e.conf :8080 и snablab.shtab-ai.ru :8443), API → snablab-backend :8200. Узкая проверка породила ложный негатив (по сути — «фантазия вместо фактчекинга»).

## Решение
Закрепить в starting-session (и downstream-скиллах) два жёстких правила:
1. НЕ СУЩЕСТВУЕТ ≠ DOWN. «DOWN» ставится только если сущность СУЩЕСТВУЕТ (юнит/контейнер/процесс есть), но failed/inactive или порт не отвечает. Отсутствие сервиса — отдельное состояние, не путать с падением.
2. Многоисточниковая верификация перед «НЕ СУЩЕСТВУЕТ». Прежде чем утверждать, что сервиса нет, проверить ВСЕ: (a) systemctl list-units --all; (b) docker ps -a; (c) ss -ltn (порт); (d) grep -rl <name> /etc/nginx/sites-enabled; (e) директория деплоя/статики из конфига; (f) ps aux | grep <name>. «НЕ СУЩЕСТВУЕТ» только если нет во ВСЕХ источниках.

Профили агентов (startup-profile.md) должны указывать правильный способ проверки для nginx-раздаваемых компонентов (путь деплоя + nginx-конфиг), а не только юнит/порт.

## Последствия
- Положительные: агенты перестанут ложно заявлять об отсутствии nginx-раздаваемых сервисов; состояние «НЕ СУЩЕСТВУЕТ» не смешивается с «DOWN».
- Стоимость: проверка чуть дольше (доп. grep nginx + ls деплоя).
- Риски: если профиль устарел и указывает неверный способ проверки — агент всё равно может ошибиться; нужна периодическая валидация профилей против живых сервисов.

## Реализация
- starting-session SKILL.md шаг 6 — правила добавлены (v3.1.0 + hard rule).
- bestia/startup-profile.md — фронт описан как живой nginx-SPA.
- Расширенная перепроверка всех 8 профилей — ложных негативов больше нет (только snablab-frontend, исправлен).
