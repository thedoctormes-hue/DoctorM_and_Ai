# ADR-039: Политика ротации и очистки дискового пространства

## Контекст

Диск лаборатории заполнен на 77% (44G/59G). Анализ показал отсутствие единой политики ротации и очистки. Каждый сервис настроен в вакууме — кто пишет на диск, но никто не убирает за собой.

**Выявленные генераторы мусора:**
- `/tmp` — 346M, 23,448 файлов старше 1 дня (systemd-tmpfiles-clean не настроен)
- `/var/backups/postgresql/` — 495M, 122 файла без ротации
- Docker — 5.2G (2.2G образов + 448M томов + 2.9G containerd)
- `/var/log/journal/` — 273M (нет лимита)
- `/root/LabDoctorM/.git/` — 805M (не существует на момент выполнения — удалён ранее)

**Корневая причина:** отсутствие централизованной политики ротации. Каждый сервис настроен в момент создания, никто не подумал о накоплении мусора.

## Решение

Установить стандарт: **при создании любого сервиса — сразу настроить ротацию его артефактов**.

### Конкретные меры

1. **/tmp — автоматическая очистка:** `D /tmp 1777 root root 10d` в tmpfiles.conf
2. **Journald — ограничение:** `SystemMaxUse=100M`, `SystemMaxFileSize=50M`, `MaxRetentionSec=7day`
3. **Docker — еженедельная очистка:** `docker image prune -a -f && docker volume prune -f && docker container prune -f`
4. **PostgreSQL бэкапы — полная ротация:** все БД, не только snablab
5. **Git — регулярное обслуживание:** `git gc --aggressive` для каждого репозитория в projects/
6. **Мониторинг:** проверка диска каждый heartbeat, алерт при >80%

**Итого освобождено:** ~2.5G

## Автоматизация (хвосты)

### Еженедельный Docker prune
- Скрипт: `/root/LabDoctorM/scripts/docker-prune.sh`
- systemd timer: `docker-prune.timer` (каждое воскресенье 03:00 UTC)
- Удаляет: остановленные контейнеры >7 дней, неиспользуемые образы, dangling volumes

### Мониторинг диска
- Скрипт: `/root/LabDoctorM/scripts/disk-monitor.sh`
- Cron: каждые 6 часов
- Алерт WARNING: >80%, CRITICAL: >90%
- Лог: `/var/log/disk-monitor.log`
- Мониторит: /tmp, /var/backups, journal, docker reclaimable

## Последствия

**Положительные:**
- Диск: 77% → 74% (44G → 41G использовано)
- Автоматическая предотвращение накопления мусора
- Снижение риска отказа сервисов из-за заполненного диска

**Риски:**
- При удалении бэкапов — потеря старых резервных копий (митигация: 7 дней хранения достаточно)
- При Docker prune — удаление неиспользуемых томов (митигация: проверка перед первым запуском)
