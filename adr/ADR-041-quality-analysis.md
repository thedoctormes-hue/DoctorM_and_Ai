# ADR 041: Анализ качества работы агента — фактчекинг

**Дата:** 2026-06-20
**Автор:** kotolizator (по запросу ЗавЛаба)
**Статус:** reference

## Контекст

ЗавЛаб отметил высокое качество работы в сессии 2026-06-20 и попросил найти корневую причину с фактчекингом.

## Факты по сессии

### Что было сделано:
- Полный аудит конфигурации 8 агентов
- 5 улучшений (ADR 039) — внедрены, протестированы, задокументированы
- 3 рекомендации по тонкой настройке (ADR 040) — внедрены, протестированы, задокументированы
- 33 теста написаны, все проходят
- 5 коммитов в main без конфликтов

### Что пошло не так:
- startupContext не применился при первом патче (ошибка в Python-коде)
- 16 тестов упали из-за использования `d` вместо `self.defaults`
- Gateway рестарт сбросил startupContext
- streikbrekh создан gateway при reindex, удалён повторно

## Анализ корневых причин качества

### 1. Чёткая постановка задачи (факт)
- ЗавЛаб дал конкретную цель: «Полное погружение в конфигурацию»
- Ограничения ясны: «Только факты, никаких фантазий», «Только списки»
- Формат ответов задан: списки, структурированный текст

**Вывод:** Качество входных данных = качество выходных.

### 2. Следование документации (факт)
- Все 8 рекомендаций (5 + 3) подтверждены документацией OpenClaw
- Ни одного вывода без ссылки на docs.openclaw.ai
- Значения параметров взяты из config-agents.md

**Вывод:** Работа по инструкции вместо импровизации = предсказуемый результат.

### 3. Контроль качества на каждом шаге (факт)
- ЗавЛаб запросил проверку → найдены 2 ошибки → исправлены
- ЗавЛаб попросил объяснить maxConcurrent → разобрали
- ЗавЛаб попросил задокументировать → создан ADR

**Вывод:** Итеративная проверка ловит ошибки до того как они накапливаются.

### 4. Инструменты работали стабильно (факт)
- exec, edit, write, git — все отвечали
- pytest проходил после исправлений
- git push без конфликтов

**Вывод:** Стабильная инфраструктура = меньше отвлечений.

### 5. Тесты как страховка (факт)
- 33 теста покрывают все изменения
- При каждом изменении конфига — прогон тестов
- Ошибки найдены и исправлены до коммита

**Вывод:** Тесты ловят регрессии автоматически.

## Корневая причина

**Системный фактор:** ЗавЛаб выстроил процесс — я выполняю инструкции.

**Ключевые факторы качества:**
1. Чёткая постановка задачи с ограничениями
2. Следование документации вместо импровизации
3. Итеративная проверка на каждом шаге
4. Тесты как страховка от регрессий
5. Стабильная инфраструктура

**Что НЕ является причиной:**
- «Талант» или «интуиция» агента — все решения из документации
- «Креативность» — задача была алгоритмической
- «Опыт» — я не накапливаю опыт между сессиями

## Рекомендации для воспроизводимости

1. Давать конкретные задачи с ограничениями
2. Запрашивать фактчекинг для каждого решения
3. Проверять результат на каждом шаге
4. Писать тесты при каждом изменении
5. Документировать в ADR при завершении
