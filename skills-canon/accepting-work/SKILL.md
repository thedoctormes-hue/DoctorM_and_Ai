# Agent Acceptance Protocol

Обязательный протокол приёмки работы AI-агента в лаборатории.
Агент НЕ может заявить «задача выполнена» без прохождения всех 5 шагов (+ шаг 6 при завершении сессии).

## Обязательный триггер

**«Сдай работу»** — фраза, которая запускает этот скил.
Агент обязан применить протокол когда пользователь говорит «сдай работу».

## Когда применять

- Пользователь говорит «сдай работу»
- При завершении любой задачи по написанию/изменению кода
- При завершении настройки/конфигурации
- При любой работе, результат которой попадает в git-репозиторий
- При завершении сессии (если были изменения в коде)

## Границы применимости
- Не заменяет code review — это протокол само-приёмки агента ПЕРЕД сдачей
- Не выполняет автоматическое исправление тестов/линтера — только проверяет и сообщает
- Не работает вне git-репозиториев (шаг 3 пропускается с сообщением)
- Не применим к задачам без кода (документация, аналитика) — для них режим без кода
- Не управляет git-ветками/стратегиями — только коммит через lab-commit.sh
- Не сканирует секреты в истории — только в staged изменениях

## Чек-лист качества
- [ ] Все 5 шагов прошли или имеют обоснование пропуска
- [ ] Тесты: команда + результат + coverage (или smoke-test для мелких правок)
- [ ] Anti-Default UI Check пройден (фронтенд) или N/A
- [ ] Линтер + typecheck: команда + результат (или skip с причиной)
- [ ] Regression: полный прогон / smoke-test / пропущен с обоснованием
- [ ] Документация: список обновлённых файлов (или N/A)
- [ ] Git: коммит через lab-commit.sh, security clean, секреты не в tracked
- [ ] Хвосты: временные файлы удалены, worktree убран, бэклог записан
- [ ] Отчёт в формате выше предоставлен ДО объявления «готово»
- [ ] При завершении сессии: finishing-session запущен

## Анти-паттерны
- ❌ «Задача выполнена» без прохождения accepting-work — INC-020, INC-029
- ❌ Пропуск шага 1 (тесты) «потому что мелочь» — INC-016
- ❌ Пустой коммит или коммит без сообщения — INC-022
- ❌ Секреты в staged/committed — INC-013, INC-015, INC-016
- ❌ Placeholder-комментарии (// TODO, // rest of) в продакшн-коде — INC-018
- ❌ Default UI (purple gradients, centered hero, Loading... текст) — PAT-006
- ❌ Неатомарные коммиты (смешивание рефакторинга и фичи)
- ❌ Забытый worktree / временные файлы — грязный git status
- ❌ Завершение сессии без finishing-session — потеря памяти/инцидентов

## Режим без кода

Если задача НЕ связана с кодом (аналитика, исследование, документация без git):
- Шаг 1 (тесты) — пропустить
- Шаг 3 (гит-процедуры) — пропустить
- Шаги 2, 4, 5 — выполнить полностью
- В отчёте указать: «Без кода — шаги 1,3 пропущены»

## 5 обязательных шагов (+ шаг 6 при завершении сессии)

### 1. Покрыть тестами + линтер + typecheck

- Все новые/изменённые модули покрыты unit-тестами
- Для AI-генерированного кода: ≥85% покрытие
- Для критических путей (auth, payments, data): ≥85% всегда
- **Для мелких исправлений** (1-5 строк, комментарии, форматирование, опечатки): ≥60% допустимо, если изменение не влияет на логику. При этом smoke-тест (запуск модуля без ошибок) обязателен.
- Edge cases покрыты: граничные значения, ошибки, конкурентный доступ
- Тесты запущены и проходят (exit code 0)
- В отчёте: команда + результат (X passed, Y failed, Z% coverage)
- Нет placeholder-комментариев: "// ... rest of ...", "// TODO: implement", "// similar to above"
- Все импорты из существующих зависимостей (проверить через grep в package.json)

**Линтер и typecheck (после тестов):**

```bash
# TypeScript — typecheck
npx tsc --noEmit 2>&1

# Python — линтер
ruff check . 2>&1 || npx ruff check . 2>&1

# JS/TS — линтер (если есть скрипт в package.json)
npm run lint 2>&1
```

Запускать только если проект соответствующего типа (наличие tsconfig.json, pyproject.toml, package.json).

**Regression-проверка:**

- Для стандартных изменений: запустить ВСЕ тесты проекта, не только изменённые модули
- Для мелких изменений (1-5 строк, форматирование, опечатки): smoke-test достаточен (импорт модуля + вызов ключевой функции)
- Если полный прогон тестов занимает >60s — можно ограничиться изменённым пакетом/модулем, но зафиксировать это в отчёте

### Anti-Default UI Check

Перед сдачей фронтенд-задачи проверь:

- Нет purple gradients (from-purple-*, to-purple-*, from-violet-*, to-violet-*)
- Нет centered hero с заголовком + подзаголовком + CTA как единственного контента страницы
- Нет всех кнопок одного цвета (primary/secondary/destructive различимы)
- Нет "Loading..." текста вместо Skeleton/Spinner компонента
- Нет "No data" / "Нет данных" текста вместо EmptyState компонента

### 2. Актуализировать документацию

- README/документация обновлена под изменения
- Новые модули документированы
- Нетривиальные решения объяснены (почему так, а не иначе)
- Комментарии в коде актуальны и отражают реальную логику
- В отчёте: список обновлённых файлов

### 3. Выполнить гит-процедуры

**Предварительная проверка:**

```bash
# Проверить что git установлен
if ! command -v git &>/dev/null; then
  echo "[skip] Git не установлен — шаг 3 пропущен"
  # Переход к шагу 4
fi

# Проверить что мы в git-репозитории
if ! git rev-parse --is-inside-work-tree &>/dev/null; then
  echo "[skip] Не в git-репозитории — шаг 3 пропущен"
  # Переход к шагу 4
fi
```

**Git не установлен или не в репозитории → пропустить шаг 3, сообщить в отчёте.**

- 3a. Закоммитить изменения через `lab-commit.sh`:
  ```bash
  # Для инцидентов (DoctorM_and_Ai):
  /root/LabDoctorM/projects/DoctorM_and_Ai/scripts/lab-commit.sh <агент> -m "сообщение"
  # Для кабинетов (проекты):
  /root/LabDoctorM/projects/<кабинет>/bin/lab-commit.sh <агент> -m "сообщение"
  ```
- 3b. Проверить `git status` — чистый
- 3c. Security-проверка (вместо мусорной «гигиены репозитория»):
  ```bash
  # Проверить что нет новых секретов в staged-файлах
  git diff --cached | grep -iE '(password|secret|token|api.?key)' && echo "[warn] Найдены потенциальные секреты!" || echo "Security: clean"
  ```
- Коммиты атомарны, сообщения осмысленные
- .env и секреты НЕ попали в коммит
- В отчёте: хеши коммитов, список файлов, статус пуша

**Обработка ошибок шага 3:**
- `git` не найден → пропустить шаг 3, сообщить в отчёте
- `git rev-parse` вернул ошибку → мы не в репозитории, пропустить шаг 3, сообщить
- `lab-commit.sh` не найден → искать: `find /root/LabDoctorM/projects -name lab-commit.sh -type f`
- Секрет в staged → убрать из индекса, добавить в .gitignore, зафиксировать
- Нечего коммитить → не создавать пустой коммит

### 4. Прибрать все «хвосты»

- Временные файлы, debug-логи, артефакты сборки — удалены
- Worktree убран (если создавался)
- Мёрж-конфликты разрешены
- Незавершённые задачи — записаны в бэклог
- В отчёте: список удалённых файлов + бэклог

### 5. Отчитаться

- Отчёт предоставляется ДО объявления о завершении задачи
- Для мелких исправлений (1-2 строки) допускается сокращённый отчёт: тесты + гит

### Шаг 6: Если сессия завершена

После успешной приёмки задачи, если сессия завершена:
- Запусти **finishing-session** для записи итогов в память и проверки инцидентов
- finishing-session запишет результаты в `memory/YYYY-MM-DD.md` и обновит `MEMORY.md`
- Это гарантирует полное закрытие сессии: и код принят, и контекст сохранён

## Формат отчёта

```
## ✅ Задача выполнена

**1. Тесты**
<команда> → <результат>, coverage: X%
- Anti-Default: пройден / не пройден (что найдено)
- Placeholder-комментарии: нет / найдены (файлы)
- Lint: проходит / ошибки (количество)
- Typecheck: проходит / ошибки (количество)
- Regression: passed / failed / skipped (smoke-test для мелких правок)

**2. Документация**
- <файл> — что обновлено

**3. Git**
- <хеш> — <описание>
- Security: clean / найдено (что именно)
- Секреты в tracked: нет

**4. Хвосты**
- Удалено: <файлы>
- Бэклог: <задачи>

**5. Следующий шаг**
- Если сессия завершена → запусти finishing-session
```

## Обработка ошибок

- **Git не установлен** → пропустить шаг 3, сообщить в отчёте
- **Не в git-репозитории** → пропустить шаг 3, сообщить в отчёте
- **Коммит упал** → записать изменения в бэклог, сообщить пользователю
- **Файл заблокирован** → попробовать позже, сообщить
- **Секрет в staged** → убрать из индекса, добавить в .gitignore, зафиксировать
- **Тесты не прошли** → не коммитить, сообщить пользователю, предложить исправления
- **Нечего коммитить** → не создавать пустой коммит
- **Линтер не установлен** → пропустить линтер, зафиксировать в отчёте
- **Typecheck недоступен** → пропустить, зафиксировать в отчёте

## Правила

- Если любой из 5 шагов невыполним — агент указывает почему и что сделал вместо этого
- Нельзя пропускать шаги «потому что это мелочь»
- НЕ создавать пустые коммиты
- НЕ коммитить .env и секреты
- НЕ выполнять разрушительные операции (удаление веток, stash, gc) — это вне скоупа приёмки

## Связанные скилы

- **finishing-session** — завершение сессии (симметричная пара). Запускается после accepting-work если сессия завершена. Отвечает за память и инциденты.
- **starting-session** — старт сессии (начало цикла)
