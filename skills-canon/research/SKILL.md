# Research Lab — система веб-поиска лаборатории

## Принцип
Основной инструмент — **free-api-hunter оркестратор** (exec): 4 провайдера,
15 ключей, гарантированный fallback + `verify`-режим кросс-проверки.
Нативный `web_search` — только fallback последнего рубежа (когда оркестратор недоступен).

## Источники: веб + семантическая память (обязательно оба)
`research` ищет **инсайты и паттерны** (НЕ пошаговые инструкции «как починить») в ДВУХ равноправных корпусах:
1. **Веб** — через free-api-hunter оркестратор / SearXNG-шлюз (issues, форумы, документация, дискуссии).
2. **Семантическая память лабы** — ТОЛЬКО через MCP-тул `memory-gateway__search_memory` (бэкенд ALM/AnythingLLM): прошлые инциденты, ADR, паттерны, решения из истории лаборатории.

**Универсальный скепсис (обязательно):** и веб, и память МОГУТ ВРАТЬ. Каждый результат поиска — под сомнением, пока не проверен. Проверяй цитируемый артефакт (а не доверяй выдаче), перекрёстно сверяй веб↔память (совпали ли паттерны из разных источников). Если семпамять недоступна (напр. embedder down) — работай по вебу с пометкой «семантическая память недоступна».

## Когда использовать
- **Quick (~1 мин):** простые факты → режим `factual` (1 провайдер)
- **Standard (1–3 мин):** анализ, 2–3 провайдера или `verify`
- **Deep (2–5 мин, зависит от числа провайдеров и латентности):** критичные
  решения → `deep_research` + фактчекинг. Тайминг НЕ гарантирован — быстрее
  при кэш-хите, медленнее при scrape полного контента.

## Быстрый старт
```bash
ORCH=/root/LabDoctorM/projects/free-api-hunter/scripts/search-orchestrator.sh
# Quick факт
bash $ORCH "<запрос>" factual 5
# Verify (критичные факты — обязательно): 4-й арг = порог пересечения URL (def 2)
bash $ORCH "<запрос>" verify 5 2
# Deep: параллельно по всем провайдерам
bash $ORCH "<запрос>" deep_research 10
```

## Методология (Deep Research)
1. Декомпозиция запроса на подвопросы.
2. Запуск оркестратора с нужным режимом.
3. Scrape полного контента при необходимости.
4. Приоритет источников — теперь ФОРСИТСЯ механикой: оркестратор сортирует
   выдачу по авторитетности домена (docs/офиц. → GitHub → Q&A/форумы → прочее),
   затем по corroboration и свежести (`_domain_priority` в каждом результате).
   Ручная приоритизация больше не обязательна, но проверяй топ глазами.
5. Фактчекинг: минимум **2 независимых источника** на утверждение.
6. Формирование отчёта с confidence levels.

## Фактчекинг (обязательно)
- **High** — 3+ независимых источника, вкл. офиц. документацию.
- **Medium** — 2 источника или 1 авторитетный.
- **Low** — 1 источник или противоречивые данные.

### ⚠️ Никогда не доверяй полю `answer` вслепую
Провайдеры (Tavily и др.) возвращают AI-синтезированный `answer`, который
**может противоречить собственным `results`**. Правила агента:
- Для критичных фактов — режим `verify`. Он даёт ДВА НЕЗАВИСИМЫХ сигнала
  в `_meta.verification` (не один обманчивый `verified`):
  - `sources_corroborated` — пересеклись ли URL Tavily и SearXNG (≥ порога).
  - `answer_consistent` — семантическая сверка: встречаются ли ключевые
    слова/числа из `answer` в реально найденных источниках. `unbacked_numbers`
    ловит цифры из ответа, которых НЕТ ни в одном источнике.
  Составной `answer_status`: `verified` (оба true) /
  `answer_contradicts_sources` (URL совпали, но ответ противоречит —
  раньше молча проходило как verified!) /
  `sources_corroborated_answer_unchecked` / `answer_consistent_low_corroboration` /
  `unverified_low_overlap` / `*_unavailable` (грейсфул-деградация, не провал).
  `verified:true` = «источники подтвердили И ответ им не противоречит», НЕ «ответ верен».
  Читай `results` даже при `verified`.
- **Свежесть:** старые источники (> ~18 мес) → `_meta.stale=true`; если такие
  в топ-5 → `stale_sources` в `_meta.contradictions`. Boilerplate (куки-баннеры)
  вырезается, генерированный synthesis помечен как extractive (не проверенный).
- **Всегда цитируй `results`, а не пересказывай `answer`.**
- **Soft-ban ≠ мёртвые ключи:** ответ `401`/`429`/`Unauthorized` — временный
  бан (пауза + ротация ключей), НЕ «все ключи мертвы». Не делай вывод, что
  скил сломан, по одному сбою — делай реальный end-to-end вызов и смотри
  `_meta.provider_used` / `_meta.fell_back`.

## Резервный путь (если оркестратор недоступен)
Оркестратор — единственный основной путь (нативный `web_search` в этой среде
выключен), поэтому при его падении используй прямой запрос к SearXNG-шлюзу:
```bash
curl -s "http://localhost:8889/search?q=<запрос>&format=json" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);[print(r['url']) for r in d.get('results',[])[:10]]"
```
SearXNG (порт 8889) агрегирует 5 платных пулов + бесплатные движки с реактивным
failover, так что даёт результаты даже без обёртки оркестратора. Это не полный
функционал (нет verify/дедупа/синтеза), но снимает полный outage скилла.
