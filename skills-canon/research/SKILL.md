# Research Lab — система веб-поиска лаборатории

## Когда применять
- Критичные решения, требующие проверки достоверности
- Архитектурные выборы и сравнение технологий/подходов
- Проверка безопасности и уязвимостей
- Сбор данных из множества источников с кросс-проверкой
- Любой ответ без верификации = потенциальная галлюцинация → сначала research

- Любой ответ без верификации = потенциальная галлюцинация → сначала research

## Границы применимости
- Основной путь — free-api-hunter оркестратор (exec); нативный `web_search` НЕ fallback (только резерв последнего рубежа)
- Семантическая память ТОЛЬКО через MCP `memory-gateway__search_memory`
- НЕ доверять полю `answer` вслепую — читать `results`
- Без фактчекинга (≥2 источника) утверждение не верифицировано

## Чек-лист качества
- [ ] Выбран режим (factual/verify/deep_research) по критичности
- [ ] Для критичных фактов — `verify` (пересечение URL Tavily↔SearXNG ≥2)
- [ ] Проверен `_meta.answer_status` (verified ≠ истинен; читать results)
- [ ] Фактчекинг ≥2 независимых источника; confidence выставлен
- [ ] Семантическая память опрошена через memory-gateway (для утверждений о лабе)
- [ ] При падении оркестратора — резервный curl к localhost:8889 (не полный функционал)

## Анти-паттерны
- ❌ Слепое доверие полю `answer` (может противоречить results)
- ❌ `verified` = «ответ верен» (это пересечение источников, не истинность)
- ❌ Нативный web_search как основной путь (только резерв)
- ❌ Прямые вызовы lab_search.py/mcp-memory/onnx вместо memory-gateway
- ❌ Вывод «скил сломан» по одному 401/429 (soft-ban ≠ мёртвые ключи)
- ❌ Пропуск фактчекинга при критичных решениях

## Принцип
Основной инструмент — **free-api-hunter оркестратор** (exec): 4 провайдера,
15 ключей, гарантированный fallback + `verify`-режим кросс-проверки.
Нативный `web_search` НЕ используется как fallback — резервный путь последнего
рубежа (когда процесс оркестратора недоступен) — прямой запрос к локальному
SearXNG (`localhost:8889`).

## Источники: веб + семантическая память (обязательно оба)
`research` ищет **инсайты и паттерны** (НЕ пошаговые инструкции «как починить») в ДВУХ равноправных корпусах:
1. **Веб** — через free-api-hunter оркестратор / SearXNG-шлюз (issues, форумы, документация, дискуссии).
2. **Семантическая память лабы** — ТОЛЬКО через MCP-тул `memory-gateway__search_memory` (бэкенд ALM/AnythingLLM): прошлые инциденты, ADR, паттерны, решения из истории лаборатории.

**Универсальный скепсис (обязательно):** и веб, и память МОГУТ ВРАТЬ. Каждый результат поиска — под сомнением, пока не проверен. Проверяй цитируемый артефакт (а не доверяй выдаче), перекрёстно сверяй веб↔память (совпали ли паттерны из разных источников). Если семпамять недоступна (напр. embedder down) — работай по вебу с пометкой «семантическая память недоступна».

## Когда использовать
- **Quick (~1 мин):** простые факты → режим `factual` (1 провайдер)
- **Standard (1–3 мин):** анализ, 2–3 провайдера или `verify`
- **Deep (2–5 мин, зависит от числа провайдеров и латентности):** критичные
  решения → `deep_research` + фактчекинг. Тайминг НЕ гарантирован — быстрее
  при кэш-хите, медленнее при scrape полного контента.

## Быстрый старт
```bash
ORCH=/root/LabDoctorM/projects/free-api-hunter/scripts/search-orchestrator.sh
# Quick факт
bash $ORCH "<запрос>" factual 5
# Verify (критичные факты — обязательно): 4-й арг = порог пересечения URL (def 2)
bash $ORCH "<запрос>" verify 5 2
# Deep: параллельно по всем провайдерам
bash $ORCH "<запрос>" deep_research 10
```

## Методология (Deep Research)
1. Декомпозиция запроса на подвопросы.
2. Запуск оркестратора с нужным режимом.
3. Scrape полного контента при необходимости.
4. Приоритет источников — задаётся конфигом `configs/domain_policy.json`
   (списки `promote`/`demote`/`block`): оркестратор вычисляет `_domain_priority`
   для каждого результата и сортирует выдачу (promote вверх, demote вниз),
   затем по `provider_count` и свежести. Ручная приоритизация не обязательна,
   но проверяй топ глазами.
5. Фактчекинг: минимум **2 независимых источника** на утверждение.
6. Формирование отчёта с confidence levels.

## Фактчекинг (обязательно)
- **High** — 3+ независимых источника, вкл. офиц. документацию.
- **Medium** — 2 источника или 1 авторитетный.
- **Low** — 1 источник или противоречивые данные.

### ⚠️ Никогда не доверяй полю `answer` вслепую
Провайдеры (Tavily и др.) возвращают AI-синтезированный `answer`, который
**может противоречить собственным `results`**. Правила агента:
- Для критичных фактов — режим `verify`. Поле `_meta.verification` содержит
  ЧЕСТНЫЕ сигналы (проверка лёгкая, keyword/regex — НЕ тяжёлый NLI):
  - `url_overlap_verified` (bool) — пересеклись ли URL Tavily и SearXNG
    (≥ `threshold`, по умолчанию 2). Это НЕ проверка истинности ответа.
  - `overlap_count` (int) — сколько URL совпало.
  - `answer_grounding` (0..1) — доля ключевых терминов/чисел `answer`,
    найденных в текстах реальных источников (Tavily + SearXNG). Лёгкая
    keyword-проверка: ловит грубые галлюцинации (напр. «systemd auto-restarts»).
  Составной `answer_status`:
    `verified` — overlap≥threshold И grounding≥0.6 (либо answer отсутствует);
    `answer_ungrounded` — overlap ок, НО grounding<0.6 (ответ противоречит
      источникам — раньше молча проходило как verified!);
    `unverified_synthesis` — overlap<threshold;
    `single_source_unverified` — выжил только один провайдер (overlap не считается),
      но живые данные отданы;
    `both_sources_unavailable` — оба провайдера недоступны.
  Никакой флаг НЕ означает «ответ верен». `verified` = «источники пересеклись
  И ключевые термины ответа есть в источниках», НЕ «ответ истинен».
  Читай `results` даже при `verified`.
- **Профили поиска (deep research vs fast):** единый инстанс SearXNG,
  разделение через `categories` (НЕ порты). Оркестратор прокидывает `categories`
  из env `SEARCH_CATEGORIES` (выставляется скиллом) в запрос к `localhost:8889`;
  default — `general`.
- **Авто-деградация:** для factual/content/dynamic/broad и deep_research
  оркестратор автоматически фолбэчит на локальный SearXNG (`localhost:8889`)
  при падении внешних провайдеров (последний хоп `run_chain`). `verify`-режим
  при падении одного провайдера отдаёт `single_source_unverified` с живыми
  данными. Если сам процесс оркестратора упал — резервный путь: прямой
  `curl http://localhost:8889/search?q=...` (см. ниже).
- **Свежесть:** старые источники (> ~18 мес) → `_meta.stale=true`; если такие
  в топ-5 → `stale_sources` в `_meta.contradictions`. Boilerplate (куки-баннеры)
  вырезается, генерированный synthesis помечен как extractive (не проверенный).
- **Всегда цитируй `results`, а не пересказывай `answer`.**
- **Soft-ban ≠ мёртвые ключи:** ответ `401`/`429`/`Unauthorized` — временный
  бан (пауза + ротация ключей), НЕ «все ключи мертвы». Не делай вывод, что
  скил сломан, по одному сбою — делай реальный end-to-end вызов и смотри
  `_meta.provider_used` / `_meta.fell_back`.

## Резервный путь (если оркестратор недоступен)
Оркестратор — основной путь. При падении самого процесса оркестратора
(не путать с падением отдельных провайдеров — те фолбэчат автоматически
на локальный SearXNG) используй прямой запрос к локальному SearXNG-шлюзу:
```bash
curl -s "http://localhost:8889/search?q=<запрос>&format=json" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);[print(r['url']) for r in d.get('results',[])[:10]]"
```
SearXNG (порт 8889) агрегирует 5 платных пулов + бесплатные движки с реактивным
failover, так что даёт результаты даже без обёртки оркестратора. Это не полный
функционал (нет verify/дедупа/синтеза), но снимает полный outage скилла.
