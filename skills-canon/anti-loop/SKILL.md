# Anti-Loop Protocol v2

## Назначение
Многоуровневая защита от зацикливания AI-агента. Останавливает бесконечные повторы, эскалирует пользователю.

Основан на:
- INC-021 (пример; файл инцидента отсутствует в registry): Сова зациклилась на одной операции 5+ раз
- INC-023: Сова 6 раз повторила "Gateway is restarting"
- Best practices 2025-2026: Circuit Breaker, Retry-Classify, Cost Control

## Когда применять
- Агент повторяет одно и то же действие/команду 3+ раза подряд
- Пользователь повторяет один запрос 3+ раза
- Сервис в цикле рестартов (3+ рестарта с тем же результатом)
- Любая ситуация повторяющихся неудачных действий
- Триггеры: «стоп петля», «зациклился», «anti loop»

## Границы применимости
- НЕ отключает circuit breaker принудительно: если `disabled: true` — эскалация обязательна, даже по прямой просьбе
- НЕ повторяет ту же команду с теми же параметрами (менять подход между попытками)
- НЕ игнорирует Cost Control: максимум 3 попытки/подход, 5 суммарно
- НЕ продолжает без явного решения пользователя после остановки
- Persistence: состояние в `anti-loop-state.json` (не коммитить в git)

## Чек-лист качества
- [ ] Остановиться — не отправлять очередной одинаковый ответ/команду
- [ ] Классифицировать ошибку: transient / permanent / unknown
- [ ] Сообщить пользователю: попытки, тип, причина + альтернативы A/B
- [ ] Ждать решения пользователя (НЕ продолжать самовольно)
- [ ] Persistence: обновить `failures[tool].count`, при ≥3 → `disabled: true`
- [ ] Лимиты: ≤3 попытки одним подходом, ≤5 суммарно; при нуле прогресса — СТОП
- [ ] Сброс circuit breaker — только по явному запросу «сбросить» или рестарту сессии

## Анти-паттерны
- ❌ Повтор одной команды с теми же параметрами — INC-021, INC-023
- ❌ Игнор permanent error (продолжать retry вместо смены подхода)
- ❌ Превышение лимита попыток (больше 3/5)
- ❌ Сброс circuit breaker без явного запроса пользователя
- ❌ Эскалация не выполнена при `disabled: true`
- ❌ Gateway restart loop без СТОП и эскалации (3+ рестарта = остановить)

## Persistence: anti-loop-state.json

Состояние circuit breaker сохраняется между вызовами в файле `~/.openclaw/skills/anti-loop/references/anti-loop-state.json`.

Формат:
```json
{
  "failures": {
    "tool_name": {
      "count": 3,
      "last_failure": "2026-06-28T09:30:00Z",
      "type": "permanent",
      "disabled": true
    }
  },
  "session_attempts": {
    "task_description_hash": 4
  },
  "last_updated": "2026-06-28T09:30:00Z"
}
```

**Правила persistence:**
- При каждом сбое инструмента — обновить `failures[tool_name].count` и `last_failure`
- При достижении count >= 3 — установить `disabled: true`
- Если `disabled: true` — инструмент НЕ используется даже если пользователь просит напрямую (эскалация обязательна)
- Сброс disabled: при рестарте сессии или по явному запросу пользователя "сбросить circuit breaker"
- `session_attempts`  tracks количество попыток по задаче (хеш описания для privacy)
- Файл создаётся автоматически, не коммитить в git

## Три уровня защиты

### Уровень 1: Retry-Classify (классификация ошибок)

**НЕ повторять одну и ту же команду дважды.**

Классифицировать ошибку перед повторной попыткой:
- **Transient** (сеть, timeout, rate limit) → можно повторить с экспоненциальной задержкой
- **Permanent** (нет файла, нет прав, неверный формат) → НЕ повторять, менять подход
- **Unknown** → 1 повторная попытка максимум, потом эскалация

### Уровень 2: Circuit Breaker (предохранитель)

Если инструмент или операция неоднократно сбоит — отключить его автоматически:
- 3 сбоя подряд на один инструмент → НЕ использовать этот инструмент в текущей сессии
- Вернуть пользователю: "Инструмент X временно недоступен (3 сбоя). Предлагаю альтернативу Y."

### Уровень 3: Cost Control (контроль расходов)

Каждая повторная попытка стоит токенов = денег:
- Максимум 3 попытки на одну задачу с одним подходом
- Максимум 5 попыток суммарно (все подходы)
- Если прогресс нулевой — СТОП, даже если попытки не исчерпаны

## Когда срабатывает

### Триггер 1: Агент повторяет действие
- Одно и то же действие/команда выполняется 3+ раза подряд
- Результат каждый раз одинаковый (ошибка, таймаут, отказ)

### Триггер 2: Пользователь повторяет запрос
- 3+ одинаковых запроса → признать непонятость, переформулировать

### Триггер 3: Сервис в цикле рестартов
- 3+ рестарта с тем же результатом → СТОП, эскалация

## Процедура при срабатывании

### Шаг 1: Остановиться
Прекратить повторяющееся действие. НЕ отправлять очередной одинаковый ответ.

### Шаг 2: Классифицировать ошибку
- Transient/Permanent/Unknown (см. Уровень 1)

### Шаг 3: Сообщить пользователю
```
⚠️ Anti-loop: остановлена операция X
Попыток: N
Тип ошибки: [transient/permanent/unknown]
Причина: [кратко]

Предлагаю:
- Вариант A: [альтернатива]
- Вариант B: [другая альтернатива]

Что делаем?
```

### Шаг 4: Ждать решение
НЕ продолжать без явного указания пользователя.

## Стратегия попыток

| Попытка | Действие | Задержка |
|---------|----------|---------|
| 1 | Стандартный метод | 0 |
| 2 | Альтернативный метод (другой инструмент/подход) | 1-2 сек |
| 3 | Fallback / ручное вмешательство | 2-5 сек |
| 4+ | **СТОП, эскалация** | — |

Между попытками ОБЯЗАТЕЛЬНО менять подход. НЕ повторять одну и ту же команду с теми же параметрами.

## Примеры

### Gateway restart loop
```
Попытка: systemctl restart → ошибка (нет DBUS)
Попытка: systemctl restart → та же ошибка → permanent error
⚠️ ANTI-LOOP: 2 неудачи, ошибка permanent.
Предлагаю: kill -9 <PID> && запуск вручную. Продолжить?
```

### API timeout loop
```
Попытка: curl → timeout
Попытка: curl → timeout → transient, но нет прогресса
Попытка: curl → timeout
⚠️ ANTI-LOOP: 3 таймаута. Сервис недоступен.
Предлагаю: проверить статус сервиса, попробовать альтернативный endpoint.
```

### Повторяющийся запрос пользователя
```
Пользователь (3-й раз): "покажи итоговый вариант"
⚠️ ANTI-LOOP: предыдущие ответы не показали то что нужно.
Уточни: какой именно файл/раздел показать?
```
