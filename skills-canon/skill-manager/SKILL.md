# Skill Manager Protocol v2

## Назначение
Безопасное управление жизненным циклом скилов с версионированием, проверкой зависимостей и атомарными операциями.

Основан на:
- INC-022: удаление скила без проверки allowlist
- Текущий инцидент: переименование сломало 4 скила
- Best practices: semantic versioning, dependency checking, atomic operations

## Принцип
Скил — это зависимость. Изменение требует проверки allowlist, конфига, ссылок.

## Версионирование

Каждый скилл обязан иметь в frontmatter:
- `version: "x.y.z"` (semantic versioning)
- `last_reviewed: "YYYY-MM-DD"`
- `status: active | proposal | deprecated`

## Единый вход: skill_workshop (ОБЯЗАТЕЛЬНО)

Все операции со скилами — ТОЛЬКО через тул `skill_workshop` (create/update/revise/apply/reject/quarantine).
Ручное редактирование `openclaw.json` (allowlist `defaults.skills` / `agents.list[].skills`) **запрещено** (обходной путь, см. ADR-0056).
Каноничная инструкция: `DoctorM_and_Ai/docs/skills-creation-methodology.md`.

## Обязательная регистрация

Скил считается принятым только после записи в реестр `myrmex-control/skill-registry.json`
(через `skill_workshop` — необратимый шаг). Нет записи в реестре = скил не существует для лабы.

## Создание скила
1. Вызвать `skill_workshop` action=create с описанием и proposal_content (строго по канону methodology)
2. Заполнить frontmatter (name, description ≤160, version, author, requires) — валидация автоматическая
3. `skill_workshop` сам регистрирует скил в `myrmex-control/skill-registry.json` и рассылает allowlist (НЕ руками!)
4. Проверить: `openclaw skills list | grep <name>` → ready
5. Контракт-проверка: соответствие канону + запись в реестре + нет битых ссылок (`scripts/skill-link-lint.sh`)

> 🔴 КРИТИЧНО (allowlist replace≠merge): шаги 2 и 3 ОБЯЗАТЕЛЬНЫ оба.
> Непустой `agents.list[].skills` **ПОЛНОСТЬЮ ЗАМЕНЯЕТ** `agents.defaults.skills` — он НЕ мержится
> с ним. В нашем конфиге у КАЖДОГО из 9 агентов есть explicit-список, поэтому добавить
> имя только в `defaults` НЕДОСТАТОЧНО — скил получит статус `excluded` и агент его не увидит.
> Managed-скилы активируются ТОЛЬКО agent-allowlist-ом; `skills.allowBundled` их не касается
> (он гейтит лишь bundled-скилы).

## Переименование скила
**Критично: сначала добавить новое везде → потом удалить старое.**
1. Создать новую директорию, скопировать SKILL.md
2. Обновить `name` и `version` в frontmatter (инкремент minor)
3. Добавить новое имя в конфиг (defaults + все агенты)
4. Удалить старое имя из конфига
5. Удалить старую директорию
6. Проверить: новый ready, старого нет

## Удаление скила
1. Grep по всем SKILL.md — нет ли ссылок на удаляемый
2. Удалить из `agents.defaults.skills`
3. Удалить из `agents.list[].skills` каждого агента (ВСЕХ 9!)
4. Удалить `~/.openclaw/skills/<name>/`
5. Проверить: скил отсутствует в `openclaw skills list`

**Запрещено:** удалять скил на который есть ссылки в других скилах.

> 🔴 КРИТИЧНО: как и при создании, шаги 2 и 3 ОБА обязательны — непустой agent-список
> заменяет defaults, не мержится. Забытое имя в одном из agent-списков оставит мёртвую
> ссылку (broken reference), которую потом покажет `openclaw doctor`.

## Обновление скила
1. Бэкап SKILL.md
2. Обновить содержимое
3. Обновить version (patch) и last_reviewed
4. Убедиться что имя не изменилось
5. Проверить что скил по-прежнему в конфиге

## КРИТИЧНО: allowlist replace≠merge и drift исключённых скилов

Уроки, исправляющие старые неточные заметки лабы:

- **replace≠merge.** Непустой `agents.list[].skills` полностью заменяет `agents.defaults.skills`, не мержится. Чтобы скил был виден агенту с explicit-списком — он должен быть И в `defaults`, И в каждом `agents.list[].skills`. Добавить только в `defaults` — недостаточно.
- **drift исключённых скилов (паттерн INC-018 / INC-022).** Дефект 2026-07-13: managed-скилы `deep-dive`, `ebsh`, `root-cause-archaeologist`, `yandex-suite` как новые не попали в allowlist (ни в defaults, ни во все agent-списки) → статус `excluded`, агенты их не видели. Исправлено добавлением всех 4 имён во все allowlist-списки. При каждой операции проверяй, что имя есть И в defaults, И во всех 9 agent-списках.
- **`skills.allowBundled` не активирует managed/workspace-скилы.** Он гейтит ТОЛЬКО bundled-скилы. Старое замечание лабы «allowBundled не активирует скилы» было неточным: правильная формулировка — allowBundled вообще не касается managed/workspace, они активируются только agent-allowlist-ом.
- **Рестарт gateway — только ЗавЛаб.** Субагент/агент рестарт не делает; для активации скила обычно достаточно skills-watcher + новой сессии/next turn.
- **Walkthrough-гейт обязателен.** После создания/правки пройди скил шаг за шагом и найди, где агент спотыкается (см. методологию лабы, раздел 8).

## Общие правила
- Бэкап конфига перед любой операцией
- Валидация JSON после изменений
- Не более 1 скила за операцию (атомарность)
- Перед удалением — проверить отсутствие ссылок в других скилах
- Перед переименованием — убедиться что новое имя не занято

## Обязательный контракт скила (SKILL.md)

Каждый скил в `skills-canon/` ОБЯЗАН содержать следующие элементы. Без них — скил считается **неполным** и не подлежит включению в allowlist.

### 1. Frontmatter (YAML) — обязательные поля
```yaml
---
name: "<kebab-case>"          # ключ для allowlist, slash-команды
 description: "..."            # ≤160 байт, one-line
 version: "X.Y.Z"              # semver, ОБЯЗАТЕЛЬНО
 author: "<agent-name>"        # кто создал/владеет
 last_reviewed: "YYYY-MM-DD"   # дата последнего ревью
 status: active | deprecated   # active по умолчанию
 user-invocable: true | false  # может ли юзер вызвать через /skill
 triggers:                     # фразы/паттерны/scope для авто-активации
   phrases: []
   patterns: []
   scope: []
 metadata:
   openclaw:
     requires:                 # зависимости для pre-check
       bins: []
       env: []
       config: []
     primaryEnv: "..."         # основная env-переменная (если есть)
---
```

### 2. Тело (Markdown) — обязательные секции (в указанном порядке)

#### `## Когда применять`
- Конкретные триггеры: когда агент ДОЛЖЕН вызвать этот скил
- Не абстрактные "полезно", а чёткие сигналы (фраза, паттерн, контекст)
- Минимум 3 пункта

#### `## Границы применимости`
- Чего скил **НЕ делает** (anti-scope)
- Где заканчивается ответственность скила и начинается другого
- Зависимости от других скилов / внешних инструментов

#### `## Чек-лист качества`
- Проверки, которые агент должен выполнить ПЕРЕД тем как сказать "done"
- Каждый пункт — верифицируемое условие (PASS/FAIL)
- Минимум 4 пункта

#### `## Анти-паттерны`
- Типичные ошибки применения этого скила (что НЕ делать)
- Связь с инцидентами: "см. INC-XXX / PAT-YY"
- Минимум 3 пункта

### 3. Машиночитаемый шаблон
Ссылка на канонический шаблон: `skills-canon/SKILL-TEMPLATE.md`
- Используется `skill-creator` / `skill-manager` для генерации нового скила
- Валидируется CI-гейтом после массовой миграции

### 4. Пример полного минимального скила
```markdown
---
name: "example-skill"
description: "Пример минимального полного скила"
version: "1.0.0"
author: "antcat"
last_reviewed: "2026-07-17"
status: active
user-invocable: true
triggers:
  phrases: ["пример скила"]
  patterns: ["example skill"]
  scope: ["демо"]
metadata:
  openclaw:
    requires:
      bins: ["jq"]
    primaryEnv: "EXAMPLE_KEY"
---

# Example Skill — демо скила по контракту

## Когда применять
- Нужно продемонстрировать структуру контракта
- Создаётся новый скил — используй как шаблон
- Проверка соответствия контракту

## Границы применимости
- Не выполняет реальную работу — только структура
- Не заменяет документацию OpenClaw
- Требует jq в PATH

## Чек-лист качества
- [ ] Frontmatter валидный YAML со всеми обязательными полями
- [ ] Все 4 обязательные секции тела присутствуют
- [ ] version в semver формате
- [ ] triggers не пустые

## Анти-паттерны
- Копирование без адаптации под задачу → INC-XXX
- Пустые секции "для галочки" → неполный скил
- Отсутствие version → разрыв контракта
```

## Обработка ошибок

### Конфиг не проходит валидацию
1. Откатить бэкап: `cp ~/.openclaw/openclaw.json.bak.* ~/.openclaw/openclaw.json`
2. Проверить синтаксис JSON: `python3 -m json.tool ~/.openclaw/openclaw.json > /dev/null`
3. Исправить синтаксическую ошибку (лишняя запятая, незакрытая скобка)
4. Повторить валидацию

### Скил не появляется в `openclaw skills list`
1. Проверить что директория существует: `ls ~/.openclaw/skills/<name>/SKILL.md`
2. Проверить frontmatter: `name` должен совпадать с именем директории
3. Проверить что скил добавлен И в `agents.defaults.skills`, И в `agents.list[].skills` каждого агента (replace≠merge! defaults один недостаточно) — иначе статус `excluded`
4. Дождаться обновления snapshot: skills-watcher обычно сам обновляет список на следующем turn-е/новой сессии. **Рестарт gateway — прерогатива ЗАВЛАБА; агент/сабагент не рестартирует.**
5. Проверить снова: `openclaw skills list | grep <name>`

### Скил есть в конфиге, но директория отсутствует
1. Это broken reference — удалить имя из конфига
2. Сообщить пользователю: "Обнаружена битая ссылка на скил <name>, удалено из конфига"
3. Зафиксировать в инцидентах если скил был нужен

### Переименование сломало зависимые скилы
1. Найти все ссылки: `grep -r "old-name" ~/.openclaw/skills/*/SKILL.md`
2. В каждом зависимом скиле заменить старое имя на новое
3. Проверить что все ссылки обновлены
4. Сообщить пользователю сколько файлов обновлено

### Permission denied при создании директории
1. Проверить владельца: `ls -la ~/.openclaw/skills/`
2. Исправить: `sudo chown -R $(whoami) ~/.openclaw/skills/<name>/`
3. Повторить создание

### Дублирование имени при переименовании
1. ОТМЕНИТЬ операцию
2. Сообщить пользователю: "Имя <new-name> уже занято. Выберите другое или удалите существующий скил."
3. НЕ применять частичные изменения
