# Fact-Check Protocol v2

## Назначение
Многоуровневый протокол проверки фактов и предотвращения галлюцинаций. Каждое утверждение — проверено или помечено как предположение.

Основан на:
- INC-016, INC-018, INC-019: фантазии вместо фактов
- Сам скилл запрещает неверифицируемые ссылки (в т.ч. «best practices 2025-2026» без источника) — поэтому такая ссылка удалена

## Когда применять
- Любое утверждение без источника (цифры, даты, версии, факты)
- Данные из внешних API и выводы на основе кода/конфигов
- Рекомендации для ЗавЛаба (до выдачи советов)
- Перед коммитом — проверка что не сломает зависимости/ADR
- Триггеры: «проверь факты», «фактчекинг», «это правда?»

## Границы применимости
- НЕ заменяет семантический поиск по лабе — он ОБЯЗАТЕЛЕН через `memory-gateway__search_memory` (MCP)
- НЕ использует запрещённые пути: `lab_search.py`, `mcp-memory :8087`, `onnx-embedder :8082`, native `memory_search`
- НЕ выдаёт confidence без основания (High требует 3+ источника)
- НЕ изобретает ссылки на несуществующие файлы/документы
- Один grep по файлам недостаточен — семантический поиск обязателен для утверждений о лабе

## Чек-лист качества
- [ ] Семантический поиск через `memory-gateway__search_memory` (для утверждений о лабе)
- [ ] Code verification: grep/cat/ls для утверждений о коде/конфигах/файлах
- [ ] Cross-reference: минимум 2 независимых источника (3+ для критичных)
- [ ] Confidence scoring выставлен (High/Medium/Low/Unknown)
- [ ] Маркировка неопределённости при confidence < High
- [ ] Поведенческие/прогнозные утверждения — прослежен render-path ИЛИ сделан прогон (статика кода не считается фактом поведения)
- [ ] Красные флаги проверены (цифры без источника, противоречия)
- [ ] Ответ в формате Verified / Partially / Unverified с источниками

## Анти-паттерны
- ❌ Фантазия вместо факта — INC-016, INC-018, INC-019
- ❌ Цифры/даты/версии без источника
- ❌ Ссылки на несуществующие файлы/документы
- ❌ «очевидно» без фактической базы
- ❌ Прямые вызовы `lab_search.py` / `mcp-memory :8087` / `onnx-embedder :8082` / native `memory_search` (только `memory-gateway`)
- ❌ Confidence High без 3+ независимых источников
- ❌ Один grep вместо семантического поиска по лабе

## 7 методов проверки

### 1. Cross-Reference (перекрёстная проверка)
- Минимум **2 независимых источника** на утверждение
- Для критичных решений — **3+ источника**, включая официальную документацию
- Источники: документация → GitHub → сообщества → кейсы

### 2. Source Citation (цитирование)
- Каждый факт — с указанием источника
- Формат: `утверждение [источник: URL/файл]`
- Если источник недоступен — пометить как "непроверено"

### 3. Code Verification (проверка кодом)
- Утверждения о коде → проверить через `grep`, `cat`, `ls`
- Утверждения о конфигах → проверить через `cat`, `systemctl status`
- Утверждения о файлах → проверить через `ls`, `test -f`, `wc -l`

### 4. Lab Semantic Search (семантический поиск по лабе)
- **ОБЯЗАТЕЛЬНО** перед фактчекингом утверждений о лаборатории: сверься с семантической памятью, чтобы не потерять знания (ADR, правила, инциденты, паттерны).
- **Единственный путь — MCP-тул `memory-gateway__search_memory`** (MCP-сервер `memory-gateway`, бэкенд ALM/AnythingLLM): `memory-gateway__search_memory(query="...", top_k=5)` — гибрид vector+lexical (RRF) по всей лаборатории.
- Прямые вызовы `lab_memory_search` (mcp-memory :8087), `lab_search.py`, скилла `labsearch`, `onnx-embedder :8082` и native `memory_search` — ЗАПРЕЩЕНЫ (см. APPEND_SYSTEM.md). Если MCP недоступен — fallback на grep по реальным путям, пометь confidence Low.
- Одного grep по файлам недостаточно — семантический поиск находит релевантные артефакты, которые grep не выцепит.
- Если оба пути упали → fallback на grep по реальным путям и пометь confidence как Low.

### 5. Confidence Scoring (оценка достоверности)
- **High** — 3+ независимых источника, включая официальную документацию
- **Medium** — 2 источника или 1 авторитетный
- **Low** — 1 источник или противоречивые данные
- **Unknown** — нет источников, предположение

### 6. Hallucination Red Flags (красные флаги)
- Конкретные цифры без источника (цены, даты, версии)
- Ссылки на несуществующие файлы/документы
- Уверенный тон при отсутствии данных
- Противоречия с известными фактами

### 7. Uncertainty Marking (маркировка неопределённости)
- "Я не уверен" — если confidence Medium или ниже
- "Требует проверки" — если единственный источник
- "Предположительно" — если вывод сделан на основе косвенных данных

## Процесс фактчекинга

### Перед ответом на сложный вопрос:
1. Проверить через лабораторный семантический поиск (через MCP `memory-gateway__search_memory`) — найти релевантные ADR/правила/инциденты
2. Проверить через code verification (grep/cat/ls)
3. Проверить через cross-reference (2+ источника)
4. Оценить confidence level
5. Если confidence < High — пометить в ответе

### Перед рекомендацией:
1. Проверить текущее состояние (реальные данные)
2. Проверить через лабораторный семантический поиск (через MCP `memory-gateway__search_memory`) — прошлые решения (ADR, инциденты, паттерны)
3. Проверить внешние лучшие практики
4. Сформулировать с confidence level

### Перед коммитом:
1. Проверить что изменение корректно (diff, тесты)
2. Проверить что не сломает зависимости (grep по проектам)
3. Проверить что соответствует ADR/паттернам

## Поведенческие и прогнозные утверждения (render-path) — ОБЯЗАТЕЛЬНО
Утверждения вида «система выведет X», «следующий отчёт покажет Y», «при 9/9 OK скроется Z» — это НЕ факт о коде, а предсказание ПОВЕДЕНИЯ системы под условием.
- Статика кода (наличие строки/формата, grep) НЕ достаточна: код может быть, но путь рендера (условия сборки отчёта, COLLAPSE-TO-GREEN, early-return, ветки по overall) может изменить вывод.
- ПРАВИЛО: прежде чем утверждать поведение системы, ПРОСЛЕДИ путь генерации вывода (render-path) ИЛИ сделай реальный прогон/замер.
  - Как: найди функцию, которая формирует вывод (напр. `build_report`), прочитай ветки (что прячется при `overall=="OK"`, какие категории показываются всегда), и только тогда делай вывод.
  - Если не можешь проследить путь — пометь `INFERENCE conf=<X>%`, не выдавай за факт.
- Связано с red line #0 (Доминики): фактчекинг ВСЕГДА, включая трейс рендера для поведенческих утверждений.
- Инцидент-урок: INC-2026-07-16-151459-ce8984 (предсказала «отчёт покажет verbose» по наличию строки в коде, но не прочитала `build_report` → COLLAPSE-TO-GREEN скрыл строку → расхождение с фактом).

## Красные линии

- **НЕ** выдавать желаемое за действительное
- **НЕ** изобретать цифры, даты, версии
- **НЕ** ссылаться на несуществующие файлы
- **НЕ** использовать "очевидно" без фактической базы
- **НЕ** утверждать поведение системы («выведет/скроет/покажет X», «следующий отчёт будет Y») без трейса render-path ИЛИ реального прогона — статика кода недостаточна (см. раздел выше)
- Если не знаешь — скажи "не знаю, проверю"

## Обработка ошибок

### Cross-reference невозможен
- Если найден только 1 источник → пометить confidence как Medium/Low и честно сообщить
- Если источники недоступны (нет сети, 403, 404) → использовать code verification (grep/cat/ls) как единственный метод и пометить confidence как Low
- Если нет ни одного источника → сказать «не знаю, проверю» и предложить план проверки

### Нет доступа к источникам
- Документация недоступна (офлайн) → проверить локальные копии (`docs/`, `adr/`, `projects/*/docs/`)
- API недоступен → использовать кэшированные данные с пометкой «может быть устаревшим»
- Файл заблокирован → попробовать альтернативный путь или пометить как «требует ручной проверки»

### Конфликт между источниками
- Два источника противоречат друг другу → пометить confidence как Low, привести оба источника, не выдавать одно за истину
- Официальная документация vs сообщество → приоритет у официальной документации, но сообщить о расхождении
- Старый источник vs новый → приоритет у нового, но проверить дату публикации
- Если конфликт критичен для решения → эскалировать ЗавЛабу с описанием обоих позиций
