# Session Start — Старт сессии

## Назначение

Скил запускается в начале каждой сессии. Загружает контекст агента, показывает сводку по инцидентам и готовит handoff-состояние для downstream-скиллов. Старт ПЕРСОНАЛИЗИРОВАН под конкретного агента: чеклист и тон вывода строятся по роли агента, а не «под копирку».

Развёрнутый мониторинг состояния системы (диск / uptime / load / сервисы / docker / failed) при старте НЕ выполняется — он вынесен в hourly cron-монитор (агент Доминика), который сам маршрутизирует выявленные проблемы нужному агенту. См. причину редакции #1 (2026-07-13, решение ЗавЛаба).

## Правила сессии

Следуй правилам из AGENTS.md (уже в контексте).

## Что OpenClaw загружает автоматически

НЕ перечитывай — уже в system prompt:
- AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md
- MEMORY.md (в main session)
- memory/YYYY-MM-DD.md (сегодня + вчера, через startupContext)

## Чеклист старта сессии

Обязательные проверки (выполняй параллельно):

0. Персона агента (индивидуализация):
   - Прочитать `<workspace>/IDENTITY.md` и `<workspace>/startup-profile.md` (если есть)
   - ⚠️ ВНИМАНИЕ: скрипт `audit_lab.sh` НЕ существует (проверено: ни в `.qwen/scripts`, ни в системе). Команда `bash /root/LabDoctorM/.qwen/scripts/audit_lab.sh` из IDENTITY.md устарела и ВЫПОЛНЯТЬСЯ НЕ ДОЛЖНА. Аудит myrmex.json делай вручную (сверка с реальностью), как указано в startup-profile.md. Не доверяй ссылкам на audit_lab.sh в AGENTS.md/IDENTITY.md.
   - Взять: роль, зону ответственности, релевантные сервисы/эндпоинты, тон вывода
   - Строить чеклист ПОД эту роль, а не универсально
   - Пример: mangust → git/код/онтология; raven → алерт-пайплайн/логи; bestia → API снабжения

1. Инциденты: `ls /root/LabDoctorM/projects/DoctorM_and_Ai/incidents/ 2>/dev/null | grep -v README | grep -v template | tail -10`

2. Незавершённая работа агента (напоминание, НЕ действие):
   - При старте агент вспоминает СВОЮ работу, которую не довёл до конца,
     и докладывает ЗавЛабу, чтобы добить. В лабе git ОБЩИЙ — все агенты
     коммитят в одни репы, поэтому ВСЁ СКОПИРОВАНО на AGENT_ID (без
     фильтра по автору агент видел бы ЧУЖОЕ).
   - «Не доведён до конца» = коммит агента, которого НЕТ в базовой ветке
     (origin/main / origin/master / @{upstream}), независимо: локально-
     сиротский ИЛИ в feature-ветке, но не слитой.
   - AGENT_ID (приоритет: env → IDENTITY.md `id:` → имя каталога workspace):
     ```bash
     AGENT_ID="${AGENT_ID:-${LAB_AGENT:-}}"
     [ -z "$AGENT_ID" ] && [ -f IDENTITY.md ] && \
       AGENT_ID="$(grep -m1 -E '^id:' IDENTITY.md | sed -E 's/^id:[[:space:]]*//; s/[[:space:]]*$//')"
     AGENT_ID="${AGENT_ID:-$(basename "$PWD")}"
     ```
   - Корзины СВОЕГО по всем `projects/*` с .git:
     ```bash
     for d in /root/LabDoctorM/projects/*/; do
       [ -d "${d}.git" ] || continue
       # база сравнения (origin/main → origin/master → @{upstream} → алерт)
       base=""
       for cand in origin/main origin/master; do
         git -C "$d" rev-parse --verify "$cand" >/dev/null 2>&1 && { base="$cand"; break; }
       done
       [ -z "$base" ] && base="$(git -C "$d" rev-parse --abbrev-ref --symbolic-full-name @{upstream} 2>/dev/null)"
       if [ -z "$base" ]; then
         echo "$(basename "$d"): ⚠️ НЕТ БАЗЫ (ни origin/main/master, ни upstream) — не могу сравнить с main"
         continue
       fi
       # (а) незакоммиченное — только в репо, где у агента есть ветка
       has=$(git -C "$d" for-each-ref --format='%(refname:short)' refs/heads/ 2>/dev/null | grep -c "^$AGENT_ID/" || true)
       [ "${has:-0}" -gt 0 ] && git -C "$d" status -s 2>/dev/null | while read l; do echo "$(basename "$d"): [DIRTY] $l"; done
       # (б)+(в) свои коммиты не в базе — по ВСЕМ локальным веткам
       for b in $(git -C "$d" for-each-ref --format='%(refname:short)' refs/heads/ 2>/dev/null); do
         git -C "$d" log --oneline "$base..$b" --author="$AGENT_ID" 2>/dev/null | \
           while read h m; do echo "$(basename "$d") [$b]: $h $m"; done
       done
       # (г) СВОИ коммиты под ОБЩИМ аккаунтом agents@ — утечка атрибуции
       #     (страж теперь блокирует новые; старые надо переатрибутировать)
       for b in $(git -C "$d" for-each-ref --format='%(refname:short)' refs/heads/ 2>/dev/null); do
         git -C "$d" log --oneline "$base..$b" --author="agents@labdoctorm.ru" 2>/dev/null | \
           while read h m; do echo "$(basename "$d") [$b]: ⚠️ AGENTS@ $h $m"; done
       done
     done
     ```
   - Если нашлось своё (а)/(б)/(в) — вывести ⚠️ напоминание:
     «ЗавЛаб, осталась незавершённая работа: <список>.
     Сейчас разберусь в корне и доведу до main / исправлю.»
   - Если нашлось (г) — отдельный ⚠️:
     «Найдены коммиты под общим аккаунтом agents@ (утечка атрибуции).
     Переатрибутирую или доложу ЗавЛабу.»
   - НЕ коммитить/пушить автоматически при старте — только напомнить.
     Коммит/пуш — по завершении работы (finishing-session) или по команде.

3. Проверка инъекции контекста (без дублей):
   - Ожидаемый набор авто-загрузки уже в system prompt: AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md, MEMORY.md (main), memory/YYYY-MM-DD.md (через startupContext).
   - Убедись, что каждый файл инъектирован РОВНО ОДИН раз. Дублирование = один и тот же блок/файл дважды в контексте.
   - Если видишь дублирование — СООБЩИ ЗавЛабу, НЕ исправляй сам. Это «размазывание контекста» (context sprawl): жжёт токены и смазывает фокус сессии.
   - Само правило чистоты: вывод скила НЕ должен ре-стейтить то, что уже в инъекте (напр. не дублировать роль агента из IDENTITY в выводе). Только добавленное: инциденты, напоминания, handoff.

4. Семантический поиск / fact-check (точки входа, НЕ мониторинг):
   - ДВЕ точки входа для проверки утверждений против реальности ДО того, как заявлять факт:
     (1) скилл `labsearch` → `lab_search.py` / FAISS (движок onnx-embedder:8082)
     (2) MCP-сервер `memory` (`mcp-memory.service`, порт 8087, read-only) — для MCP-клиентов
   - Используй для проверки утверждений против реальности ДО того, как заявлять факт.
   - ПРАВИЛО «НЕ СУЩЕСТВУЕТ ≠ DOWN» (обязательно): если сервис из профиля/памяти НЕ найден ни в `systemctl list-units --all`, ни в `docker ps -a`, ни на своём порту (`ss -ltn`) — это ЕЩЁ НЕ доказывает, что его нет. Web-приложение может быть отдано nginx как статический SPA или reverse-proxy БЕЗ собственного юнита/контейнера/порта. Прежде чем написать «НЕ СУЩЕСТВУЕТ», дополнительно проверь: (a) `grep -rl <name> /etc/nginx/sites-enabled` — включён ли конфиг, (b) директорию деплоя/статики из этого конфига (напр. `/var/www/<name>`), (c) `ps aux | grep <name>` на предмет процесса. Только если НЕТ во ВСЕХ источниках (юнит/контейнер/порт/nginx-конфиг/деплой/процесс) — пиши «НЕ СУЩЕСТВУЕТ». НИКОГДА не путай отсутствие юнита с отсутствием сервиса. «DOWN» ставь ТОЛЬКО если юнит/контейнер/процесс СУЩЕСТВУЕТ, но failed/inactive ИЛИ порт определён но не отвечает.
   - Известные декоммисированные/несуществующие (НЕ считать сервисом, НЕ помечать DOWN): `context-api` (порт 8100) — выведен из эксплуатации; семантический поиск жив через onnx-embedder:8082 (lab_search.py / FAISS) и MCP memory:8087.

5. Anti-loop state: состояние anti-loop НЕ хранится как файл по умолчанию (каталог references/ отсутствует). НЕ трактуй отсутствие файла как «сброшен». Проверяй через скилл anti-loop; если состояние не задано — считай `{}` (это норма, не инцидент).

6. Handoff-состояние (для downstream-скиллов):
   - Записать `memory/<date>-session-state.json`:
     `{ agent, role, checks_summary, open_incidents, open_questions, relevant_services }`
   - Это читают deep-dive / root-cause-archaeologist / finishing-session (убирает «каждый спрашивает заново»)

## Что НЕ нужно делать при старте

- НЕ выполнять развёрнутый мониторинг состояния (диск/uptime/сервисы/docker/failed) — он в hourly cron-мониторе Доминики, не здесь.
- НЕ пушить коммиты автоматически при старте — только напомни (пуш по завершении или по команде).
- НЕ дублировать в выводе то, что уже инъектировано (роль/правила AGENTS.md) — только добавленное; скил без мусора.
- НЕ проверять сервисы по одному через `systemctl is-active` — мониторинг не наша зона при старте.
- НЕ читать логи сервисов (это зона диагностики, не старта)
- НЕ перечитывать bootstrap-файлы
- НЕ делать git операции из корня LabDoctorM (не репозиторий)
- НЕ обновлять память — зона finishing-session
- НЕ закрывать/открывать инциденты — зона finishing-session

## Известные ограничения

- startupContext работает только в main session, НЕ в subagent
- memory/YYYY-MM-DD.md загружается только через startupContext (one-shot при /new, reset)
- Полностью отключить автосброс сессии невозможно (только daily/idle)
- startup-profile.md опционален: без него старт работает как универсальный, но без персонализации
- Развёрнутый мониторинг состояния вынесен в hourly cron (Доминика) — здесь не дублируется
