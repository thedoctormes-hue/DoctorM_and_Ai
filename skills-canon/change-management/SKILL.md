# Change Management Protocol v2

## Назначение
Безопасное управление изменениями в лаборатории. Каждое изменение — документировано, согласовано, проверено.

Основан на:
- INC-026: добавление неподдерживаемого поля `tools.allow` в `agents.list` уронило gateway при старте (OpenClaw 2026.6.9); `openclaw doctor` пропустил — он валидирует синтаксис, а не поддержку конкретной версией
- ADR-0048 «Стандарт управления конфигурацией Gateway» (`workspaces/kotolizator/adr/0048-gateway-config-management-standard.md`): любое изменение `openclaw.json`/`models.json` — бэкап → `openclaw config validate` → `openclaw doctor` → мониторинг 24ч
- Best practices: ITIL change management, GitOps, infrastructure-as-code

## Принцип

**Каждое изменение = 4 шага.** Без исключений.

## Процедура

### Шаг 1: План (Plan)

Перед любым изменением:
1. Определить **что** меняем и **почему**
2. Оценить **риски** (что сломается если что-то пойдёт не так)
3. Определить **rollback plan** (как откатить)
4. Записать в формате:

```markdown
## Изменение: [краткое описание]
- Что: [конкретно что меняется]
- Почему: [обоснование]
- Риск: [low/medium/high]
- Rollback: [как откатить]
- Влияние: [что затронет]
```

**Примеры классификации рисков:**

| Риск | Примеры | Типичные действия |
|------|---------|-----------------|
| Low | Правка комментария, обновление README, форматирование | Выполнить, уведомить |
| Medium | Изменение переменной окружения, обновление зависимости, правка скрипта | Показать план, ждать подтверждения |
| High | Изменение openclaw.json, удаление проекта, смена пароля | Показать план + rollback, ждать явного подтверждения |

**Ошибки шага 1:**
- Не удалось определить что меняем → уточнить у пользователя, НЕ гадать
- Не удалось оценить риск → считать по умолчанию High
- Rollback невозможен (нет бэкапа) → создать бэкап перед продолжением, НЕ делать изменение без возможности отката

### Шаг 2: Согласование (Approve)

- **Low risk** (документация, комментарии, форматирование) → выполнять, уведомить в конце
- **Medium risk** (конфиги проектов, скрипты, настройки) → показать план, ждать подтверждения
- **High risk** (конфиги агентов, gateway, инфраструктура) → показать план + rollback, ждать явного подтверждения

### Шаг 3: Выполнение (Execute)

1. Создать бэкап изменяемого файла: `cp file file.bak.$(date +%s)`
 2>/dev/null
2. Внести изменение
3. **Если меняется `openclaw.json` или `models.json`** — ОБЯЗАТЕЛЬНО (по ADR-0048):
   - `openclaw config validate` — должен пройти ДО применения
   - `openclaw doctor` — проверка системы
   - после применения: мониторинг 24ч (`openclaw status`, `journalctl -u openclaw --since "24h ago" | grep -iE 'error|warn'`)
   - ⚠️ `doctor` НЕ проверяет, поддерживает ли конкретная версия OpenClaw ту или иную структуру (урок INC-026) — будь осторожен с нестандартными полями
4. Закоммитить через `lab-commit.sh` (если проект) или записать в память
5. Если изменение требует рестарта сервиса → использовать `safe-restart` протокол

**Ошибки шага 3:**
- Бэкап не удалось создать (нет прав, нет места) → СТОП, сообщить пользователю, НЕ делать изменение
- Файл заблокирован другим процессом → подождать 5 сек, повторить; если снова заблокирован → эскалация
- `lab-commit.sh` не найден → искать: `find /root/LabDoctorM/projects -name lab-commit.sh -type f`
- Коммит упал (pre-commit hook) → исправить причину (скорее всего секрет или .env), повторить
- Рестарт сервиса не удался → использовать rollback, сообщить пользователю

### Шаг 4: Проверка (Verify)

1. Проверить что изменение применилось: `grep`, `cat`, `systemctl status`, `curl`
2. Проверить что ничего не сломалось: тесты, health checks, логи
3. Если проблема → выполнить rollback
4. Записать результат в память

**Ошибки шага 4:**
- Проверка не прошла (изменение не применилось) → проверить синтаксис, пути, права; попробовать применить заново
- Проверка выявила поломку → немедленный rollback, сообщить пользователю, предложить отложить изменение
- Невозможно проверить (нет тестов, нет health check) → пометить как "нерифицированное", запланировать проверку позже
- Rollback тоже не сработал → критический инцидент, эскалация, писать в incidents/

## Правила

- **Всегда** документировать изменение (commit message, memory, или файл)
- **Всегда** иметь rollback plan перед началом
- **Никогда** не менять конфиги агентов без согласования
- **Никогда** не удалять бэкапы до подтверждения что изменение работает
- Одно изменение = один коммит (атомарность)

## Формат отчёта об изменении

```
✅ Изменение выполнено: [описание]
📁 Файлы: [список]
🔄 Rollback: [команда для отката]
📊 Результат: [OK/FAILED]
```
