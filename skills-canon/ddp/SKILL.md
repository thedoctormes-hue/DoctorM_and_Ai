# ДДП — Детальное Докапывание Проекта (максимальное погружение)

## Назначение
Полный аудит проекта цепочкой скилов: погружение → корень ошибок → фактчекинг → поиск инсайтов/паттернов (веб + семпамять). DDP **НЕ делает анализ сам** — делегирует 5 под-скилам (deep-dive, root-cause-archaeologist, fact-check, research, labsearch; PAT-004: не дублируем). DDP — оркестратор: порядок, ворота, передача находок.

## Вход (обязателен)
Конкретный путь к проекту, напр. `/root/LabDoctorM/projects/doctorm-unify-protocol`.
Без пути — СТОП и спроси: «В какой проект погружаемся? Дай путь».

## Контракт находок (findings) — JSON
Каждый этап возвращает находки; оркестратор **нормализует** их в этот список (под-скилы могут отдавать текст или свой формат — оркестратор маппит в схему; см. «Переноска»). Обязательные поля:
```
{ "type": "error"|"contradiction"|"blindspot"|"fragility"|"root_cause"|"other",
  "severity": "critical"|"high"|"medium"|"low",
  "summary": "краткое описание",
  "evidence": "ссылка на файл/строку/факт проекта",
  "confirmed": true|false|null }
```
Семантика `type`:
- `error` — утверждение/поведение, противоречащее коду/доке/тестам/рантайму проекта, либо известный anti-pattern/риск.
- `contradiction` — внутреннее противоречие (дока vs код, два утверждения против друга).
- `blindspot` — неучтённый случай/зона, которую проект или дока игнорирует.
- `fragility` — хрупкое место, зависящее от внешних условий/порядка/доброй воли без гарантии.
- `root_cause` — гипотеза корня проблемы (вывод root-cause-archaeologist, шаг 2).
- `other` — всё прочее.
Поле `confirmed` выставляет fact-check (шаг 3): подтверждённый факт = true, опровергнутый = false, не проверялось = null. Используется Воротами C.
Ворота и потолок считаются по ПОЛЯМ **нормализованного** JSON, а не по догадке агента.

**ОБЯЗАТЕЛЬНО:** при вызове каждого под-скила включи в промпт этот шаблон и потребуй вернуть findings в максимально близкой к схеме форме; оркестратор доводит до схемы (выставляет type/severity по смыслу, confirmed=null по умолчанию).

## Конвейер (0.Предусловие + 5 шагов + потолок, ворота зеркалят бриф)

### 0. Предусловие (pre-flight) — ВСЕГДА
Перед конвейером проверить, что 5 под-скилов доступны и включены: `deep-dive`, `root-cause-archaeologist`, `fact-check`, `research`, `labsearch`. Если хотя бы один отсутствует/выключен — СТОП и сообщить ЗавЛабу, какого не хватает. DDP от них зависит (PAT-004) и сам анализ не делает.

### 1. Исследование проекта (deep-dive) — ВСЕГДА
Вызвать `deep-dive` на проект. Результат: структура + `findings` (в формате контракта).
Сохранить `findings` (см. «Переноска»).

### Ворота А — в findings есть записи с type=error (любой severity) ИЛИ severity≥high (любой type)?
- НЕТ → Синтез (с пометкой «ошибок и high/critical находок не найдено»).
- ДА → шаг 2.

### 2. Поиск корня находок (root-cause-archaeologist) — УСЛОВНО
Вызвать `root-cause-archaeologist` на ошибки из `findings` (сгруппировать в кластеры связанных, вызвать на каждый кластер или топ-N по severity; скил заточен на ОДНУ проблему).
Результат: гипотезы корня (type=root_cause, с evidence).

### Ворота B — после анализа корня остались дефекты, стоящие проверки фактами?
- НЕТ (root-cause показал, что это особенности/не ошибки) → Синтез.
- ДА → шаг 3.

### 3. Фактчекинг, что проблема реальная (fact-check) — УСЛОВНО
Вызвать `fact-check` на **ВЫВОДЫ root-cause** (гипотезы корня), не на сырые находки deep-dive. `fact-check` проверяет гипотезы корня по коду/доке/тестам/рантайму проекта (grep/cat/ls, cross-reference, confidence).
Результат: подтверждённые фактами корни vs опровергнутые. Оркестратор выставляет им `confirmed=true/false` в нормализованном JSON (для Ворот C).

> Почему fact-check ПОСЛЕ root-cause: root-cause строит гипотезу ПОЧЕМУ; fact-check эту гипотезу проверяет. Кольцо «нашли → объяснили → подтвердили» логично и не требует менять порядок брифа.

### Ворота C — в нормализованных findings есть записи с confirmed=true?
- НЕТ → Синтез (с пометкой «ошибки не подтверждены фактами»).
- ДА → шаг 4.

### 4. Поиск решения / инсайтов и паттернов (research) — УСЛОВНО
Вызвать `research` на подтверждённые ошибки/корни. `research` ищет НЕ инструкцию, а **инсайты и паттерны**:
- в вебе (issues, форумы, документация, дискуссии);
- в семантической памяти лабы: **основной движок — lab_memory_search (MCP-тул :8087)**; labsearch (скилл, :8082) — резервный при падении MCP. При расхождении результатов двух движков — перекрёстная сверка, не доверяй одному.
- **всегда ставит результаты под сомнение** и проверяет где возможно (перекрёстная сверка веб↔память; проверка цитируемого артефакта, а не доверие к выдаче).
Результат: паттерны/инсайты (как эта ошибка встречается у других, известные решения, best practices).

### 5. Финальный отчёт (Синтез) — ВСЕГДА
Собрать воедино. Базовое понимание проекта — из синтеза `deep-dive` (Фаза 5); DDP **ДОПОЛНЯЕТ** его корнями (шаг 2), фактами (шаг 3) и инсайтами (шаг 4) — не пересказывая проект.
Итог + рекомендации ЗавЛабу.

## Потолок усилий (обязателен)
- После шага 1 выделить **3–5 самых важных проблем** (по severity, кластерами).
- В отчёте явно: «Выделено 3–5 ключевых проблем. Это НЕ значит, что проблем больше нет — это значит, что, возможно, понадобятся ещё проходки в новых сессиях».
- Остальные находки (ниже порога или сверх лимита) — в кратком списке «найдено, но глубоко не анализировалось», НЕ выбрасываются молча.

## Переноска findings (cross-stage handoff)
Оркестратор ведёт скил в своей сессии. Отдельные вызовы скилов не делят LLM-переменные — явно передавай релевантную часть `findings` в промпт каждого вызова. Для длинных цепочек продублируй `findings` в `memory/<date>-ddp-state.json` и цитируй.

## Правила
- ❌ Не реализовывать анализ самому — вызывай скилы.
- ❌ Не пропускать ворота (не запускай следующий этап без основания по брифу).
- ❌ Не выдавать предположение за факт (fact-check обязателен перед research).
- ❌ Не доверять слепо ни вебу, ни семпамяти — всё под сомнением, проверяй.
- ✅ Каждый этап — отдельный вызов скила со своим протоколом.
- ✅ `findings` передаются в промпт каждого вызова (см. Переноска).
- ⚠️ Таблицы не использовать в Telegram — списки / ASCII-схемы.

## Формат вывода
```
🧬 Максимальное погружение: <проект>

📂 Исследование (deep-dive):
<краткое резюме + находки>

🔍 Корни ошибок (root-cause):
<гипотезы корня>

✅ Фактчекинг (fact-check):
<подтверждённые / опровергнутые>

🔬 Инсайты и паттерны (research):
<веб + семпамять, с оговоркой о недостоверности>

🎯 Итог + рекомендации:
<что делать ЗавЛабу>
📌 Потолок: выделено 3–5 ключевых проблем; не значит, что других нет — возможны проходки в новых сессиях.
```

## Заметки
- DDP исследует и аудитирует, НЕ исправляет (исправление — за ЗавЛабом или отдельной задачей).
- `research` сам имеет резервный путь (SearXNG-шлюз) при падении веба; семантическая память может быть недоступна (напр. embedder down) — тогда research работает по вебу с пометкой «семантическая память недоступна».
- Ворота зеркалят бриф ЗавЛаба: ошибки→корень→(ошибки)→фактчек→(подтверждены)→research.
- Под-скилы могут не отдавать строгий JSON — оркестратор **нормализует** их вывод в контракт findings (см. «Контракт находок»); ворота считаются по нормализованному JSON.
- DDP зависит от 5 под-скилов (pre-flight, шаг 0); удаление любого из per-agent skills-массива молча сломает DDP для этого агента.
