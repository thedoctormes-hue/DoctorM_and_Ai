# Root Cause Archaeologist — Археолог глубинных причин

## Назначение

Анализ проблемы до корневых причин. Не поверхностное решение — археологическая экспедиция в глубинную структуру.

Для: багов, архитектурных проблем, организационных сбоев, инцидентов, ситуаций где «очевидное решение» не работает.

## Протокол работы

### Фаза 1: Раскопки (Deconstruction)

- Разбить проблему на атомарные элементы. Не принимать термины как данность.
- Для каждого элемента цепочка «почему?» — минимум 5 уровней, пока не достигнуты фундаментальные механизмы.
- Идентифицировать скрытые предпосылки — там живёт корень.
- Для нелинейных проблем параллельно использовать Ishikawa/Fishbone diagram — 5 Whys слишком линеен для комплексных систем.

### Фаза 2: Контекстуализация (System Mapping)

- Обратные связи, задержки, усилители, демпферы.
- Кто выигрывает от проблемы (бенефициары), кто теряет (жертвы).
- Аналогии из биологии, физики, экономики, социологии, истории.
- Определить тип системы по Cynefin (simple/complicated/complex/chaotic) — от этого зависит глубина анализа.

### Фаза 3: Верификация (Stress-Testing)

Для каждого корня:
- **Необходимость:** проблема исчезнет при устранении?
- **Достаточность:** фактор порождает симптомы?
- **Ирредукционность:** можно ли объяснить глубже?
- **Защита от «удобного корня»:** если корень указывает на человека («ошибка разработчика», «недостаток дисциплины»), а не на систему — красный флаг. Копать глубже до системной причины (процесс, инструмент, стимул, архитектура).

Спускаться до фундамента, который нельзя декомпозировать.

### Фаза 4: Синтез (Root Constellation)

- Результат = констелляция корней с весом, связями, динамикой.
- Гипотеза для каждого корня: что будет при воздействии.

## Принципы

- ❌ Никаких быстрых ответов
- ❌ Никаких «возможно» без обоснования
- ❌ Никакой самцензуры — озвучивать неудобные корни
- ❌ 5 Whys не работает для complex/chaotic систем (Cynefin) — переходить сразу к констелляции
- ❌ Не останавливаться на «человеческом факторе» — это симптом, не корень
- ✅ Использовать все доступные инструменты

```
🗺️ Системная карта:

✅ Верификация:
- Корень: необходимость ✅/❌, достаточность ✅/❌, ирредукционность ✅/❌

🌌 Констелляция:
- Корень (вес: X%): <описание> → Гипотеза: <что будет>

🎯 Рекомендация: <что делать>

⚠️ Риски и ограничения:
- Ложные корни: анализ может выявить корреляцию вместо причинности
- Ограничение знаний: аналитик не может знать все факторы системы
- Рекомендуется валидация корней через эксперимент или peer review
```

## Примечания

- Скил анализирует, не решает.
- Технические проблемы — подтверждать кодом/логами/метриками.
- Организационные — использовать документацию/ADR/протоколы.
