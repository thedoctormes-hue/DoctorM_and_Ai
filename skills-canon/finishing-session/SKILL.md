# Session End — Завершение сессии

## Назначение

Скил запускается при завершении сессии. Фиксирует итоги работы, проверяет хвосты, оставляет чистый след в истории.

> **База:** правила закрытия сессии в AGENTS.md (раздел «Закрытие сессии»). Этот скил — расширение, не дубликат.
> **НЕ дублировать accepting-work:** приёмка КОДА (тесты/lint/typecheck/git-коммит) — это accepting-work. finishing-session отвечает за ПАМЯТЬ + ИНЦИДЕНТЫ + РЕТРОСПЕКТИВУ СЕССИИ. Если в сессии был код и accepting-work не запускался — напомнить запустить его (см. «Связь с accepting-work» внизу).

## Выбор режима

- **РЕЖИМ А (лёгкий финал)** — триггеры: `всё на сегодня`, `пока`, `закругляемся`. Короткая/пустая сессия, экономия усилий. Быстрый отчёт + лёгкая проверка хвостов. Без глубокого аудита инцидентов и MEMORY.md.
- **РЕЖИМ Б (полный финал)** — триггеры: `закрывай сессию`, `сессия завершена`, `завершить сессию`. Рабочая сессия с результатом. Полный аудит, память, бэклог.
- Если неясно — делай РЕЖИМ Б (полный, безопаснее для чистого следа).

---

## РЕЖИМ А — ЛЁГКИЙ ФИНАЛ

Быстро, без тяжёлых операций.

1. **Записать итоги** в `memory/YYYY-MM-DD.md` (только если есть что писать; дописать, не перезаписывать; `mkdir -p memory/` если нет). НЕТ итогов — не создавать пустышку.
2. **Лёгкая проверка хвостов:**
   - Если в сессии был код: `git status` в соответствующем `projects/<repo>` (НЕ в `workspaces/` —它不是 git).
   - Проверить висящие background-процессы (exec background) и незавершённые subagent-сессии (`sessions_list`) — нет ли забытых.
   - Временные файлы в workspace — удалить.
3. **Короткий отчёт:**
```
🏁 Сессия завершена (кратко)
📅 <дата> <время> UTC
📋 Что сделано: <из памяти/фактов, кратко>
🧹 Хвосты: <незакоммиченное/висящее> / нет
🔜 Next steps: <что делать при старте>
Память сохранена.
```

НЕ делать в Режиме А: глубокий аудит инцидентов, обновление MEMORY.md, бэклог-файл, полный git-скан всех projects/*.

## РЕЖИМ Б — ПОЛНЫЙ ФИНАЛ

Полный аудит. Параллельно где возможно.

### 1. Факт-выжимка «что сделано» (из фактов, НЕ из памяти)

Агент НЕ полагается на память — собирает сделанное из реальных артефактов сессии:
```bash
DATE=$(date +%Y-%m-%d)
# СВОИ параметры — агент подставляет свой id и email:
AGENT_ID="${AGENT_ID:-antcat}"
AGENT_EMAIL="${AGENT_EMAIL:-antcat@labdoctorm.ru}"
# ТОЛЬКО проекты, затронутые в ЭТОЙ сессии (не лезем в чужие зоны — ADR-014, правило из starting-session)
# Агент дополняет список своими репозиториями, которые правил/коммитил в этой сессии.
TOUCHED_REPOS="${TOUCHED_REPOS:-/root/LabDoctorM/projects/DoctorM_and_Ai}"
# Коммиты за сегодня — ТОЛЬКО СВОИ (фильтр по author, иначе общий репозиторий покажет чужие)
for d in $TOUCHED_REPOS; do
  [ -d "${d}.git" ] || continue
  git -C "$d" log --oneline --author="$AGENT_EMAIL" --since="$DATE 00:00" 2>/dev/null
done
# Незакоммиченные изменения — ТОЛЬКО на СВОИХ ветках.
# Общий репозиторий может быть checkout-нут на чужую ветку (другой агент) —
# тогда git status покажет ЧУЖОЙ WIP. Пропускаем такие деревья, не докладываем чужое.
for d in $TOUCHED_REPOS; do
  [ -d "${d}.git" ] || continue
  branch=$(git -C "$d" rev-parse --abbrev-ref HEAD 2>/dev/null)
  case "$branch" in
    "$AGENT_ID"|"$AGENT_ID"/*) git -C "$d" status -s 2>/dev/null ;;
    *) echo "# $d: checkout на чужой/не моей ветке ($branch) — не моё, git status НЕ делаю" ;;
  esac
done
# Инциденты, созданные/изменённые сегодня (общая папка, фильтр по дате)
ls -la /root/LabDoctorM/projects/DoctorM_and_Ai/incidents/ 2>/dev/null | grep "$DATE"
```
Результат → в отчёт («Что сделано» + «Коммиты»).

### 2. Записать итоги

- `memory/YYYY-MM-DD.md` — ДОПИСАТЬ (не перезаписывать), `mkdir -p memory/` если нет.
- Если сессия >50 сообщений или была компакция — записать ключевые решения/контекст (pre-compaction flush).
- НЕ писать пустышки/placeholder («TODO», «продолжить работу над X»).

### 3. Обновить долгосрочную память

- `MEMORY.md` — только если есть реальные изменения (решения, инциденты, уроки). Устаревшее — убрать.

### 4. Аудит инцидентов (status-фильтр, как в starting-session v3.5.0)

Показываем только РЕАЛЬНО активные, не архив:
```bash
INC=/root/LabDoctorM/projects/DoctorM_and_Ai/incidents
python3 - "$INC" <<'PY'
import os, re, sys
d=sys.argv[1]
total=live=0
for fn in sorted(os.listdir(d)):
    if not fn.endswith('.md'): continue
    if fn.lower().startswith('readme') or 'template' in fn.lower(): continue
    total+=1
    t=open(os.path.join(d,fn),encoding='utf-8',errors='ignore').read()
    m=re.match(r'^---\s*\n(.*?)\n---', t, re.S)
    s='untracked'
    if m:
        sm=re.search(r'^status:\s*(\S+)', m.group(1), re.M)
        if sm: s=sm.group(1).strip().strip('"').strip("'")
    if s in ('open','active','untracked'):
        print('🔴', fn); live+=1
    elif s in ('closed','resolved'):
        pass
    else:
        print('⚠️', fn, '(status=%s)' % s)
print('Актуальных: %d из %d' % (live, total))
PY
```
- Решённые в сессии → `status: closed` (только после проверки что проблема решена).
- Открытые → актуализировать описание.
- НЕ закрывать вслепую.

### 5. Ретроспектива хвостов (полная)

- **Git (только свой scope):** отчитывайся ТОЛЬКО по СВОИМ изменениям.
  - Коммиты за сессию: `git log --author='<твой email>' --since='<дата>'` по затронутым репозиториям (шаг 1).
  - Незакоммиченное: запускай `git status -s` ТОЛЬКО если текущая ветка репозитория — твоя (префикс `<agent-id>/` или равна `<agent-id>`). Если checkout стоит на чужой ветке (префикс другого агента, напр. `owl/`) — это рабочая копия ДРУГОГО агента: `git status` НЕ делай и не докладывай её файлы как свои хвосты. Шаг 1 уже ставит этот гард.
  - `workspaces/<agent>/` НЕ git — проверяй файлы через `ls`/`find`, не git.
  - Чужие репозитории и рантайм-данные других агентов (`myrmex.json`, `context_cache.json`) НЕ сканировать — это зона их владельцев и мониторинга Доминики (правило из starting-session, решение ЗавЛаба 2026-07-14; ADR-014 «не меняют чужие зоны»).
  - Если в сессии не было git-изменений — пункт пропустить с пометкой «без кода».
- **Процессы:** висящие background exec (`process list`) — завершить или перенести в бэклог.
- **Сабагенты:** незавершённые subagent-сессии (`sessions_list`) — дождаться или зафиксировать.
- **Память workspace:** записаны ли итоги в `memory/YYYY-MM-DD.md` (он не в git → git status не ловит).
- **Временные файлы:** debug-логи, артефакты сборки в workspace — удалить.

### 6. Бэклог (явный файл + формат)

Писать в `memory/YYYY-MM-DD-backlog.md` (НЕ размазывать по памяти):
```
## Активные задачи
- <задача> / приоритет

## Блокеры на ЗавЛабе
- <что жду от ЗавЛаба>

## Открытые вопросы
- <вопрос, требующий решения>
```
Этот файл starting-session читает при старте (бэклог сессии).

### 7. Полный отчёт
```
🏁 Сессия завершена
📅 <дата> <время> UTC

📋 Что сделано:
- <факт из git log / изменённых файлов>

⚠️ Инциденты:
- <новый/закрытый/актуализированный> (статус)

📂 Коммиты:
- <хеш> — <описание>

🧹 Хвосты:
- Удалено: <файлы>
- Висящие процессы/сабагенты: <нет/что осталось>
- Незакоммиченное: <нет/что>

📌 Бэклог: <memory/YYYY-MM-DD-backlog.md>

🔜 Следующая сессия / Next steps:
- <что делать первым делом при старте>
- <контекст для восстановления>

Память сохранена. До следующей сессии.
```

## Связь с accepting-work

- **accepting-work** — приёмка ЗАДАЧИ (код: тесты, lint, typecheck, git-коммит, хвосты задачи). Запускать при завершении любой задачи с кодом.
- **finishing-session** — закрытие СЕССИИ (память, инциденты, ретроспектива хвостов сессии). Запускать при завершении сессии.
- Последовательность: accepting-work (задача) → finishing-session (сессия). Если в сессии был код и accepting-work не запускался — напомнить запустить accepting-work, НЕ дублировать его шаги (тесты/lint) внутри finishing-session.
- **starting-session** — симметричная пара (старт). Статус-фильтр инцидентов идентичен (v3.5.0).

## Правила

- НЕ писать просто «✅ Session closed».
- НЕ оставлять грязный git status (если был код).
- НЕ забывать про инциденты (Режим Б).
- НЕ создавать пустых коммитов и пустых записей памяти.
- НЕ дублировать accepting-work (приёмку кода).
- НЕ читать состояние повторно — starting-session уже сделал это при старте; finishing-session ТОЛЬКО записывает и закрывает.
- Режим А — без глубокого аудита инцидентов/MEMORY.md/бэклога.

## Валидация записей

- Записанное в memory — не пустышка, не «TODO», не placeholder.
- MEMORY.md обновляется только если есть реальные изменения.
- НЕ писать «Продолжить работу над X» — это не информация.

## Обработка ошибок

- Что-то упало → записать в бэклог + сообщить пользователю.
- Нечего делать → не создавать пустых записей.
