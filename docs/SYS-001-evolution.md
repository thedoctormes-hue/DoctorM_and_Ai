# EVOLUTION — Система эволюции лаборатории

> Лучшая в мире система самообучения для AI-лаборатории.
> Синтез лучших практик: CrewAI Memory, Generative Agents, Reflexion, Voyager, MADR.

## Миссия

Превратить каждый рабочий цикл лаборатории в накопление знаний. Не терять инсайты. Не дублировать решения. Не захламлять систему мусором. Автоматически консолидировать наблюдения в знания, знания в артефакты, артефакты в улучшения.

## Ключевые определения

### Инсайт
Знание, извлечённое из **результата** (tool_output), а не из описания действия (tool_input). Инсайт — это то, что изменило понимание системы.

Пример мусора: "Выполнил grep по коду для поиска getCwd" — это действие, не инсайт.
Пример инсайта: "15 вызовов getCwd в 8 файлах — баг с CWD системный" — это знание.

### Артефакт
Любой .md-файл с YAML frontmatter, который **является** решением/конвенцией/знанием для лаборантов. Артефакт — верифицированная единица знания с ID, статусом и жизненным циклом.

### Инструмент
Исполняемый код/скрипт/конфиг, который **делает** что-то. Хуки, скиллы, роли — это инструменты. Они не содержат знаний в frontmatter (кроме своих .md-описаний).

## Иерархия знаний

```
Наблюдение (raw)
  ↓ фильтр качества (Reflexion)
Инсайт (verified knowledge)
  ↓ батчевая консолидация (Generative Agents)
Артефакт (registered knowledge)
  ↓ верификация временем
Улучление (evolution applied)
```

### Инсайт → Артефакт
Инсайт становится артефактом, когда лаборант подтверждает его ценность. Без подтверждения — остаётся в индексе памяти (MEMORY.md), при отсутствии повторного подтверждения — отмирает (decay).

### Артефакт → Улучшение
Активный артефакт (ADR, PAT, RUL) применяется хуками/скиллами в рабочих циклах. Эфект отслеживается через метрики.

## Система инсайтов v3

### Принципы (из лучших практик)

1. **Захват вывода, не вдоха** (ReAct: Observation = result). Инсайт из tool_output, не tool_input.
2. **Только значимые события** (Reflexion). Фильтр: неудачи, аномалии, открытия, подтверждения. Быт — не инсайт.
3. **Батчевая консолидация** (Generative Agents). 100 наблюдений → 3 вопроса → 3 ответа. Не каждый инсайт отдельно.
4. **Композитный скоринг** (CrewAI). relevance × 0.5 + recency × 0.3 + importance × 0.2
5. **Экспоненциальный decay** (CrewAI). `0.5^(age_days / 30)` — старые знания теряют вес.
6. **Верификация через повтор** (Anthropic Claude Code Memory). Инсайт записывается только после 2-го подтверждения.
7. **Ограниченный буфер рефлексий** (Reflexion). Макс 3 рефлексии в контексте.

### Архитектура capture → consolidation → registration

```
PostToolUse hook
  ↓
insight_catcher.sh v3
  ├─ Извлечь tool_output (result), НЕ tool_input (intent)
  ├─ Фильтр качества (Reflexion: failure/anomaly/discovery only)
  ├─ Фильтр длины (мин 20 символов осмысленного текста)
  ├─ Дедупликация (SHA256 хэш content)
  └─ Записать в insights_queue.json (status: probing)
       ↓
insights_consolidator.sh (каждые 30 мин)
  ├─ Берёт probing-инсайты из очереди
  ├─ Группирует по семантической близости
  ├─ Батч из N наблюдений → генерирует 3 вопроса → отвечает
  ├─ Скоринг: relevance × 0.5 + recency × 0.3 + importance × 0.2
  ├─ При similarity > 0.85 к существующему артефакту → update
  ├─ При relevance > 0.8 и новый context → create artifact
  └─ Помечает обработанные как consolidated
       ↓
При подтверждении лаборантом (2-й раз)
  └─ Регистрация как артефакт через register_artifact
```

### Формат записи в insights_queue.json

```json
{
  "version": 3,
  "insights": [
    {
      "id": 1,
      "timestamp": "2026-06-03T12:00:00+03:00",
      "tool_name": "run_shell_command",
      "tool_output_hash": "sha256_выхода",
      "content": "Извлечённое знание из tool_output",
      "status": "probing | consolidated | registered | dropped",
      "hash": "sha256_контента",
      "relevance": 0.0,
      "importance": 0.5,
      "source_event": "failure | anomaly | discovery | confirmation",
      "confirmation_count": 0,
      "related_artifacts": []
    }
  ]
}
```

### Статусы инсайтов

| Статус | Значение |
|--------|----------|
| probing | Новый, ожидает консолидации |
| consolidated | Обработан батч-консолидацией |
| registered | Подтверждён лаборантом, стал артефактом |
| dropped | Отфильтрован (мусор, дубль, неценный) |

### Пороги и числа (из лучших практик)

| Параметр | Значение | Источник |
|----------|----------|----------|
| Консолидация: similarity  merge | > 0.85 | CrewAI |
| Дедупликация внутри батча | ≥ 0.98 | CrewAI |
| Скоринг: semantic weight | 0.5 | CrewAI |
| Скоринг: recency weight | 0.3 | CrewAI |
| Скоринг: importance weight | 0.2 | CrewAI |
| Recency half-life | 30 дней | CrewAI |
| Reflection trigger: N наблюдений | 100 | Generative Agents |
| Reflection output: N вопросов | 3 | Generative Agents |
| Reflection: max в буфере | 3 | Reflexion |
| Skill: max попыток генерации | 4 | Voyager |
| Мин. длина контента инсайта | 20 символов | Наше |

## Система памяти

### MEMORY.md — индекс (макс 200 строк / 25KB)

Из Anthropic Claude Code Memory: MEMORY.md — это **индекс**, а не хранилище. Каждая запись: одна строка ~150 символов, ссылка на topic-файл.

```markdown
# Memory Index

- [topic_name](topic_file.md) — однострочный hook
```

Детали загружаются on demand. Если индекс растёт >200 строк — консолидация (объединить мелкие топики, удалить неактуальные).

### Topic-файлы

Организация по темам, не по хронологии. Каждый файл:

```markdown
---
name: topic_name
description: Однострочное описание для поиска
type: user | feedback | project | reference
---

Содержание: правило + Why + How to apply
```

### Правила записи

- Запись только после **2-й коррекции/подтверждения** от лаборанта (Anthropic)
- При захвате: **Why** (причина) + **How to apply** (применение)
- Старые записи с decay < 0.1 (90+ дней без обращений) — в архив
- Не записывать: кодовые паттерны, пути к файлам, git-историю — это выводимо из кода

## Жизненный цикл артефактов

### Создание
1. Инсайт накапливается в очереди (probing)
2. Консолидация группирует и формирует кандидата
3. Лаборант подтверждает (1-й раз → в индекс, 2-й раз → артефакт)
4. register_artifact создаёт файл с frontmatter в нужной папке

### Обновление
- Артефакты **иммутабельны** — не редактируются (Nygard MADR)
- Изменение = новый артефакт с `supersedes: OLD-NNNN`
- old помечается `status: superseded` (не удаляется!)

### Устаревание
- Recency decay: через 90 дней без обращений → `status: deprecated`
- Через 180 дней без обращений → `status: archived`
- Принудительная проверка 1 раз/квартал (artifact_audit.sh)

### Суперседирование

```
ADR-005: Использовать MongoDB (accepted)
  ↓
ADR-012: Переход на PostgreSQL (accepted)
  └── supersedes: ADR-005
      └── ADR-005.status = superseded
```

## Автоматический аудит

### artifact_audit.sh (еженедельно)

Проверяет:
1. **Консистентность frontmatter** — все ли обязательные поля заполнены
2. **Дубликаты** — SHA256 контента (без frontmatter), порог > 0.95
3. **Битые ссылки** — все `supersedes:`, `related:` указывают на существующие файлы
4. **Пустые шаблоны** — артефакты, созданные шаблоном, но не заполненные
5. **Статус-дрифт** — `proposed` > 30 дней без движения
6. **Размер MEMORY.md** — не > 200 строк
7. **Размер insights_queue.json** — не > 100 probing записей

### Метрики здоровья

| Метрика | Целевое | Критическое |
|---------|---------|-------------|
| Размер MEMORY.md | < 200 строк | > 250 строк |
| Размер insights_queue.json | < 100 probing | > 200 probing |
| Дубликаты артефактов | 0 | > 0 |
| Битые ссылки | 0 | > 3 |
| Пустые шаблоны | 0 | > 5 |
| Время консолидации | < 30 мин | > 60 мин |

## Границы системы

### Что НЕ является инсайтом
- Описание действия ("выполнил grep", "запустил тест")
- Бытовая информация ("перезапустил сервер", "обновил пакет")
- Одноразовые операции без системного паттерна
- Дубли уже существующих артефактов

### Что НЕ является артефактом
- Исполняемый код (хуки, скиллы, роли — это инструменты)
- Временные файлы, логи, кэш
- Документация к внешним проектам (gastro-bot, myrmex и т.д.)
- Файлы без frontmatter

### Что НЕ хранится в памяти
- Кодовые паттерны, пути к файлам, git-история — выводимо из кода
- Эпhemeral состояние текущей сессии
- Списки PR, активность — есть git log / git blame
