## Осиротевшие зоны (инциденты есть, правила/ADR нет)

### ⚠️ data_loss / восстановление (5 инцидентов)
Семантический поиск НЕ нашёл лаб-широкого правила по бэкапам/DR.
Есть только проектные runbook'и (vpn-daemon/runbook_restore, remote-access/emergency-vpn-restore)
и скрипты (backup-policy-audit.sh, backup-verify.sh, disk-healing.sh — незакоммичены).
**Кандидат на создание:** `RUL-` или ADR «Дисциплина бэкапов и восстановления лаборатории»
(что бэкапить, как часто, где хранить, как восстанавливать, кто ответственный).

### ⚠️ Единый RACI ролей агентов
Роли описаны разрозненно в IDENTITY.md каждого агента (owl, kotolizator, …),
но нет СВОДНОЙ матрицы «кто за что отвечает / кто строит, кто аудитит / эскалация».
**Кандидат на создание:** `docs/agent-raci.md` — сводная матрица ответственности.

## Полезные артефакты, которые СТОИТ создать (не дублируя)

1. **Этот файл** (`incident-knowledge-map.md`) — связует инциденты с существующими
   знаниями; предотвращает повторное изобретение правил.
2. **`RUL`/ADR по бэкапам и DR** — закрыть осиротевший кластер data_loss.
3. **`docs/agent-raci.md`** — сводная матрица ролей (аудит↔сборка↔координация).
4. **Security-runbook** (свести git-what-to-commit + vulnerability-plan + секрет-скан)
   — один источник правды по безопасности, вместо 3 разрозненных.

## Как пользоваться
Новый инцидент → проверить здесь, в каком failure-моде. Если есть связанный артефакт —
сослаться, не создавать новый. Если попал в «осиротевшие» — это сигнал создать правило/ADR.
