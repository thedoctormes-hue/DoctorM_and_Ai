# Методология создания и редактирования скилов OpenClaw (лаборатория ЗавЛаба)

**Статус:** авторитетный рабочий документ лаборатории
**Дата актуальности:** 2026-07-13 (факты верифицированы по локальным docs OpenClaw и выводу `openclaw skills list`)
**Применение:** все операции со скилами в `/root/LabDoctorM` и на хосте `~/.openclaw`

Скил — это markdown-инструкция (`SKILL.md`) с YAML-frontmatter, которая учит агента *как и когда* пользоваться инструментами. Ниже — практическое руководство: когда создавать, где хранить, как писать/править/переименовывать/удалять, и главное — как не сломать allowlist.

---

## 1. Когда создавать скил

- **Повторяющееся последовательное действие = повод создать скил.** Если одну и ту же цепочку шагов агент (или ты) выполняешь в третий раз вручную — выноси её в скил.
- Хорошие кандидаты:
  - Протокол/регламент (приёмка работы, аудит, фактчек, завершение сессии).
  - Обёртка над внешним CLI/API (whisper, sherpa-tts, yandex, git-лаб).
  - Многошаговый рабочий процесс, который легко забыть или сократить.
- **НЕ** создавай скил ради одноразовой задачи и ради «более красивого текста» — скил стоит держать в prompt-каталоге только если он реально переиспользуется.
- Сомневаешься — проверь сначала ClawHub (`openclaw skills install`) и внутренний `labsearch`, может уже есть готовое.

---

## 2. Где живут скилы (6 уровней загрузки)

OpenClaw загружает скилы из нескольких корней. **Чем выше уровень (меньше цифра), тем выше приоритет** — при совпадении имени побеждает корень с меньшим номером.

1. **Workspace skills** — `<workspace>/skills` (высший приоритет). Виден **только агенту** этого workspace. Для нашей лабы `<workspace>` = `/root/LabDoctorM/workspaces/<агент>/`.
2. **Project-agent skills** — `<workspace>/.agents/skills`. Только агенту этого workspace.
3. **Personal-agent skills** — `~/.agents/skills`. Виден всем агентам машины.
4. **Managed / local skills** — `~/.openclaw/skills`. Виден **всем агентам хоста**. *Это основное место кастомных скилов лабы.*
5. **Bundled skills** — поставляются вместе с установкой OpenClaw.
6. **Extra directories** — `skills.load.extraDirs` + plugin skills (низший приоритет).

**Для нашей лаборатории:**
- `managed` (`~/.openclaw/skills`, уровень 4) — общие скилы, видны всем 9 агентам хоста. Сюда кладём то, что нужно лабе целиком (например, `skill-manager`, `300-vision`, `ebsh`, `deep-dive`, `yandex-suite` и т.д.).
- `workspace` (`<workspace>/skills`, уровень 1) — скил только для конкретного агента.
- Имя скила, slash-команда и ключ allowlist берутся из поля `name` во frontmatter (а не из имени папки). Папка — только для организации; вложенность допустима (`skills/group/myskills/SKILL.md` → имя из `name`).

## 3. Создание скила — шаг за шагом

**Шаг 1. Создать папку и `SKILL.md`.**
- Managed (для всех агентов): `~/.openclaw/skills/<name>/SKILL.md`
- Workspace (для одного агента): `<workspace>/skills/<name>/SKILL.md`
- Имя папки и `name` во frontmatter держи согласованными.

**Шаг 2. Написать frontmatter.**
Обязательные поля:
- `name` — уникальный slug: **только lowercase-буквы, цифры и дефисы** (например `deep-dive`, `root-cause-archaeologist`).
- `description` — одна строка, **не более 160 байт** (это жёсткий лимит; для кириллицы байты считаются, не символы). Показывается агенту и в выводе `/skill`.

Наши лабные расширения frontmatter (конвенция `skill-manager`):
- `version: "x.y.z"` — semantic versioning.
- `last_reviewed: "YYYY-MM-DD"`.
- `status: active | proposal | deprecated`.
- `user-invocable: true` — официальное поле: скил доступен как slash-команда пользователю.
- `triggers:` — наша лабная конвенция (кастомное поле, не часть спецификации AgentSkills): `phrases` / `patterns` / `scope` для быстрого обнаружения.

Пример минимального `SKILL.md`:

```markdown
---
name: my-skill
description: Краткое описание назначения скила (≤160 байт).
version: "1.0.0"
last_reviewed: "2026-07-13"
status: active
user-invocable: true
triggers:
  phrases:
    - "сделай то-то"
  scope:
    - рутинная операция лаборатории
---

# My Skill

Когда пользователь просит X, выполни шаги: ...
Используй `{baseDir}` для ссылки на файлы внутри папки скила.
```

**Шаг 3. Gating через `metadata.openclaw` (опционально, но желательно).**
`metadata.openclaw` — это **однострочный JSON** в frontmatter (парсер понимает только single-line ключи). Он фильтрует скил на этапе загрузки:

```markdown
---
name: my-skill
description: ...
metadata: { "openclaw": { "requires": { "bins": ["uv"], "env": ["GEMINI_API_KEY"], "config": ["browser.enabled"] }, "os": ["linux"], "always": false } }
---
```

Поля gating:
- `requires.bins` — все бинари должны быть в `PATH`.
- `requires.anyBins` — хотя бы один из списка.
- `requires.env` — каждая переменная должна быть в окружении или в конфиге.
- `requires.config` — каждый путь `openclaw.json` должен быть truthy.
- `os` — фильтр платформы (`darwin` / `linux` / `win32`).
- `always: true` — всегда включать и пропустить остальные гейты.
- `primaryEnv` — переменная, связанная с `skills.entries.<name>.apiKey`.

Без блока `metadata.openclaw` скил считается всегда допустимым (если не отключён явно).

**Шаг 4. КРИТИЧНО — зарегистрировать скил в allowlist (см. раздел 6).** Без этого скил загрузится, но будет `excluded` для агентов и агент его «не увидит».

**Шаг 5. Верификация (см. раздел 7).**

**Шаг 6. ОБЯЗАТЕЛЬНЫЙ walkthrough-гейт (см. раздел 8).**

> Альтернатива прямому редактированию файла — **Skill Workshop** (`skill_workshop` тул / `openclaw skills workshop propose-create`). Путь «предложение → проверка → apply» управляем ЗавЛабом, пишет только workspace-скилы и не трогает managed/bundled.

## 4. Редактирование скила — шаг за шагом

1. **Бэкап** `SKILL.md` перед правкой (например `cp SKILL.md SKILL.md.bak.<дата>`). Никаких `rm` до проверки.
2. Правь frontmatter и/или тело. Следи за правилами: `name` — lowercase/цифры/дефисы; `description` ≤160 байт; `metadata.openclaw` — строго одна строка JSON.
3. Если меняешь логику — подними `version` на patch (`1.0.0` → `1.0.1`) и обнови `last_reviewed`.
4. **Имя (`name`) при правке менять нельзя** — это фактически переименование (см. раздел 5). Если имя всё же меняешь, проходи процедуру переименования целиком.
5. Если скил уже в allowlist — перерегистрировать не нужно (видимость зависит от имени, не от содержимого).
6. Верификация (раздел 7) + walkthrough-гейт (раздел 8).

## 5. Переименование и удаление (принцип «скил = зависимость»)

Основной принцип: **скил — это зависимость.** Любое изменение имени/жизненного цикла затрагивает allowlist, конфиг и, возможно, ссылки из других скилов.

### Переименование
**Критично: сначала добавить НОВОЕ имя везде → потом удалить СТАРОЕ.** Никогда не удаляй старое раньше, чем новое зарегистрировано и проверено.

1. Создать новую директорию, скопировать `SKILL.md`.
2. Обновить `name` и `version` (инкремент minor) во frontmatter.
3. Добавить НОВОЕ имя в allowlist (defaults + каждый explicit-список агента).
4. Удалить СТАРОЕ имя из allowlist (defaults + каждый explicit-список).
5. Удалить старую директорию.
6. Проверить: новый `ready`, старого нет в `openclaw skills list`.
7. Найти и обновить ссылки на старое имя в других скилах: `grep -r "old-name" ~/.openclaw/skills/*/SKILL.md`.

### Удаление
1. `grep` по всем `SKILL.md` — нет ли ссылок на удаляемый скил в других.
2. Удалить имя из `agents.defaults.skills`.
3. Удалить имя из `agents.list[].skills` **каждого** агента.
4. Удалить директорию `~/.openclaw/skills/<name>/` (или workspace-эквивалент).
5. Проверить: скил отсутствует в `openclaw skills list`.

**Запрещено:** удалять скил, на который есть ссылки в других скилах, без предварительного обновления этих ссылок.

## 6. КРИТИЧЕСКОЕ ПРАВИЛО allowlist (читать каждый раз перед правкой конфига)

Видимость скила агенту определяется **allowlist**, независимо от того, где лежит файл.

- `agents.defaults.skills` — базовый allowlist, наследуется агентами, у которых нет своего списка.
- `agents.list[].skills` — **непустой** список для конкретного агента. Важно:
  - **Непустой `agents.list[].skills` ПОЛНОСТЬЮ ЗАМЕНЯЕТ `defaults` — он НЕ мержится с ним.**
  - Чтобы скил был виден агенту с explicit-списком, его надо добавить **И** в `agents.defaults.skills`, **И** в каждый `agents.list[].skills` этого агента.
  - Добавить скил только в `defaults` — **НЕДОСТАТОЧНО**, если у агента есть свой explicit-список.
- **В нашем конфиге у КАЖДОГО из 9 агентов есть explicit-список** (копия defaults + мелкие добавки). Поэтому новый managed-скил должен попасть и в `defaults`, и во все 9 agent-списков, иначе он будет `excluded` для части/всех агентов.
- `skills.allowBundled` гейтит **ТОЛЬКО bundled-скилы**. Managed- и workspace-скилы на `allowBundled` не влияют — они активируются только через agent-allowlist.
  - *Уточнение (исправляет старое замечание лабы «allowBundled не активирует скилы»):* фраза была неточной. `allowBundled` вообще не касается managed/workspace; он лишь сужает множество bundled-скилов. Managed-скилы «включаются» исключительно agent-allowlist-ом.

### Процедура правки конфига (ADR-0025 / ADR-0048)
1. **Бэкап** `openclaw.json` (например `cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.bak.<дата>`).
2. Внести правку allowlist (defaults + все agent-списки).
3. `openclaw config validate`.
4. `openclaw doctor`.
5. **Рестарт gateway — ТОЛЬКО ЗавЛаб.** Субагент/агент рестарт не делает; в большинстве случаев skills-watcher сам обновит snapshot на следующем тurn-е, явный рестарт не обязателен для активации скила.

**Корень дефекта 2026-07-13 (allowlist-drift):** `deep-dive`, `ebsh`, `root-cause-archaeologist`, `yandex-suite` как новые managed-скилы не попали в allowlist (ни в defaults, ни во все agent-списки) → получили статус `excluded` и были невидимы агентам. Исправлено добавлением всех 4 имён во все allowlist-списки. Это паттерн INC-018 / INC-022 — держи его в памяти при каждой операции.

## 7. Верификация

После любой операции со скилом (создание/правка/переименование/удаление) выполни:

- `openclaw skills list` — проверить статус. Ожидаемый статус для рабочего скила: `✓ ready`.
  - `blocked` — скил загружен, но отфильтрован гейтом (например bundled-скил не в `allowBundled`, или не хватает `requires.bins/env/config`).
  - `excluded` — скил загружен (виден хосту), но **отфильтрован из эффективного списка агента его allowlist** (классический симптом allowlist-drift).
- `openclaw config validate` — проверка синтаксиса/схемы конфига после правки allowlist.
- `openclaw doctor` — общая проверка здоровья (включая статику install policy).

Считай операцию завершённой только когда `openclaw skills list` показывает нужный статус для нужных агентов.

## 8. ОБЯЗАТЕЛЬНЫЙ walkthrough-гейт

После создания или правки скила **нельзя** ограничиваться тем, что он появился в `openclaw skills list`. Требуется пройти скил **шаг за шагом** и найти, где агент «вспотыкается»:

1. Запусти сценарий, который должен триггерить скил (реальное сообщение пользователя или `/skill <name>`).
2. Пройди каждый шаг инструкции скила вслед за агентом.
3. Отметь любые места, где:
   - агент не понял инструкцию или сошёл с неё;
   - не хватает bin/env/config, которые не проверяются gating-ом;
   - инструкция устарела (ссылка на удалённый тул, неверный путь, опечатка);
   - walkthrough ломается из-за прав доступа или симлинков.
4. Вернись в раздел 4, поправь `SKILL.md`, подними `version` (patch), снова верификация (раздел 7) + walkthrough (раздел 8).
5. Только после успешного walkthrough операция считается принятой.

## 9. Типичные ошибки лабы

- **allowlist-drift (исключённые скилы).** Новый managed-скил добавлен в файл, но забыт в allowlist → статус `excluded`, агент его не видит. Паттерн INC-018 / INC-022. Реальный случай: 2026-07-13, `deep-dive`, `ebsh`, `root-cause-archaeologist`, `yandex-suite`. Лечение — добавить имя И в `defaults`, И во все 9 agent-списков.
- **«Добавил только в defaults».** Не работает, потому что у всех 9 агентов есть explicit-списки, которые полностью заменяют defaults (не мержатся).
- **Миф про `allowBundled`.** `allowBundled` гейтит только bundled-скилы; он не «включает» и не «выключает» managed/workspace. Managed активируются только agent-allowlist-ом.
- **Рестарт gateway не тем, кем надо.** Рестарт — прерогатива ЗавЛаба. Субагент/агент не рестартует gateway; для активации скила обычно достаточно skills-watcher + новой сессии/next turn.
- **Правка `name` как «просто переименование».** Это меняет ключ allowlist и ссылки; делай по разделу 5 целиком.
- **Забытая версия/дата.** Без `version` + `last_reviewed` теряется отслеживаемость изменений (конвенция `skill-manager`).
- **Слишком длинный `description`.** Лимит — 160 байт (не символов). Превышение → ошибка при Skill Workshop и обрезка в discovery.
- **Многострочный `metadata.openclaw`.** Парсер понимает только single-line JSON — иначе gating игнорируется.

## 10. Инструменты: skill-creator (bundled) и skill-manager (custom managed)

- **`skill-creator`** — официальный bundled-инструмент OpenClaw (входит в allowlist лабы). Создание/редактирование/валидация `SKILL.md`, проверка frontmatter и gating. Используй его как первичный инструмент авторства скила.
- **`skill-manager`** — наш кастомный managed-скил (живёт в `~/.openclaw/skills/skill-manager`). Отвечает за жизненный цикл: создание, переименование, удаление по принципу «скил = зависимость», версионирование, проверку ссылок, атомарные операции. Дополняет `skill-creator` лабными правилами (allowlist во все списки, ADR-0025/0048, walkthrough-гейт).
- **Skill Workshop (`skill_workshop`)** — управляемый путь «propose → inspect → apply/reject», альтернатива прямому редактированию файла. Пишет только workspace-скилы; managed/bundled не трогает. Удобен, когда изменение должно пройти review ЗавЛаба.

**Резюме правила:** скил без корректного allowlist (defaults + все agent-списки) — мёртвый груз со статусом `excluded`. Создал/поправил → верифицировал `skills list` → прошёл walkthrough → только тогда готово.
