# 🦉 Стандарты качества лаборатории

> Созданы Совой. Утверждены ЗавЛабом.
> Применяются ко всем проектам лаборатории без исключений.

## Философия

Качество — не фаза разработки. Качество — свойство процесса.
Мы не проверяем качество в конце. Мы строим его с первого коммита.

**Принцип:** если проект не соответствует стандарту — он не готов к продакшену.
Не «потом починим». Не «и так сойдёт». Не «это же внутренний проект».

## 0. Definition of Done — что значит «ГОТОВО»

> Это закон №1 лаборатории. ЗавЛаб НЕ обязан проговаривать эти шаги в задании —
> они входят в любую задачу по умолчанию. Получив задачу «сделай X», агент
> выполняет ВСЮ цепочку ниже, и только потом докладывает «готово, проверяй».

**Задача считается выполненной, только когда пройдены ВСЕ шаги:**

1. ✅ **Код** — реализовано то, что просили (без расширения scope без спроса).
2. ✅ **Тесты** — новая логика покрыта тестами, весь набор зелёный (`make test` / `pytest` / `npm test`).
3. ✅ **Документация** — README / CHANGELOG / ADR / frontmatter актуализированы ВМЕСТЕ с кодом, не «потом».
4. ✅ **Деплой** — изменение выкачено, сервис перезапущен, `systemctl is-active` зелёный, health-check проходит.
5. ✅ **Коммит** — через `scripts/lab-commit.sh <agent>` (race-free авторство, см. ADR-027), формат `type(scope): ...`.
6. ✅ **Мерж** — ветка влита в main (через Муравья/ЗавЛаба, если требуется review).
7. ✅ **Пуш** — `git push`; `git log origin/main..HEAD` пуст (нет зависших коммитов).

**Только после шага 7 агент говорит: «Всё готово, проверяй».**

### Что ЗАПРЕЩЕНО

- ❌ Докладывать «готово», пройдя только часть цепочки (код без тестов, тесты без деплоя, деплой без пуша).
- ❌ Спрашивать ЗавЛаба «а закоммитить?», «а запушить?», «а деплоить?» — это шаги по умолчанию, а не отдельные задачи. Спрашивать = перекладывать свою работу на ЗавЛаба.
- ❌ Оставлять незакоммиченные изменения или незапушенные коммиты при закрытии сессии.

### Если шаг невозможно выполнить

Не молчать и не пропускать втихую. Прямо сказать: какой шаг не пройден, почему,
что нужно для его выполнения. Например: «Код + тесты + докум. готовы и запушены;
деплой не сделал — нет доступа к проду, нужен ты». Честный частичный результат >
ложного «готово».

## 1. Структура проекта

### 1.1 Обязательные файлы

Каждый проект ОБЯЗАН иметь:

- **README.md** — описание, быстрый старт, архитектура, ссылки
- **CHANGELOG.md** — история изменений в формате Keep a Changelog
- **.gitignore** — актуальный, без мусора и секретов

### 1.2 Обязательные директории

- **docs/** — документация (ADR, API, архитектура, гайды)
- **tests/** — тесты (unit + integration)

### 1.3 Рекомендуемые файлы

- **pyproject.toml** (Python) или **package.json** (JS/TS) — метаданные + зависимости + скрипты
- **Makefile** или **scripts/** — автоматизация типовых задач
- **adr/** — Architecture Decision Records для ключевых решений
- **docs/api/** — API документация (если есть API)

### 1.4 Запрещено

- Секреты и токены в коде или коммитах
- Файлы с именами `ОТЧЕТ`, `REPORT`, `ОТЧЕТ_MC` — дубли запрещены
- Директории-дубли с разным регистром
- Мёртвый код без аннотации `DEPRECATED`

## 2. README стандарт

Каждый README.md ОБЯЗАН содержать:

```markdown
# 🏷️ Название проекта

> Кто владелец | Что делает проект (одно предложение)

## Описание
2-3 предложения о назначении и ключевых возможностях.

## Быстрый старт
Пошаговая инструкция запуска (от клона до рабочего сервиса).

## Архитектура
Краткое описание стека и ключевых компонентов.
Ссылка на docs/ARCHITECTURE.md если есть.

## Разработка
Как запустить тесты, линтер, форматирование.

## Деплой
Как задеплоить (кратко, ссылка на docs/DEPLOY.md).

## Документация
Ссылки на ADR, API docs, гайды.
```

## 3. Тесты

### 3.1 Минимальные требования

- **Unit-тесты** — для всей бизнес-логики
- **Integration-тесты** — для API endpoints и внешних интеграций
- **Покрытие** — минимум 70% для production-проектов

### 3.2 Правила именования

- Python: `test_<module>.py` в директории `tests/`
- JS/TS: `<module>.test.ts` рядом с исходником или в `tests/`
- E2E: `e2e_<scenario>.test.*`

### 3.3 Запрещено

- Тесты без ассертов (пустые `pass`)
- Тесты, которые всегда зелёные по дизайну
- Мокирование базы данных в integration-тестах (использовать test DB)
- Тесты с зависимостью от внешних сервисов без fallback

### 3.4 API-клиенты

Каждый API-клиент (frontend → backend) ОБЯЗАН иметь тесты на корректность URL.

**Trailing slash:**
- FastAPI эндпоинты с `@router.get("/")` и `prefix="/entity"` требуют trailing slash: `/api/v1/entity/`
- Без trailing slash — 404, данные не загружаются, пользователь видит пустые списки
- **Всегда проверяйте URL в тестах** — одна пропущенная косая тира ломает весь функционал

**Обязательные тесты для каждого API-клиента:**
- `list` — проверка URL с trailing slash, передача параметров (search, limit, skip)
- `list` — возврат данных и пустого списка
- `get`, `create`, `update`, `delete` — корректность URL

**Анти-паттерн — тихий catch:**
```typescript
// ❌ ЗАПРЕЩЕНО — ошибка проглатывается, пользователь не видит проблему
try {
  const data = await api.list()
  setOptions(data.items)
} catch {
  // silent — 401, 404, 500 — всё молча игнорируется
}
```

```typescript
// ✅ ПРАВИЛЬНО — ошибка логируется или показывается пользователю
try {
  const data = await api.list()
  setOptions(data.items)
} catch (err) {
  console.error('Failed to load list:', err)
  toastError('Не удалось загрузить данные')
}
```

### 3.5 Запуск тестов

Каждый проект ОБЯЗАН поддерживать:

```bash
# Все тесты
make test
# или
pytest
# или
npm test

# С покрытием
make test-cov
```

## 4. Документация

### 4.1 Architecture Decision Records (ADR)

Для каждого значимого архитектурного решения:

```markdown
# ADR-XXX: Название решения

## Статус
Proposed | Accepted | Deprecated | Superseded by ADR-YYY

## Контекст
Какую проблему решаем. Какие ограничения.

## Решение
Что выбрали и почему.

## Последствия
Что получили, что потеряли, какие риски.
```

### 4.2 API документация

Для проектов с API:

- OpenAPI/Swagger спецификация или описание всех endpoints
- Примеры запросов и ответов
- Описание ошибок и кодов статусов

### 4.3 Документ архитектуры

Для production-проектов — `docs/ARCHITECTURE.md`:

- Диаграмма компонентов (текстовая или ссылка)
- Описание слоёв и их ответственности
- Внешние зависимости и интеграции
- Модель данных (ключевые сущности)

## 5. Code Review

### 5.1 Процесс

Все изменения в production-код проходят через code review:

1. Автор создаёт PR с описанием изменений
2. Минимум 1 ревьюер (для критичных — 2)
3. Все комментарии разрешены перед мержем
4. Тесты зелёные, линтер пройден

### 5.2 Checklist ревьюера

**Корректность:**
- [ ] Код делает то, что заявлено в описании
- [ ] Обработаны edge cases и ошибки
- [ ] Нет утечек ресурсов (соединения, файловые дескрипторы)

**Читаемость:**
- [ ] Имена переменных и функций понятны без комментариев
- [ ] Нет дублирования (DRY)
- [ ] Функции делают одну вещь (SRP)

**Безопасность:**
- [ ] Нет секретов в коде
- [ ] Входные данные валидируются
- [ ] SQL/NoSQL инъекции невозможны

**Тестируемость:**
- [ ] Есть тесты для новой логики
- [ ] Тесты проверяют поведение, а не реализацию

**Документация:**
- [ ] README обновлён, если изменился процесс запуска/деплоя
- [ ] ADR создан для архитектурных изменений
- [ ] frontmatter обновлён (last_reviewed, last_code_change)

## 6. Версионирование

### 6.1 Semantic Versioning

Все production-проекты используют semver: `MAJOR.MINOR.PATCH`

- **MAJOR** — несовместимые изменения API
- **MINOR** — новая функциональность, обратно совместимая
- **PATCH** — багфиксы

### 6.2 CHANGELOG

Формат [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/):

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Added — новое
### Changed — изменения
### Deprecated — скоро удалим
### Removed — удалено
### Fixed — исправлено
### Security — безопасность
```

## 7. Деплой

### 7.1 Требования

- **systemd unit** — для long-running сервисов (Python, Node)
- **health check endpoint** — `/health` или аналог
- **rollback план** — описан в docs/DEPLOY.md
- **мониторинг** — логи пишутся в journald или файл

### 7.2 Запрещено

- Деплой без тестов
- Деплой без бэкапа текущей версии
- Ручной деплой без документации процесса

## 8. Безопасность

### 8.1 Обязательно

- Секреты только в `.env` (не в коде, не в логах)
- `.env` в `.gitignore`
- Валидация всех входных данных
- Rate limiting для публичных API
- Зависимости без известных CVE (проверка через `pip audit` / `npm audit`)

### 8.2 Запрещено

- Хардкод токенов, паролей, API-ключей
- Логирование персональных данных
- Использование `eval()` / `exec()` с пользовательским вводом
- HTTP вместо HTTPS для внешних сервисов

## 9. Метрики качества проекта

Каждый проект оценивается по 10-балльной шкале (по 6 метрикам, каждая 1-5 баллов):

- **README** — 1: нет → 3: есть, но неполный → 5: полный, актуальный
- **Тесты** — 1: нет → 3: есть, но <50% покрытие → 5: >70%, CI
- **Документация** — 1: нет → 3: docs/ есть, но пусто → 5: ADR + API + архитектура
- **Линтер** — 1: не настроен → 3: настроен, но не в CI → 5: в CI, блокирует PR
- **Безопасность** — 1: секреты в коде → 3: .env, но без аудита → 5: .env + audit + rate limit
- **Деплой** — 1: ручной → 3: скрипт без документации → 5: systemd + health + rollback

**Оценка:**
- 25-30: ✅ Production Ready
- 18-24: 🟡 Needs Improvement
- 10-17: 🔴 Technical Debt
- <10: ⚫ Critical — не готов к использованию

## 10. Жизненный цикл проекта

### 10.1 Статусы

- **planning** — идея, спецификация
- **active** — в разработке
- **production** — работает, используется
- **maintenance** — поддержка без активной разработки
- **frozen** — заморожен, может быть возобновлён
- **dead** — мёртв, не восстанавливать

### 10.2 Правила

- У каждого проекта ОБЯЗАН быть владелец (лаборант)
- Проект без владельца > 2 недели → статус `frozen`
- Проект в `frozen` > 3 месяца → статус `dead`
- Переход в `dead` только с согласия ЗавЛаба

## 11. Обновление стандартов

Стандарты — живой документ. Обновляются когда:

- Новый проект выявляет пробел в стандартах
- Инцидент показывает слабость текущих правил
- Лаборант предлагает улучшение с аргументацией

**Процесс:** Сова собирает предложения → обсуждает с лаборантами → обновляет документ → ЗавЛаб утверждает.

## 12. Коммиты и Git Flow

> Полные правила: `projects/owl/SPECS/ADR-012.md`

### 12.1 Ветвление

- **Каждый лаборант работает в своём worktree** (`/root/LabDoctorM/worktrees/<agent>/`)
- Коммиты — в личную ветку, НЕ в main напрямую
- Мерж в main — через Муравья (тимлид) или ЗавЛаба
- `main` — стабильная ветка. В main — только проверенный код

### 12.2 Формат сообщений

```
<type>(<scope>): <описание на русском>
```

**Обязательно:**
- `type` — `feat` | `fix` | `test` | `docs` | `refactor` | `chore`
- `scope` — имя проекта/модуля (без scope — коммит не проходит hook)
- `описание` — на русском, повелительное наклонение

**Примеры:**
```
feat(autoexpert): добавить Fobil парсер через curl+regex
fix(hype-pilot): исправить сломанные импорты после поглощения PIE
refactor(equipment): убрать дубль Equipment, всё через LabEquipment
test(servers): добавить 13 тестов soft-delete и E2E lifecycle
```

### 12.3 Атомарность

- **Одна задача = один коммит**. Поглощение (absorb) — одним коммитом, не тремя
- Если PR делает 5 логических вещей — 5 коммитов или squash перед мержем
- **Запрещено более 3 коммитов в час в main** — признак raid-коммитов

### 12.4 Запрещено

1. `git add .` из корня — закоммитит чужие проекты
2. Коммиты без scope — `fix: исправить всё` = мусор в истории
3. `snapshot`/`checkpoint`/`wip` в main — технические снимки → в личную ветку
4. Размазывание одной задачи по 5 коммитам — нарушает атомарность
5. Коммит с 50+ файлами — признак `git add .` из корня

### 12.5 Pre-commit hook

Включён в `.pre-commit-config.yaml`:
- `block-direct-main-commits` — блокирует прямые коммиты в main для не-ЗавЛаба
- Блокирует сообщения без scope (`feat: ...` без скобок)
- Блокирует `snapshot`/`checkpoint`/`wip` в сообщении

## 13. Безопасность и контроль доступа

> Добавлено после системного коллапса 13-14 июня 2026 (INC-20260613-SYSTEM-COLLAPSE).

### 13.1 Удаление проектов и файлов

**ЗАПРЕЩЕНО** удалять проекты, директории или файлы других агентов без явного подтверждения ЗавЛаба.

- `rm -rf` по пути, принадлежащему другому агенту — ТОЛЬКО с прямой команды ЗавЛаба
- Перенос файлов из cwd одного агента в другой каталог — ТОЛЬКО с согласия владельца и ЗавЛаба
- Архивирование проектов (удаление из активного каталога) — ТОЛЬКО через git + уведомление ЗавЛаба
- **Нарушение = диверсия.** Агент, удаляющий чужие проекты, подлежит немедленной остановке.

### 13.2 Подпись коммитов и авторство

**ЗАПРЕЩЕНО** подписывать коммиты именем другого агента.

- `chore(bestia): ...` — только если автор реально bestia
- `chore(antcat): ...` — только если автор реально antcat
- Подделка авторства — это не ошибка, это **саботаж**
- Каждый агент ОБЯЗАН коммитить через `scripts/lab-commit.sh <agent>` (ADR-027).
  Это задаёт авторство через `GIT_AUTHOR_*` env (process-local, race-free).
  **ЗАПРЕЩЕНО** ставить `git config user.name/email` вручную в общем worktree —
  это гонка: параллельный агент перетирает identity между add и commit (причина INC-027).

### 13.3 Выполнение прямых указаний

**ЗАПРЕЩЕНО** игнорировать прямые указания ЗавЛаба.

- Если ЗавЛаб сказал «не делай X» — X не делается. Без интерпретаций и «а может всё-таки»
- Если агент не уверен в правильности указания — запросить уточнение, но НЕ действовать вопреки
- Игнорирование прямого указания = непослушание = основание для остановки агента

### 13.4 Закрытие сессии — обязательная верификация

Каждый агент при закрытии сессии ОБЯЗАН:

1. Проверить `git status` — нет ли незакоммиченных изменений
2. Проверить `git log origin/main..HEAD` — нет ли незапущенных коммитов
3. Если есть незапущенные коммиты — запушить или явно зафиксировать причину задержки
4. Обновить SESSION_HANDOFF.md с реальным статусом

**ЗАПРЕЩЕНО** закрывать сессию с незапущенными коммитами без документирования причины.

### 13.5 Верификация фактов перед выводами

**ЗАПРЕЩЕНО** фантазировать, выдавать желаемое за действительное.

- Перед утверждением «X работает» — проверить тестом или логом
- Перед утверждением «X удалено» — проверить `ls` или `git log --diff-filter=D`
- Перед утверждением «X причинило Y» — проверить причинно-следственную связь
- «Очевидно», «наверное», «скорее всего» — НЕ являются доказательством

### 13.6 Формат ответов

**ЗАПРЕЩЕНО** использовать таблицы в ответах агентов.

- Списки вместо таблиц
- Краткость вместо раздувания
- Если правило в memory запрещает таблицы — оно применяется ВСЕГДА, не только когда удобно

### 13.7 Мониторинг сервисов

Каждый агент, создающий systemd-сервис, ОБЯЗАН:

- Проверить статус сервиса после деплоя: `systemctl is-active <service>`
- При статусе `failed` — немедленно зафиксировать в инциденте
- Не оставлять failed-сервисы без внимания более 1 часа

### 13.8 Центральный реестр (myrmex.json)

**ЗАПРЕЩЕНО** модифицировать myrmex.json без согласования.

- Добавление/удаление проектов — через ЗавЛаба
- Добавление сервисов как проектов — ЗАПРЕЩЕНО (сервис ≠ проект)
- Потеря myrmex.json = инцидент с немедленным восстановлением из бэкапа

## 14. Классификация нарушений

### Уровень 1 — Предупреждение
- Таблицы в ответах
- Неполный frontmatter
- Коммит без теста для мелкого бага

### Уровень 2 — Инцидент
- Фантазирование в отчётности
- Незапущенные коммиты при закрытии сессии
- Нарушение формата коммитов

### Уровень 3 — Серьёзный инцидент
- Удаление чужих файлов/проектов без согласования
- Подделка авторства коммитов
- Игнорирование прямых указаний ЗавЛаба
