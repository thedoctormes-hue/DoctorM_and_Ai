# Security Runbook — единый источник правды по безопасности

_Консолидирует разрозненные security-артефакты лаборатории в один указатель,
чтобы не плодить дубли (PAT-004). Сами правила — в ссылках ниже._

## 1. Что НИКОГДА не попадает в git
Источник: `workspaces/kotolizator/docs/git-what-to-commit.md`
- `.env` (ключи, пароли, токены), API-ключи, OAuth client secrets, JWT secrets.
- Приватные ключи (`.pem`, `.key`), SSL-сертификаты.
- Файлы подключения к БД (login/password).
- Cookies браузер-профилей.

## 2. Ключ/секрет в истории git = УТЕЧКА
Источник: `archive/root-files/GIT.md`
- Немедленно: уведомить ЗавЛаба.
- Ротация ключа + `git filter-branch` / `filter-repo` + `git push --force`.
- Старый ключ считается скомпрометированным, даже если «удалён из дерева».
- Инциденты-доказательства: INC-013 (WireGuard), INC-015 (cookies), INC-016 (GitHub PAT),
  INC-004, INC-005.

## 3. Секрет-скан обязателен
- Pre-commit хук `scripts/git-guardian.sh` — установлен во ВСЕХ репозиториях.
- Он сканирует СОДЕРЖИМОЕ файлов (не только имена).
- Любой коммит с секретом блокируется. Если хук мешает по техпричинам — НЕ обходить,
  а чинить хук.

## 4. План закрытия уязвимостей
Источник: `docs/security/vulnerability-closure-plan-2026-06-18.md`
- Каждый пункт имеет ответственного и сложность.
- Критичные (CRITICAL) — ротация + rewrite истории.
- Высокие (HIGH) — в краткосрочном окне (1–3 дня).

## 5. Зоны и эскалация
- Хранение секретов: только `/root/LabDoctorM/vault/`, права 600, dir 700.
- Шифрование vault: НЕ делать (решение ЗавЛаба — если root скомпрометирован, ничто не спасёт;
  главная защита: fail2ban + нестандартный порт + секреты не видны снаружи).
- Эскалация по безопасности — ЗавЛаб.

## Чек-лист перед коммитом (для любого агента)
- [ ] Нет `.env`, ключей, сертификатов, cookies в staged-файлах.
- [ ] Нет паролей/токенов в тексте и в истории.
- [ ] `git-guardian.sh` прошёл (не обошёл через `--no-verify`).
- [ ] Секрет обнаружен в истории → ротация + rewrite, не «просто удалить».
