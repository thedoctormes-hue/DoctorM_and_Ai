## Инсайты из исследования

### 1. OpenClaw поддерживает openai-compatible провайдеры для memorySearch

Из документации OpenClaw:
```
memorySearch.provider = "openai-compatible"
memorySearch.model = "bge-m3"
memorySearch.remote.baseUrl = "http://127.0.0.1:8081"
```

Это значит: **не нужен отдельный плагин** — достаточно поднять HTTP-прокси с OpenAI-совместимым API.

### 2. Инкрементальная переиндексация — стандартный паттерн в production RAG

Полная переиндексация 5777 чанков занимает ~1 час. Но ежедневно меняются единицы файлов.

**Решение:** хранить хеш (SHA-256) каждого файла/чанка. При переиндексации:
- сравнить хеш с сохранённым
- обработать только изменённые/новые
- удалить удалённые из индекса

**Результат:** ежедневная переиндексация = **минуты, не часы**.

### 3. FAISS — для прототипа отлично, для production есть нюансы

**Плюсы FAISS:**
- бесплатный, open-source
- быстрый in-memory поиск
- полный контроль

**Минусы FAISS:**
- нет встроенной персистентности (при перезапуске — пересборка)
- нет аутентификации
- нет автоматического масштабирования

**Вывод:** для нашего случая (один сервер, ~6000 чанков) FAISS — оптимальный выбор. Пересборка из SQLite занимает ~30 секунд.

### 4. Гибридный поиск — best practice

Чисто семантический поиск иногда пропускает точные совпадения. Рекомендация из сообществ:
- **dense (embedding) + sparse (BM25/FTS5)** = гибридный поиск
- SQLite FTS5 уже встроен — можно использовать для lexical поиска
- Комбинация даёт лучшую точность

## План действий

### Этап 1: Завершение индексации (текущий)
- Дождаться завершения owl и raven (~30-60 мин)
- Пересобрать FAISS с полными данными

### Этап 2: FastAPI-прокси (1-2 часа работы)
Создать `lab-search-proxy.py` на FastAPI:
- `POST /v1/embeddings` → прокси к Ollama (OpenAI-совместимый)
- `POST /search` → поиск по FAISS + FTS5 (гибридный)
- `GET /health` → статус сервиса
- `POST /reindex` → запуск переиндексации (инкрементальной)

Запустить как systemd-сервис на порту 8081.

### Этап 3: Интеграция с OpenClaw (30 мин)
Обновить `openclaw.json`:
```json
{
  "agents": {
    "defaults": {
      "memorySearch": {
        "provider": "openai-compatible",
        "model": "bge-m3",
        "remote": {
          "baseUrl": "http://127.0.0.1:8081"
        }
      }
    }
  }
}
```

Перезапустить OpenClaw.

### Этап 4: Индексация .md файлов (2-3 часа)
- Добавить в embedder поддержку .md файлов из:
  - `/root/LabDoctorM/adr`
  - `/root/LabDoctorM/patterns`
  - `/root/LabDoctorM/rules`
  - `/root/LabDoctorM/specs`
  - `/root/LabDoctorM/incidents`
  - `/root/LabDoctorM/docs`
  - `/root/LabDoctorM/projects/*`
  - `/root/LabDoctorM/workspaces/*`
- Запустить полную индексацию один раз
- Настроить инкрементальную переиндексацию по cron

### Этап 5: Инкрементальная переиндексация (1 час)
- Добавить в БД поле `content_hash` для каждого чанка
- При переиндексации: сравнивать хеш → обрабатывать только дельту
- Настроить cron: ежедневно в 02:00 UTC

## Риски и митигации

1. **FAISS не масштабируется** — при >100K чанков нужна замена на Qdrant/Milvus. Для нашего случая (~10K) — не проблема.
2. **Ollama падает** — systemd с `Restart=on-failure` уже настроен.
3. **Индекс рассинхронизируется** — инкрементальная переиндексация по хешам решает проблему.
4. **Качество поиска** — гибридный поиск (dense + sparse) улучшает точность.
