# ADR-0054: Gatekeeper — протокол «dead + heal + mandatory_retry»

- **Статус:** Accepted (2026-07-12)
- **Решение по:** инцидент с агентом Муравей (порт 8091 вне пула antcat + транзитный `Session not found`)
- **Связано с:** ADR-0053 (shim = принудительный перехват), ADR-0047 (карта портов/таймеров)

## Контекст

При деплое rerank-сервиса агент Муравей вызвал `gatekeeper__register_port` (8090, затем 8091).
Gatekeeper в этот момент был мёртв (транзитная смерть MCP-сессии) — агент получил глухой
`Session not found`, без указания причины и без инструкции к действию. Агент задеплоил сервис
на 8091 (вне своего пула 8100-8119) без регистрации. Гэп вскрылся только когда ЗавЛаб спросил.

Корень: **сервер не дал агенту контрактного ответа при своей смерти**. Агент не виноват —
он не получил обратной связи.

## Решение

Введён обязательный протокол ответа при недоступности Gatekeeper:

1. Агент дёрнул сервер (`register_port` / `register_timer`).
2. Ответ содержит: **статус «сервер мертв»** + **инструкцию как вылечить** + **обязательство повторить попытку**.
3. Агент по контракту: **лечит** сервер (restart) и **повторяет** вызов регистрации.

Форма ответа клиента `gk-register` при недоступности Gatekeeper:

```
GATEKEEPER_DEAD {"status":"dead","heal":"systemctl restart mcp-gatekeeper","mandatory_retry":true,"message":"Gatekeeper мертв. Вылечи: systemctl restart mcp-gatekeeper, затем ОБЯЗАН повторить register_port."}
```

- `status: "dead"` — однозначный сигнал смерти сервера.
- `heal` — точная инструкция по лечению (restart юнита).
- `mandatory_retry: true` — агент обязан повторить вызов после лечения.
- `message` — человекочитаемая инструкция (для LLM-агента и человека).

Клиент `gk-register` возвращает exit code **2** при `dead` (сигнал mandatory_retry),
**1** при `REJECT`, **0** при `ALLOW`.

## Почему не «фоновый heal»

ЗавЛаб уточнил: не нужно лечить сервер молча в фоне. Агент должен **получить ответ**
с инструкцией и обязательством повтора. Это event-based модель (ADR-0053): событие
«сервер мертв» порождает не фоновое действие, а **контрактный ответ агенту**, который
агент обязан исполнить. Гэп не остаётся незамеченным — агент сам лечит и повторяет.

## Реализация

- `shim/gk-register` — при `Exception`/таймауте соединения с Gatekeeper возвращает
  `GATEKEEPER_DEAD <json>` и exit 2 (вместо fail-open `ERROR`).
- `shim/systemctl-wrapper` (Слой 1) — распознаёт `GATEKEEPER_DEAD` от `gk-register`,
  печатает контракт агенту и завершается с exit 2 (НЕ вызывает оригинал systemctl,
  не fail-open). При `REJECT` — блокирует (exit 1). При `ALLOW` — вызывает оригинал.
- MCP-путь (`gatekeeper__register_port`): агентам рекомендуется использовать `gk-register`
  (клиент возвращает контракт). Для прямого MCP-вызова требуется прокси-слой перед
  Gatekeeper, возвращающий тот же контракт при смерти сервера (отдельная задача).

## Последствия

- Агент всегда получает чёткую обратную связь: либо ALLOW/REJECT (с причиной), либо
  контракт «мертв + лечи + повтори».
- Гэп (порт вне пула / незарегистрированный) не остаётся незамеченным — агент сам лечит
  и повторяет, получая корректный REJECT (вне пула) и переносит порт.
- Порядок перестаёт зависеть от вопроса ЗавЛаба — контракт замыкает цикл автоматически.
- Не требует фоновых heal-демонов: агент сам исполняет контракт.

## Инцидент

- `INC-20260712-gatekeeper-no-dead-contract` — корень: Gatekeeper не возвращал контрактного
  ответа при смерти, агент получал `Session not found` и задеплоил вне порядка.
