# ADR-0047: Канонический реестр портов и таймеров

## Контекст

Агенты лаборатории постоянно путались в портах и таймерах. Глубокое погружение
(2026-07-10, Ворон) выявило корень проблемы: **не отсутствие карты, а её
рассинхрон**. В `projects/DoctorM_and_Ai/docs/` существовало минимум четыре
перекрывающихся документа, каждый вручную обновлялся разными агентами и все
оказались неверными по-своему:

- `PORT_REGISTRY.md` (2026-07-08) — содержал ложные порты (53/dnsmasq мертв,
  containerd 35625 вместо 36401, xray 58762 не слушает), не хватало реальных
  (onnx-worker 8084/8085, MCP 8086/8087).
- `TIMER_REGISTRY.md` (2026-07-08) — пропущен `krv-notify.timer`, время
  `reindex-full` указано неверно (04:30 вместо 00:02 UTC), отключённые таймеры
  (`lab-memory-healthcheck`, `reindex-incremental`) числились активными, а
  `reindex-incremental.service` реально в состоянии failed.
- `CRON_REGISTRY.md` (2026-06-22) — всё ещё показывал удалённые 04.07 raven-кроны
  как активные.
- `infra/SYSTEMD_TIMERS.md` (2026-06-21) — полностью устарел: ссылался на
  `monitor-lab.timer` / `xray-healthcheck.timer` / `tmp-cleanup.timer`, которых
  в системе нет.

Плюс `MEMORY.md` тиражировал миф про «5 MCP-серверов (8083/8086/8087/8088/8089)»
при реальных двух (8086/8087).

Ручное обновление устаревает за ~2 дня. Нужен механизм, исключающий рассинхрон.

## Решение

Установить **два кураторских реестра** как единый источник истины и **генератор
сырых сканов + drift-чек** как механизм удержания их в актуальном состоянии.
Периодика разбивается на 4 слоя планировщика, все — в одном файле.

### Источники истины

- `docs/PORT_REGISTRY.md` — кураторский реестр портов (порт, сервис, bind,
  владелец, назначение, публичность). Единственный авторитетный документ по портам.
- `docs/TIMER_REGISTRY.md` — кураторский реестр периодики в 4 слоях (см. ниже).
- `docs/PORT_SCAN.md` и `docs/TIMER_SCAN.md` — **машиногенерируемые** сырые сканы
  (вывод `ss` / `systemctl list-timers` / системного cron). Редактировать вручную
  запрещено; перезаписываются генератором.
- `bin/gen-port-timer-map.sh` — генератор: пишет `*_SCAN.md` и выводит drift
  (несовпадения скана с кураторским реестром). Запускается weekly-таймером и
  по требованию при изменении порта/таймера.

### Слои периодики (TIMER_REGISTRY.md)

- **Слой A — systemd timers (OS).** Генератор берёт из `systemctl list-timers --all`.
- **Слой B — системный cron** (`/etc/cron.d`, root crontab). Пакетные записи
  (certbot, sysstat, e2scrub_all) оставляем как есть, но регистрируем. Лаб-собственные
  задачи конвертируем в слой A.
- **Слой C — OpenClaw Gateway Scheduler** (agent cron). НЕ systemd, живёт в шлюзе
  (порт 18789). Не сканируется генератором; обновляется вручную по правилу
  «cron создан/удалён → реестр обновлён». Проверка: `openclaw cron list`.
- **Слой D — Agent heartbeat.** Фича шлюза, per-agent расписание в `HEARTBEAT.md`
  агента. Не таймер в классическом смысле; в реестре — одной строкой-ссылкой.

### Правила портов (P1–P7)

- **P1.** Единый источник истины — `PORT_REGISTRY.md`. Все прочие порт-доки
  (включая `infra/SYSTEMD_TIMERS.md`) удаляются/архивируются.
- **P2.** Живая система > документ. Расхождение `ss` vs реестр → правится реестр
  (через генератор/drift), никогда наоборот.
- **P3.** Регистрация обязательна. Любой слушающий порт = строка в реестре. Порт
  без регистрации — несанкционированный.
- **P4.** Диапазоны резерва: 3001–3099 dev/test · 8000–8099 API · 8100–8199
  вспомогательные · 9000–9099 мониторинг/метрики. Новый сервис берёт порт по
  назначению без коллизий; назначение фиксирует инфра/ЗавЛаб.
- **P5.** Публичная экспозиция — только через nginx (80/443/8443/8445) ИЛИ по
  явному разрешению ЗавЛаба. Прямой bind на 0.0.0.0 (snablab 8200, consilium 8300,
  chisel 8444, exporters 9100/9187, docker-порты) — только утверждённый и защищённый
  аутентификацией.
- **P6.** MCP-серверы = только те, что в `openclaw.json` (сейчас 8086 apikeys,
  8087 memory). Добавление/удаление → атомарно правятся `openclaw.json` +
  `PORT_REGISTRY` + `MEMORY.md`. Рассинхрон трёх = нарушение.
- **P7.** Запрет «призрачных» портов. Перед утверждением — проверка `ss -tlnp`. Не
  слушающиеся (как 53/dnsmasq, 35625, 58762) удаляются из реестра.

### Правила таймеров (T1–T8)

- **T1.** Единый источник истины — `TIMER_REGISTRY.md` (4 слоя A–D). Прочие
  таймер-доки удаляются.
- **T2.** Живая система > документ (аналог P2).
- **T3.** Регистрация обязательна. Каждый systemd-таймер и системный cron = строка.
- **T4.** Точность статуса. В реестре — реальный `is-enabled`. Disabled/inert
  (lab-memory-healthcheck, reindex-incremental) НЕ помечаются «активные/✅».
  Осиротевший failed service (reindex-incremental.service) → инцидент, не «работает».
- **T5.** Правило удаления (жёсткое): перед удалением проекта/сервиса —
  `systemctl disable --now <unit>.timer <unit>.service` → `daemon-reload` → только
  потом удаление файлов. Нарушение → регистрация инцидента.
- **T6.** Генератор автоматизирует слои A+B (пишет `*_SCAN.md`, drift-чек). Слой C
  обновляется вручную (см. T8). Слой D — ссылка.
- **T7.** Cron-гигиена. В `/etc/cron.d` не должно быть `.bak`/дубликатов. Лаб-собственный
  cronконвертируется в systemd-таймер, где возможно (systemd доступен).
- **T8.** Любой agent-cron (OpenClaw, слой C) регистрируется при создании; удаление
  без правки реестра = нарушение.

### Управление (G1–G4)

- **G1.** Генератор `bin/gen-port-timer-map.sh` — единственный способ обновить
  сырые сканы. Запуск: weekly timer + по требованию.
- **G2.** Дрифт-аудит: если live (`*_SCAN.md`) ≠ реестру > 7 дней → алерт ЗавЛабу.
- **G3.** Изменение порта/таймера = regenerate + обновить кураторский реестр +
  commit через `lab-commit.sh` + упоминание в ежедневном отчёте агента.
- **G4.** Ответственные: инфра (Бестия/антcat) — актуальность генератора; ЗавЛаб —
  утверждение публичных портов и новых диапазонов.
- **G5.** Агенты консультируются с семантической памятью лабы (`mcp__memory__search`) по фактам о портах/таймерах/ADR-0047. Кураторские реестры — часть корпуса сем-памяти (PORT_REGISTRY уже проиндексирован; TIMER_REGISTRY индексируется штатным пайплайном reindex, зона antcat). Это делает «порядок в таймерах и портах» нативно доступным каждому агенту через MCP-сервер `memory` (:8087).

### Обоснование выбора systemd над cron

Любой cron выразим через systemd-таймер (`OnCalendar` — надмножество cron + триггеры
по загрузке, jitter, catch-up пропущенных запусков через `Persistent=true`). Для
однородного systemd-хоста лаборатории systemd-таймер строго предпочтительнее по
наблюдаемости (`journalctl -u`, `systemctl list-timers` в одном месте), атомарности с
сервисом и песочнице. Cron оставляем только для: (а) пакетных cron.d (certbot/sysstat),
(б) контейнеров без systemd.

## Последствия

- Агенты получают один файл для портов и один для таймеров; неоднозначность снята.
- Рассинхрон невозможен при регулярном прогоне генератора + соблюдении T8/G3.
- Удалён устаревший `infra/SYSTEMD_TIMERS.md`; `MEMORY.md` исправлена (MCP = 2 сервера).
- Требует дисциплины: создал/удалил порт или таймер → обнови реестр (G3, T8).

## Статус

Принято (reviewed_by: ЗавЛаб, 2026-07-11). Реализовано в составе: переписанные
`PORT_REGISTRY.md` / `TIMER_REGISTRY.md`, генератор + weekly timer, архив
`infra/SYSTEMD_TIMERS.md`, чистка `/etc/cron.d/cheque-bot*.bak`, правка `MEMORY.md`.
