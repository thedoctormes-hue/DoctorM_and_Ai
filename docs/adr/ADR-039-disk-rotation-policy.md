# ADR-039: Политика ротации и очистки дикового пространства

## Контекст

Диск лаборатории заполнен на 77% (44G/59G). Анализ показал отсутствие единой политики ротации и очистки. Каждый сервис настроен в вакууме — кто пишет на диск, но никто не убирает за собой.

**Выявленные генераторы мусора:**
- `/tmp` — 346M, 23,448 файлов старше 1 дня (systemd-tmpfiles-clean не настроен)
- `/var/backups/postgresql/` — 495M, 122 файла без ротации
- Docker — 5.2G (2.2G образов + 448M томов + 2.9G containerd)
- `/var/log/journal/` — 240M+ (нет лимита)
- `/root/LabDoctorM/.git/` — 805M (нет gc)

## Решение

Установить стандарт: **при создании любого сервиса — сразу настроить ротацию его артефактов**.

Конкретные меры:

### 1. /tmp — автоматическая очистка
- В `/usr/lib/tmpfiles.d/tmp.conf` задать возраст файлов: `D /tmp 1777 root root 10d`
- Служба `systemd-tmpfiles-clean` будет автоматически удалять файлы старше 10 дней

### 2. Бэкапы PostgreSQL — полная ротация
- Расширить паттерн удаления на ВСЕ бэкапы в `/var/backups/postgresql/`
- Или разнести по директориям с индивидуальной ротацией
- Рекомендуемый срок хранения: 30 дней

### 3. Docker — еженедельная очистка
- Настроить крон: `docker image prune -a -f && docker volume prune -f && docker container prune -f`
- Частота: еженедельно (воскресенье 03:00 MSK)

### 4. Journald — ограничение размера
- `SystemMaxUse=100M`, `SystemMaxFileSize=50M`, `MaxRetentionSec=7day`
- В `/etc/systemd/journald.conf`

### 5. Git — регулярное обслуживание
- `git gc --aggressive` — ежемесячно через крон

### 6. Мониторинг
- Агент Доминика: проверка диска каждый heartbeat, алерт при >80%

## Последствия

**Положительные:**
- Освобождение ~3.5G дискового пространства
- Автоматическая предотвращение накопления мусора
- Снижение риска отказа сервисов из-за заполненного диска

**Риски:**
- При удалении бэкапов — потеря старых резервных копий (митигация: проверить репликацию)
- При Docker prune — удаление неиспользуемых томов (митигация: проверить список перед первым запуском)
