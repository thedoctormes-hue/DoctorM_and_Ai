# Стандарты качества лаборатории

## Зона ответственности

Документ определяет стандарты качества, которые аудитор (Сова) проверяет при ревью коммитов и кода.

## 1. Стандарты кода

### 1.1. Общие правила

- Код должен быть читаемым — понятные имена переменных, функций, файлов
- Комментарии только где логика неочевидна — не дублируй код словами
- Длина функции — не более 50 строк (если больше — разбить)
- Длина файла — не более 500 строк (если больше — выделить модули)
- DRY: дублирование кода — рефакторинг, не копипаста

### 1.2. Безопасность

- Секреты (ключи, токены, пароли) — только в переменных окружения, никогда в коде
- Внешние зависимости — проверять на CVE перед добавлением
- Входные данные — всегда валидировать и санитизировать
- SQL — только параметризованные запросы, никогда конкатенация
- Файловые операции — проверять пути, не допускать path traversal

### 1.3. Обработка ошибок

- Каждый внешний вызов (API, файл, сеть) — с try/catch или проверкой возврата
- Ошибки логируются с контекстом — не глотать исключения молча
- Пользователю — понятное сообщение, не stack trace

### 1.4. Тестирование

- Код с логикой — должен иметь unit-тесты
- Критичные пути — интеграционные тесты
- Тесты должны быть воспроизводимы — без зависимости от внешних сервисов
- Моки для внешних API, реальные вызовы — только в integration-тестах

## 2. Стандарты Git

### 2.1. Коммиты

- Все коммиты — только через `lab-commit.sh`
- Формат сообщения: `<тип>: <описание>` (feat:, fix:, docs:, refactor:, test:, chore:)
- Один коммит — одна логическая единица изменения
- Не коммитить: `.env`, секреты, бинарники, `node_modules/`, `.git/`

### 2.2. Ветки

- `main` — защищена, мерж только через Кота или ЗавЛаба
- Feature-ветки: `<агент>/<описание>` (например, `owl/fix-skill-references`)
- Перед push в общую ветку — сначала `pull --rebase`

### 2.3. Race protection

- Не перезаписывать чужие коммиты
- При конфликте — решить вручную, не форсировать

## 3. Стандарты документации

### 3.1. Markdown-файлы

- Все `.md` файлы — с YAML frontmatter (name, description, last_reviewed, status)
- Заголовки — иерархические, без пропуска уровней
- Ссылки — проверять актуальность при каждом ревью

### 3.2. Проекты

Каждый проект должен иметь:
- `README.md` — описание, установка, использование
- `PROJECT.md` — архитектура, решения, зависимости
- `CHANGELOG.md` — история изменений

### 3.3. ADR (Architecture Decision Records)

- Каждое архитектурное решение — в `/root/LabDoctorM/adr/`
- Формат: контекст → решение → последствия
- Одно решение — один файл

## 4. Стандарты скилов

### 4.1. Структура

- `SKILL.md` — до 200 строк (максимум 500)
- Детали выносятся в `REFERENCE.md`, `TEMPLATES/`, `SCRIPTS/`
- Ссылки между скилами — по реальному имени из frontmatter

### 4.2. Frontmatter

Обязательные поля:
- `name` — lowercase, digits, hyphens
- `description` — что делает + когда использовать, третье лицо
- `version` — семантическое версионирование
- `author` — создатель
- `last_reviewed` — дата последней проверки
- `status` — active | deprecated | draft

### 4.3. Зависимости

- Внешние API — с fallback-логикой
- Скрипты — проверять существование через `requires.bins`
- Не молчать при ошибке — сообщать пользователю

## 5. Процесс аудита

### 5.1. Что проверяет Сова

- Соответствие кода стандартам раздела 1
- Корректность коммитов (раздел 2)
- Актуальность документации (раздел 3)
- Качество скилов (раздел 4)
- Отсутствие утечек секретов

### 5.2. Критичность

- 🔴 CRITICAL — блокирует мерж, немедленное исправление
- 🟠 HIGH — исправить до следующего релиза
- 🟡 MEDIUM — исправить в рабочем порядке
- 🟢 LOW — бэклог

## Хранение файлов и артефактов (PAT-005)

**Принцип:** Каждый файл находится в предназначенном для него месте. Корень LabDoctorM — не свалка.

**Запрещено:**
- Создавать папки и файлы в корне `/root/LabDoctorM/` (кроме системных: projects/, workspaces/, vault/, .ops/, .gitignore, *.md)
- Складывать артефакты работы агентов вне их workspace
- Складывать артефакты проекта вне папки проекта
- Коммитить node_modules/, dist/, build/, .venv/, __pycache__/, .next/, .cache/

**Структура:**
- Проект (код, документация) → `/root/LabDoctorM/projects/<имя>/`
- Рабочие файлы агента → `/root/LabDoctorM/workspaces/<агент>/`
- Инфраструктура лаборатории → `/root/LabDoctorM/projects/DoctorM_and_Ai/`
- Операционные артефакты → `/root/LabDoctorM/.ops/`
- Секреты → `/root/LabDoctorM/vault/`

**Ответственность:** Каждый агент обязан знать куда класть файлы. При неясности — спросить ЗавЛаба.

**История:** Принято 29.06.2026 после инцидента массового засорения корня LabDoctorM (sqopic/, .qwen/, grafana/, prometheus/ в корне).
